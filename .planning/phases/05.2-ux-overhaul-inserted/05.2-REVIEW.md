# Phase 05.2: Code Review Report

**Reviewed:** 2026-04-16T14:30:00Z
**Depth:** standard
**Files Reviewed:** 18
**Status:** issues_found

## Summary

Reviewed the UX overhaul phase covering shell layout, character foundation, level progression, skills board, copy strings, CSS, router, and test files. The codebase is generally clean with consistent patterns, good use of Zustand selectors, and proper separation between domain logic and UI. Key concerns: one type-safety issue that could silently corrupt state via an invalid CanonicalId cast, several missing UI guard conditions in the attributes editor, and a duplicated type definition across modules.

## Critical Issues

### CR-01: Cancel action passes empty string as CanonicalId, bypassing type constraint

**File:** `apps/planner/src/features/character-foundation/origin-board.tsx:69`
**Issue:** The cancel handler casts an empty string `'' as CanonicalId` and passes it to `config.onSelect()`, which calls either `setRace('')` or `setAlignment('')`. The `CanonicalId` type is a template literal `${EntityKind}:${string | number}`, so `''` does not conform to it. While TypeScript allows this cast at compile time, the empty string propagates into the store and selectors as a truthy non-null value. Downstream, `findRace('')` and `findAlignment('')` will return `null` (no match), but `state.raceId` / `state.alignmentId` will be the empty string rather than `null`. This means guards like `foundationState.raceId !== null` (creation-stepper.tsx:31) and `state.raceId === null` (selectors.ts:143-144) will evaluate incorrectly -- the step will appear "complete" with no actual race selected, and alignment/subrace locking logic will break.
**Fix:** Pass `null` instead of a cast empty string. The `onSelect` callback and store action already accept `CanonicalId | null`:
```tsx
onCancel={() => config.onSelect(null)}
```
This aligns with `setRace` and `setAlignment` type signatures which accept `CanonicalId | null`, and properly resets the selection state.

## Warnings

### WR-01: Increase button not disabled when budget is exhausted

**File:** `apps/planner/src/features/character-foundation/attributes-board.tsx:73-74`
**Issue:** The "Aumentar" (increase) button is disabled only when `value >= maximum`. It does not check whether the point budget has remaining points. A user can keep clicking increase as long as the score is below 18, even if `attributeBudget.remainingPoints` is 0 or negative. The budget violation is flagged as `illegal` after the fact by the rules engine, but strict validation (per CLAUDE.md: "illegal server builds must be blocked, not merely warned about") requires preventing the action in the first place. The cost curve is non-linear (e.g., going from 15 to 16 costs 2 points), so a simple `remainingPoints <= 0` check is not sufficient -- the button should check whether the cost of the next increment exceeds remaining budget.
**Fix:** Compute the incremental cost and disable the button when it exceeds the remaining budget:
```tsx
const nextCost =
  (phase03FoundationFixture.attributeRules.costByScore[String(value + 1)] ?? 0) -
  (phase03FoundationFixture.attributeRules.costByScore[String(value)] ?? 0);

<NwnButton
  aria-label={`Aumentar ${ATTRIBUTE_LABELS[key]}`}
  disabled={value >= maximum || nextCost > attributeBudget.remainingPoints}
  onClick={() => setBaseAttribute(key, value + 1)}
  variant="secondary"
>
```

### WR-02: setBaseAttribute store action performs no bounds validation

**File:** `apps/planner/src/features/character-foundation/store.ts:63-68`
**Issue:** The `setBaseAttribute` action blindly sets the value without clamping to `[minimum, maximum]`. While the UI currently disables buttons at the bounds, any direct caller (tests, future keyboard input, or programmatic access) can set an attribute value outside the valid range. The `calculateAbilityBudgetSnapshot` in the rules engine will flag out-of-range values, but the invalid value persists in state until corrected.
**Fix:** Clamp the value in the store action:
```ts
setBaseAttribute: (key, value) =>
  set((state) => {
    const { minimum, maximum } = phase03FoundationFixture.attributeRules;
    const clamped = Math.max(minimum, Math.min(maximum, value));
    return {
      baseAttributes: {
        ...state.baseAttributes,
        [key]: clamped,
      },
    };
  }),
```

### WR-03: Unsafe `level as ProgressionLevel` narrowing cast on `number` type

**File:** `apps/planner/src/features/level-progression/build-progression-board.tsx:33`
**Issue:** `activeSheet.level` is typed as `number` in `ActiveLevelSheetView` (level-progression/selectors.ts:71), but is cast to `ProgressionLevel` (union of 1-16) at the call site. If a future change introduces a level value outside 1-16, this cast will silently pass a bad value to `setLevelClassId`. The `ActiveLevelSheetView.level` field should be typed as `ProgressionLevel` at the source to make this cast unnecessary.
**Fix:** In `apps/planner/src/features/level-progression/selectors.ts`, change line 71 from:
```ts
level: number;
```
to:
```ts
level: ProgressionLevel;
```
Then remove the cast in `build-progression-board.tsx:33`:
```ts
setLevelClassId(activeSheet.level, id as CanonicalId);
```

### WR-04: Switch statement in CenterContent has no default case for level sub-steps

**File:** `apps/planner/src/components/shell/center-content.tsx:23-32`
**Issue:** The `switch (activeLevelSubStep)` covers `class`, `skills`, `feats`, and `spells`, but has no `default` case. The `LevelSubStep` type is `'class' | 'skills' | 'feats' | 'spells'`, so today this is exhaustive. However, if a new sub-step is added to the `LevelSubStep` union, TypeScript will not error here because the function falls through to the default empty state return. Adding an exhaustive check (or a `default: never` assertion) would catch missing branches at compile time.
**Fix:** Add exhaustive checking after the switch:
```tsx
switch (activeLevelSubStep) {
  case 'class':
    return <BuildProgressionBoard />;
  case 'skills':
    return <SkillBoard />;
  case 'feats':
    return <PlaceholderScreen title="Dotes" body="..." />;
  case 'spells':
    return <PlaceholderScreen title="Conjuros" body="..." />;
  default: {
    const _exhaustive: never = activeLevelSubStep;
    return _exhaustive;
  }
}
```

## Info

### IN-01: Duplicated ProgressionLevel type across two modules

**File:** `apps/planner/src/lib/sections.ts:42`
**Issue:** `ProgressionLevel` is defined independently in both `apps/planner/src/lib/sections.ts` (hardcoded union) and `apps/planner/src/features/level-progression/progression-fixture.ts` (derived from `PROGRESSION_LEVELS` array). They resolve to the same type structurally, but are separate definitions. Some files import from `sections.ts` (build-progression-board, level-rail, level-sub-steps, planner-shell) while others import from `progression-fixture.ts` (skill-sheet, skill selectors, level-progression selectors). This split makes it easy for the two to drift apart.
**Fix:** Remove the definition from `sections.ts` and re-export from the canonical source:
```ts
// sections.ts
export type { ProgressionLevel } from '@planner/features/level-progression/progression-fixture';
```

### IN-02: Hardcoded Spanish strings in component bodies instead of copy module

**File:** `apps/planner/src/features/character-foundation/attributes-board.tsx:95-103`
**Issue:** Several inline Spanish strings bypass the `shellCopyEs` copy module: "Reiniciar base" (line 95), "Caracteristicas iniciales" (line 101), "Ajusta la base inicial respetando el presupuesto del servidor..." (line 102). Similarly in `build-progression-board.tsx` lines 39-41 ("Clase seleccionada"), `origin-board.tsx` lines 67-85, and `skill-board.tsx` lines 33-34. While the project is Spanish-first, centralizing all user-facing text in the copy module makes future updates and consistency easier.
**Fix:** Move these strings into `shellCopyEs` (or a feature-specific copy object) and reference them from there.

### IN-03: `useCharacterFoundationStore()` called without selector triggers re-render on any store change

**File:** `apps/planner/src/features/character-foundation/origin-board.tsx:20`
**Issue:** `const foundationState = useCharacterFoundationStore()` subscribes to the entire store, causing this component to re-render on every state change (including unrelated attribute changes when the user is on a different step). The same pattern appears in `creation-stepper.tsx:23`, `attributes-board.tsx:35`, and `skill-board.tsx:13-14`. This is not a correctness bug, but it causes unnecessary re-renders in a complex UI.
**Fix:** Use targeted selectors to subscribe only to the state slices needed, or extract needed values individually:
```ts
const raceId = useCharacterFoundationStore((s) => s.raceId);
const alignmentId = useCharacterFoundationStore((s) => s.alignmentId);
// etc.
```
