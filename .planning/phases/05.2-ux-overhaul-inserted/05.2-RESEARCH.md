# Phase 05.2: UX Overhaul - Research

**Researched:** 2026-04-15
**Domain:** CSS architecture, layout restructuring, component replacement, NWN1-replica visual system
**Confidence:** HIGH

## Summary

This phase is a pure visual and structural overhaul of an existing working planner. No new game mechanics, data catalogs, or Zustand store logic is introduced. The scope is: (1) invert the entire color palette from light parchment to dark/gold NWN1 theme, (2) replace the 7-section sidebar navigation with a creation stepper + level rail, (3) replace the summary panel with a tabbed NWN1 character sheet, (4) restructure route content from vertical-scroll card layouts to the NWN1 list+detail split pattern, and (5) overhaul typography from web-display sizing to game-UI-density sizing.

The codebase is well-structured for this change. CSS tokens are centralized in `tokens.css` (24 lines), all visual styling is in a single `app.css` file (930 lines), and the component tree has clear separation between state logic (stores, selectors, pure functions) and visual projection (route views, board components, rail/sheet pairs). The stores and selectors remain untouched; only the shell frame, navigation, summary panel, and per-route visual components change.

**Primary recommendation:** Execute as a layered transformation: tokens first (instant global palette flip), then shell structure (stepper + character sheet replace nav + summary), then per-step content screens (list+detail pattern), then typography and spacing fine-tuning. This order minimizes intermediate breakage because the token swap is atomic and the shell restructure is the single highest-impact change.

### System Architecture Diagram

```

        |
        v
+---[Shell State Store]---+     (planner-shell.ts -- activeStep, expandedLevel)
|                          |
|  activeStep changed      |
|                          |
+----+----+----+-----------+
     |    |    |
     v    v    v
[Stepper] [Center Content]  [Character Sheet]
  (left)    (fills center)    (right panel)
     |         |                   |
     |    +----+----+              |
     |    |         |              |
     v    v         v              v
  Step   Option   Detail     Tab panels read
  list   List     Panel      from existing
  from   from     from       foundation +
  store  store    store      progression +
  state  state    state      skill stores
```

Data flow is strictly unidirectional:
1. User clicks a stepper step or level button
2. Shell store updates `activeStep` / `expandedLevel`
3. Center content renders the appropriate selection screen (list + detail)
4. User makes a selection (race, class, skill rank, etc.)
5. Domain store updates (character-foundation, level-progression, skills)
6. Character sheet panel re-derives display from existing selectors

No new stores are needed. The shell store gains `activeStep` and `expandedLevel` fields replacing the current `activeSection` string.

### Pattern 1: Token-First Palette Inversion

**What:** Replace all CSS custom property values in `tokens.css` atomically, following the UI-SPEC migration table. Remove tokens that have no NWN1 equivalent (`--shadow-panel`, `--color-panel-glow`). Add new tokens (`--color-selection-border`, `--color-blocked`, `--border-frame-inner`).

**When to use:** As the very first change in the phase. Every component in `app.css` already references these tokens, so swapping values is an instant global visual flip.

**Example:**
```css
/* Source: UI-SPEC.md CSS Token Migration table [VERIFIED: 05.2-UI-SPEC.md] */

  /* Palette inversion: light -> dark */
  --color-surface: #0a0a0a;
  --color-panel: #141010;
  --color-gold: #c8a84e;
  --color-text: #e8d5a0;
  --color-text-muted: #8b7355;
  --color-destructive: #7a1f1f;
  --color-panel-edge: rgba(200, 168, 78, 0.35);
  --color-selection: rgba(200, 168, 78, 0.18);
```

### Pattern 3: Step-Driven Center Content

**What:** Instead of the current model where each route renders its own full-page section view, the center content is driven by the active stepper step. The shell frame reads `activeStep` from the store and renders the matching `SelectionScreen` component in the center column.

**When to use:** For all creation and progression steps. Each step type (race selection, alignment grid, ability scores, class selection, skill distribution, feat selection, spell selection) gets a variant of the list+detail pattern.

**Example:**
```tsx
// Pattern: shell frame selects center content based on active step

function CenterContent({ activeStep }: { activeStep: StepId }) {
  switch (activeStep) {
    case 'origin-race':
      return <RaceSelectionScreen />;
    case 'origin-alignment':
      return <AlignmentSelectionScreen />;
    case 'level-class':
      return <ClassSelectionScreen />;
    case 'level-skills':
      return <SkillDistributionScreen />;
    // ... etc
    default:
      return <EmptyStateScreen />;
  }
}
```

### Anti-Patterns to Avoid

- **Scattershot CSS changes:** Do NOT go file-by-file changing colors. Change `tokens.css` first, then fix remaining hardcoded values in `app.css`. All hardcoded `rgba(159, 122, 49, ...)` values in `app.css` (there are ~40+) must be replaced with token references.
- **Mixing structural and visual changes:** Do NOT change component structure and visual tokens in the same commit. Token swap first (atomic), then structural component changes.
- **Breaking store interfaces:** Do NOT modify the domain stores (character-foundation, level-progression, skills). Their interfaces are tested and consumed by selectors. Only the shell store changes.
- **Removing routes prematurely:** The current 7 routes (build, skills, spells, abilities, stats, summary, utilities) may need restructuring, but removing them before the stepper is wired will break all existing navigation and test expectations.

## Common Pitfalls

### Pitfall 1: Hardcoded Color Values in app.css

**What goes wrong:** The token swap in `tokens.css` changes all token values, but `app.css` contains ~40+ hardcoded `rgba(...)` color values that reference the old palette directly, bypassing tokens. These will survive the token swap and create visual inconsistencies.
**Why it happens:** Early development used inline rgba values for fine-grained opacity variants rather than creating tokens for every shade.
**How to avoid:** After the token swap, do a systematic pass through `app.css` replacing every hardcoded `rgba(159, 122, 49, ...)` (old bronze), `rgba(255, 248, 236, ...)` (old parchment-white), `rgba(32, 24, 18, ...)` (old dark-panel), etc. with the new token-based values.
**Warning signs:** Any light/warm tones visible after the token swap indicate missed hardcoded values.

### Pitfall 2: Border-Radius Sweep

**What goes wrong:** The codebase has 17 distinct `border-radius` declarations ranging from `14px` to `999px`. NWN1 uses sharp corners exclusively (`0px`). Missing even one creates a jarring visual mismatch.
**Why it happens:** Each component was styled independently with rounded corners.
**How to avoid:** Global search-and-replace of all `border-radius` values to `0` (or `0px`) in `app.css`. Then re-add `border-radius: 999px` only for the `.sr-only` utility if needed.
**Warning signs:** Any rounded corner visible in the UI post-overhaul.

### Pitfall 3: Font Weight 700 Orphan

**What goes wrong:** `fonts.css` imports `@fontsource/cormorant-garamond/700.css` but the UI-SPEC only uses weights 400 and 600. The 700 import is used once in `app.css` (line 374, `.attribute-control strong`). If the import is removed but the CSS reference remains, the browser falls back to a synthetic bold that looks wrong.
**Why it happens:** Phase 3 used 700 weight for attribute values; the new UI-SPEC uses 600 for all bold text.
**How to avoid:** Change `font-weight: 700` to `font-weight: 600` in all CSS rules, then remove the 700.css import from `fonts.css`. Verify no references remain.
**Warning signs:** Visually heavier-than-expected text in attribute controls.

### Pitfall 4: Route Regression During Restructure

**What goes wrong:** The current router defines 7 flat routes. The stepper model fundamentally changes navigation -- steps are no longer routes. If routes are removed before the stepper wiring is complete, the app breaks with no way to navigate.
**Why it happens:** Temptation to clean up the route tree early in the process.
**How to avoid:** Keep existing routes functional throughout the restructure. Wire the stepper to drive center content via store state first. Only restructure/remove routes after the stepper is the sole navigation mechanism and tests confirm it works.
**Warning signs:** Blank screens when clicking stepper steps.

### Pitfall 5: Existing Test Expectations

**What goes wrong:** Phase 2 tests (`layout-shell.spec.ts`, `theme-contract.spec.ts`, `navigation-copy.spec.ts`, `shell-routes.spec.ts`) assert specific DOM structure, CSS class names, ARIA labels, and token values that will all change. Running tests during the overhaul produces cascading failures.
**Why it happens:** Tests were written for the old visual structure.
**How to avoid:** Update tests in the same wave as the components they test. The theme-contract test checks for `--color-parchment: #e3d8c1` -- this must become `--color-surface: #0a0a0a` (or be rewritten). The layout-shell test checks for "Abrir navegacion" button -- this may no longer exist in the new shell.
**Warning signs:** Test suite turns red after the first structural change.

### Pitfall 6: Stepper State vs Domain State Confusion

**What goes wrong:** The stepper tracks "which step is active" (UI navigation state) while domain stores track "what the user has selected" (game state). Conflating them leads to bugs where navigating back to a step clears the selection, or where completing a step doesn't advance the stepper.
**Why it happens:** The current model uses `activeSection` as a simple section ID that maps 1:1 to routes. The new model has nested state (origin steps + level sub-steps) that must stay synchronized with domain state (is race selected? is level 3 configured?).
**How to avoid:** Stepper step completion state should be DERIVED from domain store selectors, not stored separately. The shell store only tracks "which step is open for viewing," while "which steps are complete" comes from existing selectors like `selectOriginReadyForAbilities` and `selectLevelRail`.
**Warning signs:** Steps showing as "complete" when no selection has been made, or vice versa.
