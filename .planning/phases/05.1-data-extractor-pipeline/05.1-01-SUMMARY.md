## Decisions Made
- **fzstd over native zstd:** Used pure JS fzstd for zero-native-dependency safety on Windows. Performance is adequate for build-time extraction of <1MB NWN payloads.
- **node:zlib for test fixtures:** Since fzstd is decompress-only, test fixture generation uses Node 24's built-in `zstdCompressSync` to create valid NSYC blobs.
- **Explicit truncated-payload check:** Added validation in NSYC decompressor for blobs with no data after the 24-byte header, since fzstd silently returns empty on empty input instead of throwing.
- **Test naming convention:** Used `.spec.ts` extension instead of plan's `.test.ts` to match the existing vitest include pattern (`tests/**/*.spec.ts`).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Changed test import from fzstd.compressSync to node:zlib.zstdCompressSync**
- **Found during:** Task 2 (RED phase test execution)
- **Issue:** fzstd is a decompress-only library; it does not export `compressSync`. Tests needed compression to build synthetic NSYC blobs.
- **Fix:** Used Node 24's built-in `node:zlib` `zstdCompressSync` for test fixture generation instead.
- **Files modified:** tests/phase-05.1/nsyc-decompress.spec.ts
- **Verification:** All 5 NSYC tests pass
- **Committed in:** 03e497c (GREEN phase commit)

**2. [Rule 2 - Missing Critical] Added truncated-payload validation in NSYC decompressor**
- **Found during:** Task 2 (GREEN phase)
- **Issue:** fzstd `decompress()` silently returns an empty Uint8Array for empty input instead of throwing. A truncated NSYC blob (header only, no payload) would pass validation but return empty data.
- **Fix:** Added explicit check: `blob.length <= NSYC_HEADER_SIZE` throws "Truncated NSYC container: no payload after header".
- **Files modified:** packages/data-extractor/src/parsers/nsyc-decompress.ts
- **Verification:** Truncated blob test passes with expected throw
- **Committed in:** 03e497c (GREEN phase commit)

**3. [Rule 3 - Blocking] Added fzstd as root devDependency for test resolution**
- **Found during:** Task 2 (GREEN phase)
- **Issue:** Tests run from the workspace root but fzstd was only installed in the data-extractor sub-package. The `@data-extractor/*` alias resolved but direct `fzstd` import in tests failed with ERR_MODULE_NOT_FOUND.
- **Fix:** Added fzstd as a root devDependency so vitest can resolve the import.
- **Files modified:** package.json, pnpm-lock.yaml
- **Verification:** All 28 tests pass
- **Committed in:** 03e497c (GREEN phase commit)

**4. [Rule 3 - Blocking] Used .spec.ts test file extension instead of plan's .test.ts**
- **Found during:** Task 2 (RED phase setup)
- **Issue:** vitest.config.ts include pattern is `tests/**/*.spec.ts` -- files named `.test.ts` would not be discovered.
- **Fix:** Named test files as `.spec.ts` to match existing convention.
- **Files modified:** All test files created with .spec.ts extension

- **Committed in:** 212427f (RED phase commit)
