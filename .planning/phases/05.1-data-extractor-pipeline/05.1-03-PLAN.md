  <name>Task 1: Implement NwsyncReader and TlkResolver</name>
  <files>packages/data-extractor/src/readers/nwsync-reader.ts, packages/data-extractor/src/readers/tlk-resolver.ts, tests/phase-05.1/nwsync-reader.test.ts, tests/phase-05.1/tlk-resolver.test.ts</files>
  <behavior>
    NwsyncReader:
    - constructor(metaDbPath, dataDbPath, manifestSha1) opens both SQLite databases
    - getResource(resref, restype) returns decompressed Buffer for a known resource
    - getResource for a resref not in the manifest returns null
    - getResource for pb_tlk_v6 (restype 2018) returns a valid TLK binary
    - listResources(restype) returns all resref names for the given type in the manifest
    - close() closes both database connections

    TlkResolver:
    - constructor(baseTlk: TlkTable, customTlk: TlkTable) stores both tables
    - resolve(strref) for strref < 0x1000000 returns text from baseTlk
    - resolve(strref) for strref >= 0x1000000 returns text from customTlk at (strref - 0x1000000)
    - resolve(strref) returns empty string for negative or undefined strrefs
    - resolve(strref) returns empty string when custom TLK entry has no text
    - fromPaths(baseTlkPath, nwsyncReader) factory loads both TLK files and constructs the resolver
  </behavior>
  <action>
1. **NwsyncReader** (`nwsync-reader.ts`): Per RESEARCH Pattern 5 and D-01.
   - Uses `better-sqlite3` to open `nwsyncmeta.sqlite3` and `nwsyncdata_0.sqlite3`.
   - Prepares SQL statements (per RESEARCH code example):
     - `metaStmt`: `SELECT resref_sha1 FROM manifest_resrefs WHERE manifest_sha1 = ? AND resref = ? AND restype = ?`
     - `dataStmt`: `SELECT data FROM resrefs WHERE sha1 = ?`
   - `getResource(resref, restype)`: queries meta for SHA1, then data for blob, decompresses with `decompressNsyc()`, returns Buffer.
   - `listResources(restype)`: `SELECT resref FROM manifest_resrefs WHERE manifest_sha1 = ? AND restype = ?` -- returns string array.
   - `close()`: closes both Database instances.
   - Per RESEARCH Pitfall 4: ALWAYS filter by PUERTA_MANIFEST_SHA1 to avoid mixing Arelith data.

2. **TlkResolver** (`tlk-resolver.ts`): Per RESEARCH Pattern 3 and D-04/D-05/D-06.
   - Constructor takes `baseTlk: TlkTable` and `customTlk: TlkTable`.
   - `resolve(strref: number): string`: If strref >= CUSTOM_TLK_OFFSET, look up `customTlk.getString(strref - CUSTOM_TLK_OFFSET)`. Otherwise look up `baseTlk.getString(strref)`. Return empty string for null/undefined/negative inputs.
   - Static factory `fromPaths(baseTlkPath: string, nwsyncReader: NwsyncReader): TlkResolver`: reads base TLK from file, reads custom TLK (pb_tlk_v6) from nwsync, parses both with `parseTlk()`, returns new TlkResolver.

3. **Tests**: NwsyncReader tests should be integration-style: wrap in `describe.skipIf(!existsSync(NWSYNC_META_DB))` so they only run when the nwsync database is locally available. Include assertions for:
   - `getResource('classes', 2017)` returns non-null Buffer
   - `getResource('nonexistent_resource', 2017)` returns null
   - `listResources(2017)` returns array containing 'classes', 'skills', 'feat'

   TlkResolver tests use synthetic TlkTable mocks:
   - Mock baseTlk returning "Fighter" for index 0, mock customTlk returning "Paladín Sagrado" for index 100.
   - `resolve(0)` returns "Fighter", `resolve(0x01000064)` returns "Paladín Sagrado".
   - `resolve(-1)` returns "".
  </action>

All assemblers follow a common pattern:
- Accept `nwsyncReader: NwsyncReader`, `baseGameReader: BaseGameReader`, and `tlkResolver: TlkResolver` as dependencies.
- Load the relevant 2DA table(s) from nwsync first; fall back to base game if not found (per D-03, source precedence: puerta-snapshot > base-game).
- Parse 2DA content with `parseTwoDa()`.
- Resolve Name/Description columns with `tlkResolver.resolve(parseInt(cell))` (per D-06).
- Generate canonical IDs in `kind:slug` format (per canonical-id.ts contract). Slug from the 2DA `Label` column, lowercased, with underscores replaced by hyphens.
- Return a log object alongside the catalog: `{ catalog: T, warnings: string[] }` (per D-13, best-effort with warnings).

1. **class-assembler.ts** (`assembleClassCatalog`):
   - Load `classes` 2DA from nwsync (resref "classes", restype 2017).
   - Filter rows where `PlayerClass` column = "1" (per D-08: ~47 player classes).
   - For each player class row, extract:
     - `id`: `class:${Label.toLowerCase().replace(/_/g, '-')}` (e.g., "class:fighter", "class:pale-master")
     - `label`: `tlkResolver.resolve(parseInt(Name))` -- Spanish name
     - `description`: `tlkResolver.resolve(parseInt(Description))` -- Spanish description
     - `isBase`: `PrimaryAbil` column presence or `HitDie` >= 6 heuristic (base classes have simpler structure) -- OR use the row index < some threshold. Actually, determine from `PreReqTable` -- base classes have null/empty prereq tables, prestige classes have them.
     - `hitDie`: parseInt of `HitDie` column
     - `skillPointsPerLevel`: parseInt of `SkillPointBase` column
     - `primaryAbility`: map from `PrimaryAbil` column
     - `attackBonusProgression`: derive from `AttackBonusTable` column (CLS_ATK_1 = low, CLS_ATK_2 = medium, CLS_ATK_3 = high)
     - `savingThrows`: derive from `FortSaveTable`/`RefSaveTable`/`WillSaveTable` columns (CLS_SPGN_* pattern = high, CLS_SPHP_* = low -- inspect naming convention)
     - `spellCaster`: `SpellGainTable` column is not null
     - `prerequisiteColumns`: extract PreReqTable, MaxLevel, and other prereq-related columns as a raw record
     - `sourceRow`: row index
     - `skillTableRef`, `featTableRef`, `spellGainTableRef`, `spellKnownTableRef`: column values for cross-reference
   - Per D-09: Handle `FavoredSoul` sharing `CLS_SKILL_CLER` table and similar server customizations.
   - Validate output with `classCatalogSchema.parse()`.

2. **race-assembler.ts** (`assembleRaceCatalog`):
   - Load `racialtypes` 2DA from nwsync.
   - Filter rows where `PlayerRace` = "1" (per D-08: ~19 player races).
   - Handle DELETED rows (non-contiguous indices, per RESEARCH anti-pattern).
   - For each player race: extract id, label (TLK), description (TLK), ability adjustments (StrAdjust, DexAdjust, etc.), favoredClass (from `Favored` column -> class ID), size (from `SizeCategory`).
   - Load `subraces` 2DA: try nwsync first, then base game BIF fallback (per RESEARCH Pitfall 5 -- subraces not in Puerta nwsync). If neither has it, log a warning and produce empty subraces array.
   - Filter non-deprecated subraces (per D-08). Map each subrace to its parent race ID.
   - Validate output with `raceCatalogSchema.parse()`.

3. **skill-assembler.ts** (`assembleSkillCatalog`):
   - Load `skills` 2DA from nwsync (39 entries, indices 0-38, per D-09).
   - For each skill: extract id (`skill:${label-slug}`), label (TLK), abilityKey, trainedOnly.
   - Per D-09: Handle renamed skills (Leadership, Sail, CraftMastery, Climb) and custom skills (indices 28-38).
   - Handle `HideFromLevelUp` column: skills with this flag = "1" are hidden/non-player (per D-08). If the column exists, filter them out. If not, include all.
   - Load cls_skill_* tables to build `defaultClassIds` for each skill:
     - List all `cls_skill_*` resrefs from nwsync using `nwsyncReader.listResources(2017)` filtered to `cls_skill_` prefix.
     - For each cls_skill table, parse the 2DA. Map skill indices to class IDs.
     - Per D-09: 19 classes use expanded 39-skill format (indices 0-38), 40 classes use standard 28-skill format (indices 0-27). Detect by checking if the table has rows beyond index 27.
     - A skill is "class skill" for a given class if the cls_skill table has value "1" at that skill's index.
     - Per D-09: Handle shared skill tables (e.g., FavoredSoul -> CLS_SKILL_CLER). The cls_skill table name comes from the class's `SkillsTable` column in classes.2da. Multiple classes may reference the same table.
   - Assign categories per the existing skill-catalog.ts pattern (athletic, discipline, lore, perception, social, stealth, utility). Use a hardcoded category map keyed by skill index for the 28 base skills; assign "utility" as default for custom skills 28-38.
   - Load existing `skillRestrictionOverrides` from the current `compiled-skill-catalog.ts` pattern (heavy-armor tumble override) and carry them forward. The assembler does not generate restriction overrides -- those come from manual overrides.
   - Validate output with `skillCatalogSchema.parse()`.

4. **deity-assembler.ts** (`assembleDeityData`):

   - The deity data gap is expected (per RESEARCH A4). Document the gap clearly in the extraction log so the user knows to provide manual deity overrides later.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SQLite database -> NwsyncReader | Untrusted database with shared blob storage across servers |
| 2DA content -> assemblers | Parsed text data with potentially missing or malformed values |
| TLK strref -> TlkResolver | Integer references that may point outside valid TLK ranges |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.1-09 | Tampering | nwsync-reader.ts | mitigate | Always filter by PUERTA_MANIFEST_SHA1 in SQL queries; never query without manifest filter (per RESEARCH Pitfall 4) |
| T-05.1-10 | Denial of Service | nwsync-reader.ts | mitigate | Use prepared statements; close database connections in close(); do not leave connections open after extraction |
| T-05.1-11 | Tampering | assemblers | mitigate | Validate assembled catalogs with Zod schemas before returning; log warnings for malformed rows per D-13 |
| T-05.1-12 | Spoofing | tlk-resolver.ts | accept | TLK content comes from trusted local game files; no authentication needed for build-time CLI |
</threat_model>
