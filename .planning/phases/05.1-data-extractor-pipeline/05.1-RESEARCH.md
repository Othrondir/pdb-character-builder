# Phase 05.1: Data Extractor Pipeline - Research

**Confidence:** HIGH

## Summary

This phase builds the extraction pipeline that reads Puerta de Baldur server data from the local nwsync SQLite databases and base-game files, then produces typed compiled TypeScript catalogs for all character-building data. The research verified every binary format involved, confirmed the nwsync database schema, validated TLK string resolution, and discovered a critical finding: the pre-extracted 2DA files in `server-extract/` are from a **stale snapshot** with different TLK IDs and content than the current nwsync data. The extractor must read directly from nwsync as the source of truth.

The pipeline is a build-time Node.js CLI that runs in `packages/data-extractor/`. It reads two SQLite databases (nwsyncmeta + nwsyncdata_0), decompresses NSYC v3 payloads using `fzstd` (pure JS Zstandard), parses binary TLK V3.0 and text 2DA V2.0 formats, applies the source precedence contract (manual override > puerta-snapshot > base-game), and emits compiled TypeScript catalog files under `apps/planner/src/data/`.

**Primary recommendation:** Build the extractor as a Node.js CLI using `better-sqlite3` for SQLite access, `fzstd` for zstd decompression, and hand-written binary parsers for TLK V3.0 and 2DA V2.0 (both formats are simple and well-understood). Do NOT use neverwinter.nim or Python for the extractor itself -- the nwsync database provides direct access to all resources without needing BIF/KEY parsing for server data.

### Pattern 1: NSYC v3 Decompression
**What:** Every resource blob in nwsyncdata_0.sqlite3 is wrapped in the NSYC v3 container format.
**When to use:** Every time a resource is read from the nwsync data database.
**Example:**
```typescript
// Source: [VERIFIED: reverse-engineered from actual nwsync blobs]
import { decompress } from 'fzstd';

const NSYC_MAGIC = Buffer.from('NSYC');
const NSYC_HEADER_SIZE = 24; // 4 magic + 4 version + 16 sub-header

function decompressNsyc(blob: Buffer): Buffer {
  if (!blob.subarray(0, 4).equals(NSYC_MAGIC)) {
    throw new Error('Not an NSYC container');
  }
  const version = blob.readUInt32LE(4);
  if (version !== 3) {
    throw new Error(`Unsupported NSYC version: ${version}`);
  }
  const payload = blob.subarray(NSYC_HEADER_SIZE);
  return Buffer.from(decompress(payload));
}
```

### Pattern 2: TLK V3.0 Binary Parsing
**What:** Parse the NWN Talk Table format to resolve string references to Spanish text.
**When to use:** Resolving Name and Description columns from 2DA files.
**Example:**
```typescript
// Source: [VERIFIED: xoreos source + confirmed with actual dialog.tlk and pb_tlk_v6]
// Header: 20 bytes
//   0-3:   "TLK " (magic)
//   4-7:   "V3.0" (version)
//   8-11:  language ID (uint32 LE)
//   12-15: string count (uint32 LE)
//   16-19: strings data offset (uint32 LE)
// Entry table: starts at byte 20, 40 bytes per entry
//   0-3:   flags (uint32 LE) - bit 0 = has text
//   4-19:  sound ResRef (16 bytes ASCII, null-padded)
//   20-23: volume variance (uint32 LE)
//   24-27: pitch variance (uint32 LE)
//   28-31: string offset (uint32 LE, relative to strings data)
//   32-35: string length (uint32 LE)
//   36-39: sound length (float32 LE)

interface TlkTable {
  stringCount: number;
  getString(index: number): string;
}

function parseTlk(data: Buffer): TlkTable {
  const stringCount = data.readUInt32LE(12);
  const stringsOffset = data.readUInt32LE(16);
  
  return {
    stringCount,
    getString(index: number): string {
      if (index < 0 || index >= stringCount) return '';
      const entryOffset = 20 + index * 40;
      const flags = data.readUInt32LE(entryOffset);
      if (!(flags & 1)) return ''; // no text flag
      const strOffset = data.readUInt32LE(entryOffset + 28);
      const strLen = data.readUInt32LE(entryOffset + 32);
      if (strLen === 0) return '';
      return data.subarray(
        stringsOffset + strOffset,
        stringsOffset + strOffset + strLen
      ).toString('latin1'); // NWN1 uses Windows-1252/Latin-1
    }
  };
}
```

### Pattern 3: Unified TLK Resolution
**What:** Resolve any TLK string reference from 2DA files using the dual-TLK system.
**When to use:** Every Name/Description field in every 2DA file.
**Example:**
```typescript
// Source: [VERIFIED: confirmed with actual nwsync data]
const CUSTOM_TLK_OFFSET = 0x01000000; // 16777216

function resolveStrref(
  strref: number,
  baseTlk: TlkTable,
  customTlk: TlkTable,
): string {
  if (strref >= CUSTOM_TLK_OFFSET) {
    const customIndex = strref - CUSTOM_TLK_OFFSET;
    const text = customTlk.getString(customIndex);
    if (text) return text;
    // Fallback to empty - custom TLK has sparse entries
    return '';
  }
  return baseTlk.getString(strref);
}
```

### Pattern 4: 2DA V2.0 Text Parsing
**What:** Parse the NWN 2DA (two-dimensional array) text format.
**When to use:** Every .2da resource from nwsync or BIF extraction.
**Example:**
```typescript
// Source: [VERIFIED: actual 2DA files from nwsync]
// Format:
//   Line 1: "2DA V2.0"
//   Line 2: empty
//   Line 3: column headers (space-separated)
//   Line 4+: row index + values (space-separated)
// Special values:
//   "****" = null/empty
//   Quoted strings: values containing spaces are quoted
// Whitespace: columns are space-padded for alignment

interface TwoDaTable {
  columns: string[];
  rows: Map<number, Record<string, string | null>>;
  getCell(row: number, column: string): string | null;
}

function parseTwoDa(content: string): TwoDaTable {
  const lines = content.split('\n')
    .map(l => l.trimEnd())
    .filter(l => l.length > 0);
  
  // Skip "2DA V2.0" header and blank line
  // Find column header line (first non-empty line after header)
  const headerLine = lines[1]; // After "2DA V2.0" and blank
  const columns = headerLine.trim().split(/\s+/);
  
  const rows = new Map<number, Record<string, string | null>>();
  
  for (let i = 2; i < lines.length; i++) {
    const parts = splitTwoDaLine(lines[i]);
    if (parts.length < 2) continue;
    
    const rowIndex = parseInt(parts[0], 10);
    const record: Record<string, string | null> = {};
    
    for (let c = 0; c < columns.length; c++) {
      const value = parts[c + 1] ?? '****';
      record[columns[c]] = value === '****' ? null : value;
    }
    
    rows.set(rowIndex, record);
  }
  
  return {
    columns,
    rows,
    getCell(row, column) {
      return rows.get(row)?.[column] ?? null;
    }
  };
}
```

### Pattern 5: nwsync Resource Resolution
**What:** Query the nwsync SQLite databases to find and decompress a specific resource.
**When to use:** Loading any 2DA or TLK from the Puerta server snapshot.
**Example:**
```typescript
// Source: [VERIFIED: actual nwsync database schema]
import Database from 'better-sqlite3';

const PUERTA_MANIFEST_SHA1 = 'cf6e8aad5751930e345266b84a3be31d9d67f3b1';

function loadNwsyncResource(
  metaDb: Database.Database,
  dataDb: Database.Database,
  resref: string,
  restype: number, // 2017 = 2DA, 2018 = TLK
): Buffer | null {
  const meta = metaDb.prepare(`
    SELECT resref_sha1 FROM manifest_resrefs
    WHERE manifest_sha1 = ? AND resref = ? AND restype = ?
  `).get(PUERTA_MANIFEST_SHA1, resref, restype) as { resref_sha1: string } | undefined;
  
  if (!meta) return null;
  
  const data = dataDb.prepare(
    'SELECT data FROM resrefs WHERE sha1 = ?'
  ).get(meta.resref_sha1) as { data: Buffer } | undefined;
  
  return data?.data ?? null;
}
```

### Anti-Patterns to Avoid
- **Using pre-extracted 2DA files as source of truth:** The files in `server-extract/` are from an older snapshot with different TLK IDs (16780706+ range vs current 16779216+ range), different skill labels (English vs Spanish), and different counts. Always read directly from nwsync. [VERIFIED: confirmed by comparing nwsync vs pre-extracted files]
- **Treating `****` as a string value:** In 2DA format, `****` means null/absent. The parser must convert it to `null`, not store it as a literal string.
- **Assuming all TLK entries are populated:** The custom TLK (pb_tlk_v6) has 192,606 entries but is sparsely populated. Many entries have flags=0 and length=0. The parser must handle empty entries gracefully.
- **Assuming contiguous row indices in 2DA:** Some 2DA files have gaps in row indices (DELETED rows in racialtypes.2da). The parser must handle non-contiguous indices.
- **Hard-coding column positions:** 2DA columns should be resolved by name from the header line, never by position index. The nwsync version may have different column ordering or additional/removed columns vs pre-extracted files.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SQLite database access | Custom SQLite C bindings | better-sqlite3 | Stack-mandated, prebuilt binaries, synchronous API |

| TypeScript file generation | Template strings with manual escaping | A structured emitter function | Need consistent formatting, proper escaping of Spanish characters |

**Key insight:** The binary formats (TLK V3.0, BIF V1.0, KEY V1.0) are simple enough to parse in <100 lines each. The complexity is in the data assembly and cross-referencing, not in the parsing. Do not pull in a full NWN file format library when the formats are this straightforward.

## Common Pitfalls

### Pitfall 1: Stale Pre-Extracted Data
**What goes wrong:** Using the `server-extract/` 2DA files as the data source instead of nwsync. The pre-extracted files have English labels, different TLK IDs, and potentially outdated class/feat/spell counts.
**Why it happens:** The pre-extracted files look convenient and are already text. But they are from an older server snapshot.
**How to avoid:** Always read from nwsync. Use pre-extracted files only as a validation cross-reference.
**Warning signs:** If extracted data has English skill labels like "AnimalEmpathy" instead of Spanish "TratoConAnimales", you are reading the wrong source.

### Pitfall 2: TLK String Encoding
**What goes wrong:** Spanish characters like accents (a, e, i, n) render as garbage or replacement characters.
**Why it happens:** NWN1 TLK files use Windows-1252 (Latin-1) encoding, not UTF-8. The base game dialog.tlk and the custom pb_tlk_v6 both use Latin-1.
**How to avoid:** Decode TLK string data as `latin1` (which is a subset of Windows-1252 sufficient for Spanish). Convert to UTF-8 only in the compiled output.
**Warning signs:** Characters like `ó` appearing as `Ã³` (UTF-8 misinterpretation) or `?` (encoding loss).

### Pitfall 3: Column Alignment Assumptions in 2DA
**What goes wrong:** Parser breaks on lines where values contain spaces or where column alignment shifts.
**Why it happens:** 2DA format uses whitespace-delimited columns. Some values are quoted (though rare in NWN1). The nwsync version uses different padding than pre-extracted files.
**How to avoid:** Split on whitespace runs. Handle the edge case where the column count in a data row may differ from the header count (trailing `****` values can be omitted).
**Warning signs:** Rows with fewer values than columns, or values shifted to wrong columns.

### Pitfall 4: Two Manifest Problem
**What goes wrong:** Extracting from the wrong server manifest. The nwsync database contains TWO manifests: Puerta de Baldur (`cf6e8aad5751930e...`) and Arelith (`cb6bd668da518e3c...`).
**Why it happens:** The user plays on both servers. Both manifests share the same nwsyncdata_0.sqlite3 blob storage.
**How to avoid:** Always filter by the Puerta manifest SHA1 when querying manifest_resrefs. The SHA1 should be configurable but default to the known Puerta value.
**Warning signs:** Getting 754 2DA files instead of 338 (754 = combined from both servers).

### Pitfall 5: Subraces Not in Puerta nwsync
**What goes wrong:** The extractor fails to find subraces.2da in the Puerta manifest.
**Why it happens:** The `subraces` 2DA is NOT in the Puerta nwsync manifest. It is only in the Arelith manifest (verified). Puerta's subraces.2da must come from the base game BIF files or from a manual override.
**How to avoid:** Implement the BIF/KEY fallback path for resources missing from nwsync. The base game `base_2da.bif` (indexed by `nwn_base.key`) contains the default subraces.2da.
**Warning signs:** `subraces` query returning null from nwsync.

### Pitfall 6: cls_spgn/cls_spkn Table Count Discrepancy
**What goes wrong:** Assuming all spellcasting classes have spell gain/known tables in nwsync.
**Why it happens:** Only 11 of the spell-related tables are in the Puerta nwsync (e.g., cls_spgn_pal, cls_spkn_alma). Others (cls_spgn_bard, cls_spgn_cler, cls_spgn_wiz, etc.) must come from the base game.
**How to avoid:** For each class with SpellGainTable/SpellKnownTable references in classes.2da, first check nwsync, then fall back to base game BIF extraction.
**Warning signs:** SpellCaster classes with null spell gain tables.

### BIF V1.0 Header Format (for base-game fallback)

```typescript
// Source: [VERIFIED: xoreos source code analysis]
// BIF V1.0 Header:
//   0-3:   "BIFF" magic
//   4-7:   "V1  " version
//   8-11:  variable resource count (uint32 LE)
//   12-15: fixed resource count (uint32 LE, always 0)
//   16-19: variable resource table offset (uint32 LE)
//
// Variable Resource Table Entry (16 bytes each):
//   0-3:   resource ID (uint32 LE) - ignored
//   4-7:   data offset (uint32 LE)
//   8-11:  data size (uint32 LE)
//   12-15: resource type (uint32 LE)

// KEY V1.0 Header:
//   0-3:   "KEY " magic
//   4-7:   "V1  " version
//   8-11:  BIF count (uint32 LE)
//   12-15: key count (uint32 LE)
//   16-19: file table offset (uint32 LE)
//   20-23: key table offset (uint32 LE)
//   24-55: (build year/day + reserved, skip)
//
// BIF File Table Entry:
//   0-3:   file size (uint32 LE)
//   4-7:   filename offset (uint32 LE)
//   8-9:   filename size (uint16 LE)
//   10-11: drives (uint16 LE, ignored)
//
// Key Table Entry (22 bytes each):
//   0-15:  resource name ResRef (16 bytes, null-padded ASCII)
//   16-17: resource type (uint16 LE)
//   18-21: resource ID (uint32 LE)
//     - BIF index = (id >> 20) & 0xFFF
//     - Resource index within BIF = id & 0xFFFFF
```
Source: [VERIFIED: xoreos source code keyfile.cpp and biffile.cpp]

## Open Questions

1. **Deity catalog source**
   - What we know: No `deities.2da` or equivalent was found in nwsync or pre-extracted files. Domains.2da contains domain data but no deity-to-domain mapping.
   - What's unclear: Where does the Puerta server store deity data? It may be script-only or defined in the module.
   - Recommendation: Start without a deity catalog. Document it as a gap. The planner can hard-code Puerta deities from known server documentation as a manual override.

2. **Subraces.2da source for Puerta**
   - What we know: subraces.2da is NOT in the Puerta nwsync manifest. The pre-extracted version has Puerta-specific entries (GS_SU_PLANETOUCHED_AASIMAR, etc.) but those came from an older snapshot. The Arelith manifest has a subraces.2da but that is Arelith's data.
   - What's unclear: Whether the base game subraces.2da is sufficient or if Puerta has custom subraces only in an older nwsync snapshot.
   - Recommendation: Extract from base game BIF as fallback. Cross-reference with the pre-extracted file. Mark subrace data as potentially incomplete and flag for user verification.

3. **cls_spgn/cls_spkn tables missing from nwsync**
   - What we know: Only 11 spell gain/known tables are in nwsync. Base classes like Bard, Cleric, Wizard use base game tables (referenced by name in classes.2da but not overridden in nwsync).
   - What's unclear: Exact count of spell tables needed from base game BIF.
   - Recommendation: Implement BIF/KEY extraction as a fallback for any referenced table not found in nwsync. The classes.2da `SpellGainTable` and `SpellKnownTable` columns name the required tables.

4. **NWN1 TLK text encoding**
   - What we know: Latin-1 decoding produces correct Spanish characters for most text. Some entries contain NWN color tags like `<cÿÿ@>`.
   - What's unclear: Whether any entries use actual Windows-1252 characters not in the Latin-1 subset (code points 0x80-0x9F differ between the two).
   - Recommendation: Use Latin-1 for initial implementation. If character corruption is found, switch to Windows-1252 (Node.js can decode this via `TextDecoder('windows-1252')`).
