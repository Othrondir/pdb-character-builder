## Decisions Made
- **Composable assembler inputs:** Each assembler accepts pre-built dependency maps (classRows, featIdsByRow, spellIdsByRow) rather than loading everything internally. This keeps assemblers independently testable and avoids circular dependencies.
- **Spell column name mapping:** The spell assembler accepts a `spellColumnName` field per class to map spells.2da columns (Bard, Cleric, Wiz_Sorc, etc.) to canonical class IDs. This avoids hardcoding the column-to-class mapping inside the assembler.
- **Zod v4 enum-key record behavior:** When using `z.record(z.enum([...]), z.number())`, Zod v4 validates that ALL enum keys are present. Spell gain/known tables default missing levels to 0, and domain spellIds default missing levels to [].
- **Feat duplicate ID detection:** Feat canonical IDs are generated from the LABEL column. If a collision occurs (unlikely but possible with Puerta customizations), the sourceRow is appended to ensure uniqueness.

### Auto-fixed Issues

**1. [Rule 3 - Blocking] JSDoc comment with `*/` in text confuses esbuild**
- **Found during:** Task 2 (test execution)
- **Issue:** The JSDoc comment `cls_spgn_*/cls_spkn_*` contained a `*/` sequence that esbuild interpreted as closing the comment block, causing a parse error.
- **Fix:** Removed the `*` suffix from the table names in the JSDoc comment.
- **Files modified:** packages/data-extractor/src/assemblers/spell-assembler.ts
- **Committed in:** 3db7e97

**2. [Rule 3 - Blocking] Cannot mix `||` and `??` without parentheses**
- **Found during:** Task 2 (test execution)
- **Issue:** `SCHOOL_MAP[schoolRaw] ?? schoolRaw || 'unknown'` fails esbuild due to operator precedence ambiguity.
- **Fix:** Added parentheses: `SCHOOL_MAP[schoolRaw] ?? (schoolRaw || 'unknown')`.
- **Files modified:** packages/data-extractor/src/assemblers/spell-assembler.ts
- **Committed in:** 3db7e97

**3. [Rule 1 - Bug] Zod v4 record with enum keys requires all keys present**
- **Found during:** Task 2 (integration test execution)
- **Issue:** Spell gain/known tables and domain spellIds only populated keys with actual data. Zod v4's `z.record(z.enum([...]), ...)` validates all enum keys exist, causing "expected number/array, received undefined" errors.
- **Fix:** Populate all 10 spell level keys (0-9) with defaults: 0 for spell gain/known numbers, [] for domain spell ID arrays.
- **Files modified:** packages/data-extractor/src/assemblers/spell-assembler.ts, packages/data-extractor/src/assemblers/domain-assembler.ts
- **Committed in:** 3db7e97

**4. [Rule 3 - Blocking] Test file extension .spec.ts instead of plan's .test.ts**
- **Found during:** Task 1 (test creation)
- **Issue:** Vitest config only includes `tests/**/*.spec.ts`; `.test.ts` would not be discovered
