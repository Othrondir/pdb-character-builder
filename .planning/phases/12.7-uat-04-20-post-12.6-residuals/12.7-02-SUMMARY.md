# Phase 12.7 Plan 02: canIncrementSkill gate (F4 R2 skill over-allocation fix) Summary

**One-liner:** Pure `canIncrementSkill(skillId, level, snapshot)` + `nextIncrementCost` predicate pair in rules-engine, wired to the skill `+` button `disabled` prop via a planner-side boundary adapter — closes UAT F4 (user could drive Puntos restantes to -1 via unchecked `+` clicks).

## What shipped

1. **NEW** `packages/rules-engine/src/skills/skill-budget.ts` — 85 lines, zero imports, mirrors the Phase 12.3-01 `canIncrementAttribute` + `nextIncrementCost` doc-block + signature shape verbatim.
   - `export interface SkillBudgetSnapshot { pointsAvailable, pointsSpent, skills: Record<string, { costPerRank, currentRank, maxAssignableRank }> }`
   - `export function nextIncrementCost(skillId, level, snapshot): number | null` — returns `costPerRank` when under cap; `null` when skillId unknown OR `currentRank >= maxAssignableRank`. `level` param reserved for F5 carryover (Phase 12.8).
   - `export function canIncrementSkill(skillId, level, snapshot): boolean` — delegates to `nextIncrementCost`; returns `false` on `null` OR when `pointsSpent + cost > pointsAvailable`.

2. **MODIFIED** `apps/planner/src/features/skills/selectors.ts` — added type import + boundary adapter `buildSkillBudgetSnapshotFromSheet(activeSheet): SkillBudgetSnapshot` at end of file. Iterates `activeSheet.groups[*].rows[*]` and maps `row.costType === 'class'` to `costPerRank: 1` else `2`.

3. **MODIFIED** `apps/planner/src/features/skills/skill-sheet.tsx` — three surgical edits:
   - Imported `canIncrementSkill` + `SkillBudgetSnapshot` from rules-engine; imported `buildSkillBudgetSnapshotFromSheet` from `./selectors`.
   - `SkillRankRow` prop contract expanded from `{ level, row }` to `{ level, row, snapshot: SkillBudgetSnapshot }`.
   - `SkillSheet()` composes `snapshot = buildSkillBudgetSnapshotFromSheet(activeSheet)` once per render, threads as `<SkillRankRow snapshot={snapshot} />` prop.
   - `+` button `disabled` expression extended from 2 clauses to 3: `row.disabled || row.currentRank + row.step > row.maxAssignableRank || !canIncrementSkill(row.skillId, level, snapshot)`.

4. **NEW** `tests/phase-12.7/skill-budget-gate.spec.ts` — 162 lines, pure-node Vitest env, 13 tests across 3 `describe` blocks mirroring 12.3-01 structure:
   - Block 1: `nextIncrementCost` — cost=1 class, cost=2 cross-class, null at cap, null for unknown skill.
   - Block 2: `canIncrementSkill` — blocks at cap, allows exactly-enough, blocks when cost would push over, blocks at per-rank cap, fail-closed for unknown skill.
   - Block 3: UAT F4 repro — 4/4-spent snapshot blocks every skill; 3/4 spent allows class skills, blocks cross-class.

5. **NEW** `tests/phase-12.7/skill-sheet-disabled-gate.spec.tsx` — 204 lines, jsdom env, 3 suites:
   - B1: Humano+Clérigo L1 → spend full budget on class skills → every `+` button disabled.
   - B2: Sanity — before full spend, at least one `+` button is enabled (prevents regression to "always disabled").
   - B3: Belt-and-braces — `"supera los límites permitidos"` text never reachable via UI.

### Minor Plan/Runtime Clarifications (documented, not deviations)

1. **Humano+Clérigo L1 budget size** — Plan references "4 pts budget"; live runtime with default INT baseScore=8 yields 8 pts: `max(1, (2+1) + (-1)) × 4 = 8`. The UAT DOM-evidence screenshot ("Puntos disponibles: 4") must have been captured with INT manually lowered, OR with a non-Humano race. The RTL spec reads the live budget from the DOM (`<dd>` inside the `Puntos disponibles` `<dl>`) and spends exactly that many ranks — gate logic is independent of specific budget size (what matters is the invariant "spend N = budget → all + disabled"). Not a code deviation; just a fixture-arithmetic note for future authors.

2. **Adapter field mapping confirmed, no renames** — Plan Task 2 action step 2 flagged `availablePoints` vs `pointsAvailable`, `spentPoints` vs `pointsSpent`, `costType` vs `costTypeKind` as potential divergence points. Grepping `ActiveSkillSheetView` + `SkillSheetRowView` confirmed the live names are `availablePoints`, `spentPoints`, `costType` (with values `'class' | 'cross-class'`). No adapter rename was required; the plan's pseudo-code matched reality.

3. **Disabled expression line number shifted** — Plan anchors at `skill-sheet.tsx:77` (the pre-12.7-02 location of the `+` `disabled` expression). Post-edit it moved to `skill-sheet.tsx:87` because the import block expanded (canIncrementSkill + SkillBudgetSnapshot + buildSkillBudgetSnapshotFromSheet added 8 lines). The extended expression is on the correct `+` button (the `Aumentar` one); no other disabled expressions touched.

4. **Acceptance grep `buildSkillBudgetSnapshotFromSheet` count** — Plan expects `grep -c "buildSkillBudgetSnapshotFromSheet" selectors.ts` ≥ 2 ("export + maybe type or use"). Actual count = 1 (declaration only). The second reference lives in `skill-sheet.tsx` (import + call site). The intent of the grep — "function is exported and referenced" — is satisfied end-to-end (typecheck passes, integration spec passes). `SkillBudgetSnapshot` itself is referenced 4× in selectors.ts (import + JSDoc + parameter return type + inner `['skills']` type) — covering the "type" side of "export + maybe type or use".
