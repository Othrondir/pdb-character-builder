# Phase 12.7: UAT-2026-04-20 Post-12.6 Residuals — Specification

**Created:** 2026-04-20
**Ambiguity score:** 0.17
**Requirements:** 6 locked

## Goal

Close 6 of 7 findings (F1, F2, F3, F4, F6, F7) surfaced by agent-driven UAT against master 2026-04-20 immediately after Phase 12.6 closure, restoring the user's ability to advance from L1 → L2 (currently a hard blocker because `LevelEditorActionBar` mounts only inside the Progresión-Clase sub-step host) and gating skill over-allocation at the `+` button site.

## Background

Phase 12.6 closed at 10/10 UAT pass against its own success criteria, but a follow-up agent-driven probe (`.planning/UAT-FINDINGS-2026-04-20-post-12.6.md`, commit `34b84db`) surfaced 7 emergent issues:

- **F7 BLOCKER**: `apps/planner/src/features/level-progression/level-editor-action-bar.tsx` is rendered as the last child of `<LevelProgressionRow>`'s expanded slot (Plan 12.4-09 → 12.6-04 host swap). When the user navigates to `Habilidades` or `Dotes` sub-steps via `creation-stepper.tsx`, the BuildProgressionBoard host unmounts and the action bar disappears. User cannot find the advance affordance and cannot reach L2.
- **F4 major**: Skill `+` buttons in the Habilidades sub-step do not enforce `puntosGastados <= puntosDisponibles`. Verified DOM state: `Puntos disponibles: 4 / Puntos gastados: 5 / Puntos restantes: -1`. Mirrors the pre-12.3-01 attribute over-allocation defect.
- **F2 major**: Habilidades sub-step opens scrolled mid-list on first mount per session (likely auto-focus / `scrollIntoView` side-effect on the first `+` button).
- **F1, F3 minor**: Visual polish — row pills jam without separators (`1Clérigo0/1 dotes5/4 pts⚠`); per-row `Clase`/`Transclase` labels duplicate section headers.
- **F6 minor**: L1 `(class-base + INT-mod) × 4` formula verified at runtime (Guerrero 8 / Clérigo 4); L2..L20 `class-base + INT-mod` (no ×4) is unverified.

F5 (skill point carryover ≤4 to next level) is **deferred to a future Phase 12.8** — data-blocked on Puerta server source for the carryover rule.

## Requirements

1. **R1 (F7) — Stepper-global advance bar**: `LevelEditorActionBar` is visible on every Progresión sub-step (Clase, Habilidades, Dotes) for the active level.
   - Current: `apps/planner/src/features/level-progression/level-progression-row.tsx` mounts `<LevelEditorActionBar>` only inside the `{isActive && ...}` expanded slot of the active row. When `creation-stepper.tsx` switches sub-step away from `Clase`, the host unmounts and the bar disappears.
   - Target: `LevelEditorActionBar` is hoisted to a stepper-global mount (e.g., bottom of `creation-stepper.tsx` or `center-content.tsx`) so it stays visible across Clase / Habilidades / Dotes for the active Progresión level. Same `selectLevelCompletionState + computeAdvanceLabel` driver. Same `setActiveLevelSubStep('class') + setActiveLevel(N+1) + setExpandedLevel(N+1)` atomic dispatch on enabled click. Returns `null` at L16 (preserved L20 cap from Phase 06).
   - Acceptance: Agent-driven MCP Chrome probe asserts `document.querySelector('[data-testid=level-editor-action-bar]') !== null` on each of the three Progresión sub-steps for L1 with Humano + Guerrero fixture. Probe asserts the same `[data-testid^="advance-to-level-"]` element survives sub-step transitions without remount churn (stable React key).

2. **R2 (F4) — Skill `+` button gate**: `+` button hard-blocks when allocation would exceed per-level `puntosDisponibles`.
   - Current: Habilidades view allows `+` clicks past budget; UI surfaces "El reparto de este nivel supera los límites permitidos" callout post-hoc. Verified DOM: 5/4 spent.
   - Target: New pure helper `canIncrementSkill(skillId, level, snapshot)` in `packages/rules-engine/src/skills/` (or nearest existing module). Wired to `disabled` prop on every `+` button in the Habilidades skill list. Mirror exact pattern from Phase 12.3-01 `canIncrementAttribute`.
   - Acceptance: With Humano + Clérigo L1 (4 pts budget), spending 4 ranks across any combination of skills disables every remaining `+` button. Vitest unit test asserts `canIncrementSkill` returns `false` when `pointsSpent >= pointsAvailable`. Agent-driven probe asserts `document.querySelectorAll('button[aria-label*="incrementar"]:not(:disabled)').length === 0` after spending full budget.

3. **R3 (F2) — Habilidades sub-step scroll reset**: Inner scroll container starts at `scrollTop === 0` on first mount.
   - Current: `selection-screen__content` (or equivalent host) lands mid-list on Habilidades entry, likely triggered by default browser focus / `scrollIntoView` on the first `+` button.
   - Target: Mount-time effect explicitly forces `scrollContainer.scrollTop = 0` after Habilidades sub-step renders. OR remove the implicit auto-focus that triggers the scroll.
   - Acceptance: Agent-driven probe navigates Atributos → Habilidades; asserts `scrollContainer.scrollTop === 0` immediately after mount. Vitest RTL spec mounts Habilidades sub-step in jsdom and asserts no `scrollIntoView` call on any child within 50ms of mount.

4. **R4 (F3) — Drop redundant per-row category labels**: Skill rows omit `Clase`/`Transclase` text; `Solo entrenada` badge preserved.
   - Current: Each skill row prints `· Clase ·` or `· Transclase ·` between the skill name and the `+/-` buttons, even though the parent section header (`HABILIDADES DE CLASE` / `HABILIDADES TRANSCLASE`) already segments rows by category.
   - Target: Remove the per-row category text node. Keep `Solo entrenada` (per-row metadata not implied by section). No CSS-token churn, no layout-grid changes — text-deletion only.
   - Acceptance: Vitest RTL spec asserts `getAllByText('Clase')` and `getAllByText('Transclase')` return 0 inside the skill row list (only section headers contain these tokens). `getAllByText('Solo entrenada')` count unchanged from baseline.

5. **R5 (F1) — Level-progression-row header text legibility**: Row pills render with explicit separators and an explicit "Nivel N" prefix.
   - Current: `[data-level-row][data-level="1"] button.textContent === "1Clérigo0/1 dotes5/4 pts⚠"` — pills jam together without separators or the word "Nivel".
   - Target: `apps/planner/src/features/level-progression/level-progression-row.tsx` header renders pill copy with ` · ` separators (or equivalent visual spacing) AND prefixes the level number with "Nivel" — e.g. `Nivel 1 · Clérigo · 0/1 dotes · 5/4 pts · ⚠`. Visual separation may use CSS `gap` on a flex container instead of text separators, provided the rendered string parses with the same legibility.
   - Acceptance: Agent-driven probe asserts `[data-level-row][data-level="1"] button.textContent` matches `/Nivel 1.+Clérigo.+\d+\/\d+ dotes.+\d+\/\d+ pts.+[⚠✓✗🔒]/` (separators implied by `.+` matching whitespace or punctuation between tokens). Vitest RTL spec snapshots the header for L1 + L8 (locked) + L16 (terminal) and asserts the regex pattern.

6. **R6 (F6) — L2..L20 skill budget formula verification**: Confirm subsequent levels apply `class-base + INT-mod` (no ×4 multiplier).
   - Current: L1 ×4 verified at runtime (Guerrero 8 / Clérigo 4 match); L2..L20 unverified.
   - Target: Verify-only spike. If L2..L20 budget renders correctly, close as no-op. If broken, defer fix to Phase 12.8 (do NOT expand 12.7 scope).
   - Acceptance: Vitest unit spec asserts `computePerLevelBudget` skill-points field returns `class-base + INT-mod` (clamped to floor of 1) for L2..L20 across all 11 base classes at INT 8/10/12/14/16/18. Agent-driven probe at L2 with Humano + Guerrero fixture asserts the row pill `skills:N/M pts` matches the expected formula.
