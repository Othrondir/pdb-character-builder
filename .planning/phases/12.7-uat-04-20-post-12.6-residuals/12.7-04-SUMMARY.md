## Decisions Made

- **`raceId: null` instead of `race:human`** — would otherwise add `HUMAN_SKILL_POINT_PER_LEVEL = 1` per-level bonus (per-level-budget.ts L71–L73, L210). That canon is already locked by `tests/phase-12.4/per-level-budget.fixture.spec.ts` "Humano +1 SP/level canon (A2)" describe block. Testing it again here would conflate two orthogonal invariants and make a GREEN/RED signal harder to read.
- **`intAbilityIncreasesBeforeLevel: () => 0`** — strips mid-level INT bumps so each assertion exercises exactly `Math.floor((intScore - 10) / 2)`. Mid-level bumps are locked by the Pícaro L12 case in the phase-12.4 fixture; re-covering them here would explode to 7,524 assertions for no additional signal.
- **Catalog-read `classBase`** — T-12.7-04-02 mitigation. The spec description includes `(classBase=N)` in the describe label so the Vitest UI surfaces the actual catalog value; if Puerta customization changes `skillPointsPerLevel`, the describe label updates with zero code edits and the assertion shape still holds.
- **No `vitest.config.ts` edit yet** — plan D-15 mentioned adding `tests/phase-12.7/**/*.spec.tsx` to `environmentMatchGlobs` for jsdom. This plan ships only a pure-node `.spec.ts`, so no config change is needed here. The jsdom glob entry will land with 12.7-03 (R3/R5 RTL specs) where it's actually needed — keeps this plan's blast radius at a single file.
