# Phase 12.3 Plan 02: Multiclass Active-Level Switching Summary

Fixed UAT B2 (CRITICAL) + B8 (Medium) + B9 (Medium) by dispatching `setActiveLevel` alongside `setExpandedLevel` in `LevelRail.onClick`, wiring a one-line repair that closes three user-visible blockers at once via the existing selector chain.

## What Shipped

### Root Cause

Phase 4 built the level-progression store with a first-class `activeLevel: ProgressionLevel` field and a `setActiveLevel` setter, and `selectActiveLevelSheet` already reads `progressionState.activeLevel`. However, `LevelRail.onClick` only called `setExpandedLevel` on the shell store — so `progressionState.activeLevel` was stranded at its default value of `1` forever. Every class pick at the board went through `setLevelClassId(activeSheet.level, id)` where `activeSheet.level === 1` regardless of which rail button the user clicked. The header title read `NIVEL 1` forever. The Clase sub-step ✓ evaluated L1's state globally.

### Fix

`apps/planner/src/components/shell/level-rail.tsx` (`+5 / -1`):

```tsx
// Added: selector hook for the progression store's setter
const setActiveLevel = useLevelProgressionStore((state) => state.setActiveLevel);

// Changed: onClick dispatches BOTH setters atomically
onClick={() => {
  setActiveLevel(entry.level as ProgressionLevel);
  setExpandedLevel(entry.level as ProgressionLevel);
}}
```

No changes to:
- `apps/planner/src/features/level-progression/store.ts` (setter was already correct).
- `apps/planner/src/features/level-progression/selectors.ts` (selector already reads `progressionState.activeLevel`).
- `apps/planner/src/features/level-progression/build-progression-board.tsx` (title concatenation was already correct).
- `apps/planner/src/lib/copy/es.ts` (copy string `'Selecciona la clase del nivel'` has no hardcoded number).

### Regression Spec

`tests/phase-12.3/multiclass-active-level.spec.tsx` — 8 assertions across 4 suites:

- **Suite A (rail dispatch, 3 tests):** clicking level 2/5/7 radio buttons must update BOTH `progression.activeLevel` AND `shell.expandedLevel`. Successive clicks flip independently. All 3 were RED before the fix; all GREEN after.
- **Suite B (B2 no-overwrite lock, 2 tests):** Guerrero@L1 + Pícaro@L2 persists per-level; higher levels untouched. UAT repro (Humano + Guerrero + click L2 + Pícaro) keeps both. Passed on the store layer already; ships as forward-looking lock.
- **Suite C (B8 title binding, 2 tests):** `BuildProgressionBoard` header renders `nivel 1` initially; flipping `activeLevel` to 3 updates the heading to `nivel 3`. Uses `getByRole('heading', { name: /nivel N/i })` — case-insensitive since the copy is lowercase `'Selecciona la clase del nivel'`.
- **Suite D (B9 selector follow-through, 1 test):** `selectActiveLevelSheet.{level,classId}` tracks `activeLevel` across flips.
