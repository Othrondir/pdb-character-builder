## Implementation Decisions

### D-01 — Attributes overspend gate (B1, Critical)
- **Decision**: In `apps/planner/src/features/character-foundation/attributes-board.tsx`, compute the next-increment cost before rendering the `+` button. Disable the button when `remainingPoints - nextCost < 0` OR `baseValue >= maximum`. Cost table lookup reuses `FOUNDATION_POINT_BUY_COSTS` (already in `foundation-fixture.ts:208-220`) via a pure helper in `packages/rules-engine/src/foundation/` or inline if trivial.
- **Rationale**: Minimal surgical fix. Store-side prevention is tempting but harder; disabling at the UI surface matches how the `-` button already gates on `baseValue <= minimum`. Store already rejects invalid reductions; mirror that for increments.
- **Expected touchpoints**: `attributes-board.tsx` (+ helper in rules-engine/foundation).
- **Regression test**: pure helper unit test + store integration test asserting setter clamps OR selector reports `canIncrement(key) === false` when cost > remaining.

### D-02 — Multiclass per-level switching (B2, B8, B9 — Critical)
- **Decision**: Introduce an `activeLevelIndex` field in `apps/planner/src/features/level-progression/store.ts` (1-based, default 1). The level-rail button click sets `activeLevelIndex = N`. The class-picker selector reads `activeLevelIndex` and returns options for that level's context. Picking a class writes `classLevels[N-1].classId` (not always L1). Header title pulls `activeLevelIndex` to render `NIVEL {N}`. The `Clase` sub-step completeness indicator evaluates per-level, not milestone-wide.
- **Rationale**: Today the progression rail is a read-only viewer — levels 2..16 show slots but clicks do not navigate. Wiring `activeLevelIndex` through the store is the one change that unblocks multiclass end-to-end. B8 and B9 are pure byproducts: header + sub-step checks read the same active-level state.
- **Expected touchpoints**: `level-progression/store.ts` (new field + setter), `level-progression/selectors.ts` (read `activeLevelIndex`, return per-level class options + per-level completion), `level-progression/build-progression-board.tsx` (dispatch `setActiveLevel` on rail click + bind title to active level), progression-rail component if separate.
- **Regression test**: unit test on selector — setting `activeLevelIndex = 2`, calling `selectClassOptionsForLevel` returns `{ level: 2 }`, picking Pícaro persists `classLevels[1].classId === 'class:rogue'` while `classLevels[0].classId` stays unchanged. UI test asserting header binds to active level.

### D-03 — Per-level Dotes gate + slot count prompt (B3, B4 — Critical + High)
- **Decision**: FeatBoard selector (`apps/planner/src/features/feats/selectors.ts`) evaluates "valid progression" per active level, not across all 1..16 levels. When the active level has a class, the board renders even if later levels are empty. Surface `slotPrompt: "N dotes disponibles — N restantes"` (format derived from `FeatSlotsAtLevel`); render it above the feat list.
- **Rationale**: B3's block condition currently trips on global progression emptiness. Scoping the gate to the active level alone matches user intent (editing one level at a time). B4 is a straight render-passthrough once the selector exposes slot counts.
- **Expected touchpoints**: `feats/selectors.ts` (revise block condition, add `slotPrompt`), `feats/feat-board.tsx` (render prompt), tests under `tests/phase-12.3/`.
- **Regression test**: fixture-driven — L1 Guerrero only, active level 2 has no class → block message shows for L2, NOT for L1. Slot prompt appears when L1 active with class.

### D-04 — Hit-points pipeline (B6, High)
- **Decision**: Implement a pure HP selector in `packages/rules-engine/src/progression/` (e.g., `compute-hit-points.ts`) that takes `classLevels[]` + `conModifier` and returns `hitPoints`. L1 class = `hitDie + conMod` (max roll). Later levels = `hitDie/2 + 1 + conMod` per NWN convention OR a decision to always take max (simpler, server-consistent). Selector wires into `character-sheet.tsx` replacing the `--` placeholder.
- **Rationale**: Scoping: NWN1 char planners typically show max HP per level (no rolls). Server Puerta likely follows same — verify against `CompiledClass.hitDie` field. Minimum viable: class hit die for L1, average or max for subsequent.
- **Expected touchpoints**: new `compute-hit-points.ts` in rules-engine/progression + barrel export, `character-sheet.tsx` consumer, tests.
- **Regression test**: unit test on selector — Guerrero L1 CON +2 → 12 HP (d10 max + 2); Guerrero 2 CON +2 → 12 + 6 + 2 = 20 (average 5.5 rounded up to 6 + 2 per level). Exact formula locks once user confirms preference.
- **Note**: If HP formula is uncertain, default to `hitDie + (hitDie/2 + 1) * (level - 1) + conMod * level` (standard NWN1). Document decision in plan.

### D-05 — Origin-stepper decoupling (B7, High)
- **Decision**: The stepper/shell selector that computes "origin complete" currently reads a composite predicate that folds Atributos validity. Split the predicate: `isOriginStepComplete = hasRace && hasAlignment` (ignores atributos). Atributos has its own completeness predicate that stays separate.
- **Rationale**: Cosmetic rail regression — state itself is fine (race + alignment persist). Only the ✓ marker miscomputes. Decoupling predicates matches how each sub-step owns its own validity.
- **Expected touchpoints**: shell stepper selector (likely in `apps/planner/src/features/character-foundation/` or a shell-level selectors file), possibly `app/routes/construccion/*.tsx` if the predicate lives there. Scout during planning.
- **Regression test**: unit test on selector — set race + alignment, set atributos to overspend, assert `isOriginStepComplete === true` (only race+alignment matter).

### D-06 — Description paragraph rendering (B5, Medium)
- **Decision**: Apply `white-space: pre-wrap` to `.detail-panel__body` CSS class so `\n` characters render as line breaks. Alternative: split `body` prop in `DetailPanel` on `\n\n` and render as `<p>` array. Prefer CSS fix — zero logic change, works for both race + class descriptions + future DetailPanel uses.
- **Rationale**: Minimal fix. Source text already contains `\n` separators (compiled-races / compiled-classes). CSS change alone resolves rendering.
- **Expected touchpoints**: `apps/planner/src/components/ui/detail-panel.tsx` CSS block in `app.css` — search `.detail-panel__body` rule and add `white-space: pre-wrap;`.
- **Regression test**: none automated (CSS change); manual UAT in-browser.

### D-07 — Plan split (6 plans, 3 waves)
- **Decision**: Six plans mapping one bug (or tight cluster) each:
  - **12.3-01** (Wave 1, Small): B1 attributes overspend gate.
  - **12.3-02** (Wave 2, Medium): B2 + B8 + B9 multiclass active-level state. Ripples B8/B9 fold into this plan.
  - **12.3-03** (Wave 3, Medium): B3 + B4 per-level Dotes gate + slot prompt. Depends on 12.3-02 for per-level context.
  - **12.3-04** (Wave 1, Medium): B6 HP pipeline. File-disjoint from other waves.
  - **12.3-05** (Wave 1, Small): B7 origin-stepper decoupling.
  - **12.3-06** (Wave 1, Small): B5 description CSS.
- **Rationale**: Atomic per bug; blast radius contained. Wave 1 runs 4 plans in parallel (all file-disjoint: attributes-board, rules-engine/progression, stepper selector, app.css). Wave 2 runs 12.3-02 alone (touches level-progression store + selectors). Wave 3 runs 12.3-03 after multiclass lands.
- **Planner's job**: verify file-disjointness per wave; downgrade if overlap surfaces.

### D-08 — Regression test posture
- **Decision**: TDD RED → GREEN per plan under `tests/phase-12.3/`. Each bug gets a falsifiable spec:
  - B1: store/selector test asserting `canIncrement` returns false when cost > remaining + `+` button disabled state.
  - B2: level-progression selector test asserting active-level switching + per-level class persistence.
  - B3: feats selector test asserting per-level block message scope.
  - B4: FeatBoard render test asserting slot prompt appears with expected copy.
  - B5: no automated test (CSS).
  - B6: pure-selector test asserting HP formula for Guerrero L1 + L2 + multiclass L1 + L2 different classes.
  - B7: stepper selector test asserting origin completeness independent of atributos state.

<specifics>
## Specific Ideas

- **Blast radius discipline.** Each plan should NOT enlarge beyond its bug. Per-level selector changes in 12.3-02 can ripple into 12.3-03 — expect those ripples but contain them.
- **`activeLevelIndex` default = 1.** On first load, user is editing L1. Level clicks update the store; clearing the progression resets to 1.
- **HP formula decision.** If unclear, default to NWN1 canonical: L1 = max (hit die), L2+ = `floor(hit die / 2) + 1` (average rounded up). Con modifier applies to every level. This matches server behavior Puerta likely uses.
- **Atomic commit style.** Mirror prior phases: one `test(12.3-XX)` RED + one `fix(12.3-XX)` GREEN per plan, plus `docs(12.3-XX): write plan summary`.
- **Final UAT re-run.** After all 6 plans ship, re-run the Phase 11 + 12.1 + 12.2 browser UAT to confirm no regressions; then manually verify each of the 9 bugs is gone. This is the milestone-v1.0-close gate.
- **Milestone close blocker.** Do NOT attempt `/gsd-complete-milestone` until 12.3-VERIFICATION.md records 6/6 SC PASS.
</specifics>
