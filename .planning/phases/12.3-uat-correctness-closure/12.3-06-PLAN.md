The rule exists; it needs exactly ONE new declaration added: `white-space: pre-wrap;`.

CSS `white-space: pre-wrap` semantics:
- Preserves sequences of whitespace (so `\n\n` becomes a visible blank line — a paragraph gap).
- Wraps lines normally at word boundaries (so long single-line paragraphs still wrap to the container width; no horizontal overflow).
- Backward-compatible for single-paragraph descriptions (no `\n` = no added line breaks).

Source data verification:
- `apps/planner/src/data/compiled-classes.ts` line 10 (Bárbaro description): `"... legendaria.\n\n- Restricciones al alineamiento: ...\n- Dado de Golpe: d12.\n- Competencias: ..."`. The `\n\n` between the prose and the bullet list + the `\n-` starting each bullet are the intended paragraph + list breaks.
- `apps/planner/src/data/compiled-races.ts` — similar format for race descriptions.

Why this is CSS-only:
- CONTEXT D-06 locks: "Apply `white-space: pre-wrap` to `.detail-panel__body` CSS class so `\n` characters render as line breaks. Alternative: split `body` prop in `DetailPanel` on `\n\n` → `<p>` array. **Prefer CSS fix — zero logic change, works for both race + class descriptions + future DetailPanel uses.**"
- No React logic means no risk of breaking existing DetailPanel callers (feat descriptions, tooltip bodies, etc.).

Why no automated test:
- The source DOM has `\n` regardless of CSS; `getComputedStyle` in jsdom may not reliably reflect `white-space: pre-wrap` (jsdom's CSS engine is limited). Manual UAT via browser is the verification.
- Consistency with prior CSS-only fixes in the project (e.g., Phase 12.1-03 overflow fixes, which also relied on manual UAT).
