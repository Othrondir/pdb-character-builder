# Phase 12.6: UAT-2026-04-20 Residuals — Specification

**Created:** 2026-04-20
**Ambiguity score:** 0.16 (gate: ≤ 0.20)
**Requirements:** 6 locked

## Goal

Close the two UAT-2026-04-20 residuals that the hot-fix batch could not resolve:
(A1) point-buy cost curve is resolved per race from a Puerta snapshot override with fail-closed semantics; (P5) level progression presents all 20 levels on a single-screen scan surface with edit-in-place active level and per-level legality diff.

## Background

UAT 2026-04-20 surfaced 8 findings. Commits `e8181a5..9513fe6` closed P1 (ClassPicker mount) + G1 (sequential gating) + G2 (Utilidades removal) + A2 (race ability modifiers fold) + P2 (auto-advance to habilidades) + P3 (skill points per class × INT) + P4 (feat toggle) + P6 (L16 → L20). Two findings remain:

- **A1**: `packages/rules-engine/src/foundation/ability-budget.ts:34-63` takes `costByScore` as a parameter but `apps/planner/src/features/character-foundation/foundation-fixture.ts:205-223` ships a single uniform NWN1 curve (1:1 up to 14, 2:1 at 15-16, 3:1 at 17-18) applied to every race. No per-race data exists in compiled catalog, extractor, or snapshot.
- **P5**: `level-rail.tsx` shows buttons only; `level-sheet.tsx` shows the active-level editor only; user must click each L1..L20 rail button to inspect class + feats + skills — user described as "un desastre". Level range extended to 1..20 in P6 amplifies the scan problem.

## Requirements

1. **A1 — Snapshot override source**: Puerta point-buy curves ship as a hand-authored JSON snapshot keyed by canonical race id.
   - Current: No per-race curve. `foundation-fixture.ts:attributeRules` ships a single uniform `costByScore` map.
   - Target: New file `packages/rules-engine/src/foundation/data/puerta.point-buy.json` (final path resolved in `/gsd-discuss-phase 12.6`) with an entry per race in `dedupeByCanonicalId(compiledRaceCatalog.races)`. Each entry carries `{ budget, minimum, maximum, costByScore }`. Provenance doc sibling (`puerta.point-buy.md`) cites the Puerta server source each curve was vetted against.
   - Acceptance: Build-time test (`tests/phase-12.6/point-buy-snapshot-coverage.spec.ts`) asserts `Object.keys(snapshot).length === dedupeByCanonicalId(compiledRaceCatalog.races).length`; every entry passes a Zod schema (`budget > 0`, `minimum <= maximum`, all keys in `costByScore` are stringified integers within `[minimum, maximum]`). Suite fails deploy on missing race.

2. **A1 — Per-race cost resolution**: `nextIncrementCost` + `canIncrementAttribute` + `calculateAbilityBudgetSnapshot` resolve `costByScore` from the snapshot using the active `raceId`.
   - Current: UI always passes uniform `attributeRules.costByScore` into ability-budget helpers; the helpers themselves already accept the curve as a parameter (framework-agnostic shape preserved).
   - Target: New selector (`selectAbilityBudgetRulesForRace(raceId)`) in `apps/planner/src/features/character-foundation/selectors.ts` returns the per-race `AbilityBudgetRules` (keyed lookup on snapshot). AttributesBoard threads the selector output into the existing helpers. `packages/rules-engine/src/foundation/ability-budget.ts` stays framework-agnostic — no React imports, no snapshot knowledge.
   - Acceptance: Vitest proves identical `baseAttributes` with different `raceId` yield different `remainingPoints` (e.g. Elfo at `{STR:10,DEX:14,CON:10,INT:10,WIS:10,CHA:10}` vs Enano same base → distinct totals; exact numbers pending Puerta source lock).

3. **A1 — Fail-closed on missing curve**: Race without snapshot entry blocks atributos at the UI; no uniform fallback.
   - Current: No runtime gate for missing curve.
   - Target: When `selectAbilityBudgetRulesForRace(raceId)` resolves to `undefined`, `calculateAbilityBudgetSnapshot` returns `status: 'blocked'` with a new `rule:point-buy-missing` issue in `issues[]`. AttributesBoard renders `"Curva punto-compra no disponible para {raceLabel}"` as a block callout, `+` button `disabled` for every attribute, `-` button disabled, budget line hidden.
   - Acceptance: Vitest with empty snapshot + valid `raceId` → snapshot `status === 'blocked'` and `issues` contains `rule:point-buy-missing`. RTL renders the Spanish copy + asserts all 6 `+` buttons carry `disabled` attribute.

4. **A1 — Per-race regression fixture**: Canonical per-race `remainingPoints` deltas locked in a regression spec.
   - Current: Phase 12.3-01 `tests/phase-12.3/attributes-budget-gate.spec.ts` exercises the uniform curve only.
   - Target: New spec `tests/phase-12.6/ability-budget-per-race.spec.ts` iterates every race in `compiledRaceCatalog.races`, asserts canonical baseline remaining points at `{8,8,8,8,8,8}` and after a deterministic bump pattern (one attribute → 14) per race. Table-driven; fails if any race yields an unexpected delta or lacks a snapshot entry.
   - Acceptance: Suite runs under the existing vitest jsdom glob; passes green against the authored snapshot; flipping a single snapshot entry flips the spec red.

5. **P5 — Single-screen 20-level scan**: All 20 levels visible at once with inline class + feat-slot + skill-point + legality pills.
   - Current: Scanning a build requires clicking each L1..L20 rail button in `level-rail.tsx` and reading the single-level `level-sheet.tsx`. No single surface shows the full progression.
   - Target: `build-progression-board.tsx` (currently double-mounts `ClassPicker` and is the parent surface) renders a new compact per-level row element (exact component structure resolved in `/gsd-discuss-phase 12.6` — candidate: `<LevelProgressionRow />`). Each row shows: level number, class pill (empty state for unset), feat-slot pill `{chosen}/{total}`, skill-point pill `{spent}/{budget}`, legality glyph (✓ / ⚠ / ✗ / 🔒). At 1280×800 viewport all 20 rows fit inside the level-progression container without scroll.
   - Acceptance: RTL-jsdom spec `tests/phase-12.6/level-progression-scan.spec.tsx` mounts the full board with 20-level fixture at 1280×800, asserts `container.scrollHeight === container.clientHeight` on the level-progression list element, queries `[data-level-row]` and asserts `length === 20`, asserts each row contains queryable `[data-pill="class"]`, `[data-pill="feats"]`, `[data-pill="skills"]`, `[data-pill="legality"]`.

6. **P5 — Edit-in-place active level + legality diff**: Active row expands inline with full editor; every row carries legality data attribute.
   - Current: Active-level editing happens in a separate panel (`level-sheet.tsx`) side-by-side with the rail.
   - Target: Clicking a row expands it inline inside the list — `ClassPicker` + requirement rows + gains + `AbilityIncreaseControl` + `LevelEditorActionBar` (from 12.4-09) render inside the expanded row; non-active rows stay compact. Every row carries `data-legality="legal" | "incomplete" | "invalid" | "locked"`. Locked rows (G1 sequential gate already landed in hot-fix) also carry `aria-disabled="true"` and a visual dim.
   - Acceptance: RTL spec clicks `[data-level-row][data-level="5"]` in a fixture where L1..L4 are legal; asserts expanded row contains `class-picker__list` and `[data-testid="level-editor-action-bar"]`; non-active rows do NOT contain those nodes. Second fixture: L5 before class picked → `data-legality="incomplete"`; after class picked → `data-legality="legal"` flips in the same render cycle. Third fixture: L8 while L7 is empty → `data-legality="locked"` + `aria-disabled="true"`.

## Constraints

- Snapshot authoring must carry provenance: each race entry cites the Puerta server source (forum post, spreadsheet, script) in `puerta.point-buy.md` sibling doc; un-sourced entries fail code review.
- Fail-closed pattern must mirror VALI-04 — no silent fallback to the NWN1 uniform curve at runtime.
- Level range is 1..20 (P6 constraint already landed); no hardcoded `level < 16` or `level === 16` predicates may be introduced.
- `packages/rules-engine/` stays framework-agnostic — no React imports, no DOM reads in `ability-budget.ts` or the snapshot loader.
- Rules-engine shape preserved: `AbilityBudgetRules` interface at `ability-budget.ts:4-10` stays stable (no breaking change); per-race resolution happens at the selector boundary, not inside the pure helper.
- `buildDocumentSchema` schemaVersion unchanged — SHAR-05 share-URL invariant preserved.
- At 1280×800 viewport the scan surface must not scroll the level-progression list; at narrower viewports vertical scroll on the list itself is acceptable (P5 is scan-first, not responsive-first).
