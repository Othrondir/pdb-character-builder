## Implementation Decisions

### Habilidades scroll-snap (F1 + F2)

- **D-01 (auto):** Remove `scroll-snap-type: y proximity` from `.skill-board .selection-screen__content` at `apps/planner/src/styles/app.css:473` AND remove `scroll-snap-align: start` from `.skill-board .skill-sheet__row` at `apps/planner/src/styles/app.css:477`. Root-cause fix — the snap re-binding is what overrides programmatic `scrollTop=0` and produces the +/- jitter (F2). Leaving the snap with a different anchor would still reintroduce the jitter. Recommended over a snap-anchor retune because the Habilidades list does not need snap semantics (rows have no per-row content dominance; the user scans continuously).
- **D-02 (auto):** Re-wire `scrollerRef` in `apps/planner/src/features/skills/skill-sheet.tsx:151-161` to target the real overflow owner. Switch `<aside ref={scrollerRef}>` to an effect that calls `document.querySelector('.skill-board .selection-screen__content')` (or a stable DOM ref passed down from `BuildProgressionBoard`), and set `scrollTop = 0` on that element. Dependency stays `[activeSheet.level]`. Rationale: the aside `<.skill-sheet>` is not the scroller (UAT evidence: `clientHeight === scrollHeight === 1982`); keeping `scrollerRef` on the aside leaves 12.7-03 R3 broken even after the CSS fix.
- **D-03 (auto):** Regression guard ships in Playwright, not jsdom. Add `tests/phase-12.8/skill-scroll-snap.e2e.spec.ts` (new) rather than reusing the existing Vitest `skill-sheet.spec.tsx`. jsdom missed the 12.7-03 regression because it has no layout engine / no snap — repeating the Vitest pattern would re-open the same class of regression.

### Dotes multi-slot UX (F3 + F4)

- **D-04 (auto):** F3 resolution approach — **auto-scroll the first interactive row of `[data-slot-section="general"]` into viewport** when `handleSelectClassFeat` completes the class slot. Use `element.scrollIntoView({ block: 'nearest', behavior: 'smooth' })` on the first general-section row. Recommended over collapse-on-complete because (a) keeps the class section visible so the user can verify the chosen class feat and still deselect via the toggle semantic at `feat-sheet.tsx:268-275`; (b) composes cleanly with F4's per-chip deselect (no two-way coupling between summary-card state and section visibility). Recommended over a viewport-nudge banner because a banner adds copy surface that duplicates the existing "Ahora" pill + `feat-sheet__group--current` visual cue.
- **D-05 (auto):** F4 deselect control shape — **per-chip × button** rendered inside each `.feat-summary-card__item` in `apps/planner/src/features/feats/feat-summary-card.tsx`. Invokes `clearClassFeat(activeLevel)` or `clearGeneralFeat(activeLevel, slotIndex)` per chip slot. Preserves the existing `onModify` / "Modificar selección" button (which re-expands the full sheet) alongside. Recommended over whole-chip toggle because tapping a chip accidentally is common; × button disambiguates intent. Recommended over "do not collapse on same sub-step" because the collapse-on-complete decision shipped in Phase 12.4-07 (D-04) and reversing it re-opens that phase's design rationale.
- **D-06 (auto):** Extend `FeatSummaryChosenEntry` with a `slotKind: 'class-bonus' | 'general'` + `slotIndex: number` field so the chip knows which clear action to invoke. Update `<FeatBoard />` projection (`feat-board.tsx:~80-101`) to pass the richer entry shape + an `onDeselect` callback. No change to `<FeatSummaryCard>` public contract beyond the new callback prop — keeps the component boundary clean.

### Prestige gate tightening (F5)

- **D-07 (auto):** Root-cause is NOT "partial enrichment" as UAT-FINDINGS suggested — it is a **gate schema under-specification**. `ClassPrereqInput.decodedPrereqs` at `packages/rules-engine/src/progression/prestige-gate.ts:38-47` only declares `minBab|minSkillRanks|requiredFeats|minClassLevel`. Override data at `prestige-prereq-data.ts` ships `minArcaneSpellLevel`, `minSpellLevel`, `excludedClassIds`, `requiredAnyFeatGroups`, `requiredAnyRaceIds`, `requiredAnyClassLevels` — the gate never evaluates them. When a class has ONLY unprocessed fields (e.g., `pale-master` has `minArcaneSpellLevel: 3, excludedClassIds: [druid]`), `reachableAtLevelN` emits zero blockers → OPEN. `caballero-arcano` has `minArcaneSpellLevel: 5 + requiredFeats: MARTIAL_WEAPON_PROFICIENCIES` — the feats are processed but at L2 Elfo Guerrero the martial prof feats are auto-granted by Guerrero, so the only unsurfaced prereq is arcane-spell-level.
- **D-08 (auto):** Fix approach — **extend the gate schema and evaluator** to cover all override field shapes. Add cases for `minArcaneSpellLevel`, `minSpellLevel`, `excludedClassIds`, `requiredAnyFeatGroups`, `requiredAnyRaceIds`, `requiredAnyClassLevels` in `ClassPrereqInput['decodedPrereqs']` + matching branches in `reachableAtLevelN`. Per-field label templates extend `shellCopyEs.progression.prestigePrereqTemplates`. Recommended over trimming overrides because the richer shape encodes real NWN1 EE prereqs that must surface eventually; trimming loses data.
- **D-09 (auto):** Shadowdancer enrichment — extend the `'class:shadowdancer'` entry in `PRESTIGE_PREREQ_OVERRIDES` with `minSkillRanks` for `skill:moversesigilosamente` (8) + `skill:interpretar` (5 — Tumble analogue; verify against Puerta skill IDs before locking) + `requiredFeats` for `feat:dodge` + `feat:mobility`. Pattern follows the `260422-h9k` override extension (harper/campeondivino/weaponmaster). Verify final IDs against Puerta snapshot during research — the Tumble skill name is not self-evident in the Spanish skill catalog.
- **D-10 (auto):** Fail-closed branch at `prestige-gate.ts:138-144` stays as-is — it already handles the "no override data" case via `enriched: false`. No new heuristic needed because D-08 closes the "override data present but some fields unprocessed" gap at the schema level, not via a heuristic.
