**2. [Rule 3 - Blocking] Playwright fixture helper selectors drifted from plan text**
- **Found during:** Task 1 Step 6 RED verification — all 5 tests failed on `getByRole('button', { name: /^Elfo$/ })` timing out at 30s
- **Issue:** Plan's fixture helper assumed race/alignment options are `<button role="button">`. Reality: `<OptionList>` emits `<button role="option">`. Plan also skipped the explicit `Aceptar` click between origin steps (race → alignment → attributes). Class picker `aria-label` encodes `"{label} {status}"` (e.g. "Guerrero legal") so `getByRole('button', { name: /^Guerrero$/ })` never matches. Sub-step chips need `[data-substep="skills"]` not `getByRole('button', { name: /Habilidades/ })` — there are multiple Habilidades occurrences (sheet tab + block heading + sub-step) that collide.
- **Fix:** Rewrote the fixture helper to (a) `getByRole('option', ...)` for race + alignment selections, (b) add Aceptar clicks between each origin step, (c) use `[data-class-id="class:fighter"]` for class pick, (d) use `[data-substep="skills"]` for sub-step chip. Pattern documented as "selector stability via runtime data-* attributes, not localized ARIA names" — planted as a pattern for 12.8-03 Dotes fixture reuse.
- **Files modified:** tests/phase-12.8/skill-scroll-snap.e2e.spec.ts
- **Verification:** Playwright 5/5 pass after fix; fixture helper now deterministic across Spanish-copy iterations.
- **Committed in:** `9c33d36`
