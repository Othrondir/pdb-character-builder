## Warnings

### WR-01: feat-sheet auto-scroll uses unscoped `document.querySelector` for `[data-slot-section="general"]`

**File:** `apps/planner/src/features/feats/feat-sheet.tsx:288-299`
**Issue:** The F3 auto-scroll resolves `document.querySelector('[data-slot-section="general"]')` globally. The attribute is currently unique to `<FeatSheet>`, but it is also the exact selector the Playwright fixture + any future dotes surface (e.g., retraining UI, shared-build preview) would reuse. If a second dotes panel ever mounts while this one is active, `scrollIntoView` can scroll the wrong section into view (the first match in DOM order, not the one owned by this FeatSheet instance).
**Fix:** Scope the query to the component's own subtree. Hold a ref on the `<section data-slot-section="class-bonus">` or the FeatSheet root container and call `rootRef.current?.querySelector('[data-slot-section="general"]')`:
```tsx
const rootRef = useRef<HTMLElement | null>(null);
// ...inside rAF:
const generalSection = rootRef.current?.querySelector<HTMLElement>(
  '[data-slot-section="general"]',
);
```
Same pattern applied to skill-sheet (WR-02) keeps the "selector stability via local roots" discipline consistent.

### WR-02: skill-sheet scroll reset resolves first `.skill-board .selection-screen__content` in document order

**File:** `apps/planner/src/features/skills/skill-sheet.tsx:150-157`
**Issue:** D-02 intentionally swapped the React ref for a runtime `document.querySelector` because the scroller sits in a parent component. That works today, but the selector is global: any future case where two `.skill-board` regions coexist (e.g., compare-view, shared-build preview, split pane) scrolls the wrong one to 0. Also, if `<SkillSheet>` ever mounts outside a `.skill-board` parent (tests, Storybook), the effect becomes a silent no-op — the jsdom spec at `tests/phase-12.7/skill-sheet-scroll-reset.spec.tsx` already had to wrap the component in `<SelectionScreen className="skill-board">` to keep the effect observable.
**Fix:** Pass the scroller ref down from `BuildProgressionBoard` / `SelectionScreen` instead of querying the document, or at minimum scope the query to the nearest ancestor via `aside.closest('.skill-board')?.querySelector('.selection-screen__content')`. The latter keeps the change to one file:
```tsx
const asideRef = useRef<HTMLElement | null>(null);
useLayoutEffect(() => {
  const scroller = asideRef.current
    ?.closest('.skill-board')
    ?.querySelector<HTMLElement>('.selection-screen__content');
  if (scroller) scroller.scrollTop = 0;
}, [activeSheet.level]);
// <aside ref={asideRef} ...>
```

### IN-02: `FeatSheet` auto-scroll guard misses re-select-after-clear within same mount

**File:** `apps/planner/src/features/feats/feat-sheet.tsx:280-302`
**Issue:** `prevClassFeatIdRef` only fires the scroll on `null → non-null`. If the user selects feat A (scroll fires), clears A via toggle (ref → null, no scroll), then selects feat B **without leaving the class section**, ref transitions `null → "featB"` → scroll fires again. That is the desired behavior. However, when user selects feat A, then picks feat C (A → C, both non-null), the general section stays put — no second scroll, even though the user is "completing the class slot a second time". Per SPEC F3 this is fine (auto-scroll only on class-slot completion, not swap), but worth flagging so a future bug-report against "picking a different class feat doesn't nudge viewport" is traceable to design intent here.
**Fix:** None required. Optional — broaden the guard to `prev !== next && next !== null` if UX wants a nudge on every class-slot change. Document the current semantics above the ref declaration (already partially covered).

### IN-03: `anyFeatGroupSingle` copy template duplicates `feat` template wording

**File:** `apps/planner/src/lib/copy/es.ts:182`
**Issue:** `anyFeatGroupSingle: 'Requiere dote: {featName}'` is byte-identical to the existing single-feat template elsewhere in the block. Harmless today (both paths emit the same string), but if `feat` label is ever reworded (e.g., "Requiere la dote") the `anyFeatGroup` single-element fallback will drift. Consider aliasing: `anyFeatGroupSingle: shellCopyEs.progression.prestigePrereqTemplates.feat`, or a comment noting the intentional duplication for grep discoverability.
**Fix:** Add a comment (`// intentional alias of feat: — single-group collapses to the same string`) or merge the constants. Low-risk cosmetic.
