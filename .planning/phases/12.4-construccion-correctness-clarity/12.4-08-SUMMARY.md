# Phase 12.4-08: R7 parameterizedFeatFamily schema + inline <fieldset> expander Summary

## One-liner

Parameterized feat families (`Soltura con una habilidad`, `una escuela de magia`, `un arma`; `Soltura mayor con...`; `Especialización en arma`) fold into ONE canonical main-list row per family via an OPTIONAL/NULLABLE `parameterizedFeatFamily` schema field (OQ-4 backward-compat), client-side `groupIntoFamilyEntries`, and an inline `<fieldset class="feat-family-expander">` picker with `role="radiogroup"` + auto-focus + Esc-to-close keyboard.

## Problem

SPEC R7 / FEAT-01 / CONTEXT D-05: the Dotes main list renders ~1800 rows at L1 because every per-target variant (`Soltura con una habilidad (Trato con los animales)`, `(Sigilo)`, `(Disfrazarse)`, ..., × 32 habilidades) is a separate CompiledFeat row in the Puerta catalog. Same pattern for the 8 escuelas de magia and ~60 armas. Users drown in variant rows that are conceptually one choice. UAT finding F6: family must fold to ~30 canonical rows with a secondary picker for the specific target.

## Approach

### 1. Schema + extractor (pure source-of-truth layer)

`packages/data-extractor/src/contracts/feat-catalog.ts` gains `parameterizedFeatFamilySchema` exported standalone plus as an optional+nullable field on `compiledFeatSchema`:

```ts
parameterizedFeatFamilySchema = z.object({
  canonicalId: z.string().regex(/^feat:[a-z0-9-]+$/),
  groupKey: z.string(),
  paramLabel: z.string(),
}).nullable().optional();
```

Backward-compat guaranteed: rows with AND without the field parse cleanly (verified by spec A4/A5).

`packages/data-extractor/src/assemblers/feat-assembler.ts` gains a pure `detectFamily(displayLabel)` helper that regex-matches the 6 family label prefixes (`Soltura con una habilidad (...)` → `feat:skill-focus`, etc.). The assembler's main loop calls it after TLK resolution and emits the field on every CompiledFeat (null for non-family rows).

### 2. Artifact hand-patch

`scripts/patch-compiled-feats-family.mjs` applies the same regex list to `apps/planner/src/data/compiled-feats.ts` by parsing the JSON-ish source, walking each `{` feat-object block, reading the `"label"` line, and splicing a `"parameterizedFeatFamily": {...}` entry BEFORE the `"prerequisites":` key. Ran once to patch 230 variants across 6 families:

| Family                          | Variants |
|---------------------------------|---------:|
| feat:skill-focus                |       32 |
| feat:spell-focus                |        8 |
| feat:greater-spell-focus        |        8 |
| feat:weapon-focus               |       59 |
| feat:weapon-specialization      |       58 |
| feat:greater-weapon-focus       |       65 |
| **Total**                       | **230** |

(All counts from `grep -c "feat:skill-focus" compiled-feats.ts / 2`, etc., where /2 accounts for each entry appearing as both `canonicalId` and `groupKey`.)

Next `pnpm extract` run regenerates the same field for every variant because the now-shipped `detectFamily()` lives inside the assembler — the patch script is the bridge for the current environment (no Windows-local NWN install assumed) and a deterministic reference.

### 3. UI — selectors + components

`apps/planner/src/features/feats/selectors.ts`:

- **`FeatFamilyView`** — `{ canonicalId, groupKey, label, paramLabel, rowState, blockedReason, targets: FeatOptionView[], selectedTarget }`.
- **`FeatListEntry`** union — `{ kind: 'feat'; option } | { kind: 'family'; family }` consumed by FeatSheet.
- **`groupIntoFamilyEntries(options)`** — pure helper. Buckets options by `parameterizedFeatFamily?.groupKey`; emits a `FeatFamilyView` per non-empty bucket + `{kind: 'feat'}` for non-family singletons; sorts alphabetically by primary label. Family row state rolls up to `selectable` if any target is selectable, else most-informative blocked flavor (prereq > budget > already-taken).
- **`FeatBoardView`** gains `classBonusEntries` + `generalEntries` populated via `groupIntoFamilyEntries(eligibleClassFeats)` / `(eligibleGeneralFeats)`.

`apps/planner/src/features/feats/feat-family-expander.tsx` (NEW):

- `<fieldset class="feat-family-expander" data-family-id={groupKey} tabIndex={-1}>` auto-focused via `useRef + useEffect` so `Esc` keydown fires without prior internal click.
- `<legend class="feat-family-expander__legend">Elige {paramLabel}</legend>` (Spanish-first).
- `<div role="radiogroup">` with one `<label>` per target; each wraps `<input type="radio" name="feat-family-{groupKey}" disabled={rowState !== 'selectable'} onChange={() => onSelectTarget(target.featId)} />` + label span + optional pill for blocked-already-taken.

`apps/planner/src/features/feats/feat-sheet.tsx`:

- Replaced `FeatPickerList` calls with `FeatEntryList` which consumes `FeatListEntry[]`.
- New `FeatFamilyRow` button carries `data-family-id`, `aria-expanded`, `{N} objetivos` pill (plural-aware), and — when a target is chosen at the active level — an italic `feat-picker__reason` showing the chosen target label so the folded row still communicates selection state.
- Local `expandedFamilyId` state ensures single-expander discipline (clicking a second family closes the first).

### Auto-fixed Issues

**1. [Rule 3 — Blocking issue] Family detection uses label-prefix, not id-prefix, because Puerta catalog IDs are opaque slugs**

- **Found during:** Task 0 RED authoring (scouting compiled-feats.ts to write the RED assertions).
- **Issue:** The plan body suggested detecting families by id prefix (e.g. `id.startsWith('feat:skill-focus:')`). In practice the Puerta `compiled-feats.ts` has NO feats with `feat:skill-focus:` prefix — the canonical-id generator runs `slugify(TLK_LABEL)` producing opaque slugs like `feat:skillfocusanim`, `feat:weapfocclub`, `feat:grspllfcsabj`. An id-prefix regex would match nothing.
- **Fix:** Detection regex-matches the Spanish display label instead (`'Soltura con una habilidad ('`, `'Soltura con una escuela de magia ('`, `'Soltura mayor con...'`, `'Especialización en arma(s) ('`). 6 regexes live in `detectFamily()` inside both `feat-assembler.ts` AND `scripts/patch-compiled-feats-family.mjs` (must stay in sync). The plan body explicitly authorized this adjustment ("If the project's feat id convention differs from the prefixes above..., adjust the detection logic accordingly").
- **Files modified:** `packages/data-extractor/src/assemblers/feat-assembler.ts`, `scripts/patch-compiled-feats-family.mjs`.
- **Commit:** `344778c` (bundled into GREEN).
- **Why Rule 3:** Blocks implementation because the plan's literal detection strategy would yield zero matches. Rule-3 label-based detection achieves the SPEC R7 groupKey contract without SPEC softening.

**2. [Rule 3 — Blocking issue] `Especialización en armas` (plural) label variant**

- **Found during:** Task 1 GREEN verification — Suite A3 `weaponSpec.length > 5` failed after first patch run (only 1 match).
- **Issue:** Initial regex used `Especializaci[oó]n\s+en\s+arma\s*\(` (singular 'arma'), but the Puerta catalog emits BOTH `'Especialización en arma (clava)'` (1 row, legacy singular) and `'Especialización en armas (daga)'` (~58 rows, plural). Only the 1 singular matched.
- **Fix:** Regex updated to `Especializaci[oó]n\s+en\s+armas?\s*\(` (optional `s`) in both `feat-assembler.ts` and the patch script; re-ran patch script; 57 more variants tagged (total went from 173 to 230).
- **Files modified:** `packages/data-extractor/src/assemblers/feat-assembler.ts`, `scripts/patch-compiled-feats-family.mjs`, `apps/planner/src/data/compiled-feats.ts`.
- **Commit:** `344778c` (bundled into GREEN).
- **Why Rule 3:** RED assertion lock; no SPEC softening — R7 says weapon-specialization IS a family and must fold, so catalog coverage had to reach ≥ ~30 variants.

**3. [Rule 3 — Test fixture correction] L1 Humano+Guerrero pre-fill class-bonus slot so general section renders**
