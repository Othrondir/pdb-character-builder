# Phase 12.4: Construcción Correctness & Clarity — Specification

**Created:** 2026-04-19
**Ambiguity score:** 0.17 (gate: ≤ 0.20)
**Requirements:** 9 locked

## Goal

The Construcción flow (rail level picker + per-level Clase / Habilidades / Dotes sub-steps) renders a legible, correctness-backed editor for an NWN1 Enhanced Edition build on the Puerta de Baldur server: class picker groups separate base from prestige with rule-evidence gating, feat-slot + skill-point budgets render as authoritative UI truth from pure selectors, a dynamic "Continuar al nivel N+1" advance button surfaces completion state, the Habilidades route splits class vs transclase cost visually, the Dotes route exposes per-row selectability with a locked visibility policy (unavailable stays visible) + collapse-on-complete, parameterized feat families fold behind a secondary picker, extractor + UI strip DELETED sentinel rows, the Dotes scroll lives on the list column, and L1 sub-steps render neutral until explicitly affirmed by the user.

## Background

Current state grounded in the 2026-04-19 UAT sweep (see `.planning/UAT-FINDINGS-2026-04-19.md`):

- `apps/planner/src/features/level-progression/class-fixture.ts` already exposes `PlannerClassKind = 'base' | 'prestige'` and a `BASE_CLASS_ALLOWLIST` (Phase 12.2-03 decision) — the data model supports grouping but the UI does not render base and prestige in separate sections. No `reachableAtLevelOne` field yet; extractor does not surface BAB / skill-rank / feat prereqs for prestige gating. This is an active tech-debt note inside `class-fixture.ts` comments.
- `apps/planner/src/features/feats/*` renders a flat Dotes list. Only `Esquiva` surfaces a prereq badge. No slot counter, no selectability states, no collapse-on-complete, no family folding for `Soltura con una habilidad / escuela de magia`. Scroll overflow currently attaches to the right detail column instead of the list column.
- `apps/planner/src/features/skills/skill-board.tsx` renders skills alphabetically with muted inline `Clase` / `Transclase` tags — not grouped, cost-per-rank not hinted.
- No per-level feat-slot or skill-point header counters anywhere in the UI. `featSlots` / `skillPointBudget` symbols do not exist in `packages/rules-engine` or `apps/planner`.
- Advance affordance is absent from level sub-steps; user must click the next numbered rail tile to progress.
- `packages/data-extractor/src/parsers/two-da-parser.ts` has some DELETED detection but `assembleFeatCatalog` does not filter sentinel rows before emission.
- Opening L1 after Atributos auto-seeds class to Explorador and shows Clase / Habilidades / Dotes sub-step checkmarks before user interaction — `X1` regression from an ancestral default-seed decision.

Phase 12.3 closed milestone v1.0's 9 UAT blockers. Phase 12.4 closes the next UAT wave — 8 findings (F1..F8) + 1 regression (X1) — which surface correctness + clarity gaps that block prestige-class UX from being trustworthy.

## Requirements

1. **Class picker base/prestige grouping + gating**: Picker renders explicit sections with rule-evidence prereq gating.
   - Current: Flat alphabetic list mixing 11 base classes with every prestige class; all rows clickable at L1 regardless of prerequisites.
   - Target: Two headered sections (`Clases básicas` above, `Clases de prestigio` below) inside the same picker. Prestige rows whose prerequisites fail at the currently-edited level are disabled with inline unmet-prereq copy (e.g. `Requiere BAB ≥ 5`, `Requiere 8 rangos de Sigilo`). At L1, every prestige row is disabled.
   - Acceptance: RTL spec asserts (a) two `<section>` containers with correct `aria-labelledby`, (b) all 11 base class ids rendered inside `Clases básicas`, (c) every prestige id rendered inside `Clases de prestigio` in disabled state at L1, (d) prestige rows with met prereqs at a constructed L8 build are enabled and prereq copy hidden.

2. **Contextual advance button with dynamic label**: Level editor footer exposes a single advance affordance reflecting completion state.
   - Current: No forward-advance button inside any L1 sub-step; user progresses by clicking next numbered rail tile directly.
   - Target: `LevelEditorActionBar` footer renders an `NwnButton` whose label is (a) `Continuar al nivel N+1` when all level-N slots filled and enabled, OR (b) `Faltan {N} dotes que asignar en este nivel` (disabled) when feat slots unfilled, OR (c) `Faltan {N} puntos de habilidad por gastar` (disabled) when skill-point budget not spent. Button dispatches `setActiveLevel(N+1)` + `setExpandedLevel(N+1)` + `setActiveLevelSubStep('class')` on click.
   - Acceptance: RTL spec asserts the three label states render under the three corresponding fixture builds; click on enabled state dispatches all three store mutations in a single batch; button is hidden on L16 (terminal level).

3. **Authoritative per-level feat-slot + skill-point math surfaced in UI**: A pure `@rules-engine/progression/per-level-budget` selector computes feat slots + skill points per level, and the UI renders them as header counters.
   - Current: Math is scattered and not exposed in UI; no `featSlots` / `skillPointBudget` selectors exist.
   - Target: `computePerLevelBudget(build, level)` returns `{ featSlots: { general, classBonus, raceBonus }, skillPoints: { budget, spent, remaining } }` deterministically. Source of truth follows Phase 1 decision `manual override > Puerta snapshot > NWN1 EE canon`. Canon covers: Humano L1 bonus feat, class bonus feats (Guerrero L1/L2, Mago L1+L5+L10+L15, Monje L1/L2/L6, Pícaro L10/L13/L16, Guardián enano bonus), L1 skill-point quadruple, Pícaro/Explorador INT-based bonus, prestige-class transitions. Dotes panel header shows `Dotes: {chosen}/{slots}`; Habilidades header shows `Puntos de habilidad: {spent}/{budget}`.
   - Acceptance: Fixture regression spec asserts exact `featSlots` + `skillPointBudget` values for canonical builds L1..L16 (single-class Guerrero, single-class Mago, single-class Pícaro, single-class Explorador, Humano+Guerrero, Humano+Mago). Selector is pure (no `window`, no `fetch`, no store import) and p95 < 1ms on L16 full-build in a vitest bench run.

4. **Habilidades class vs transclase split with cost hint**: Skill board renders two headered sections with visible cost-per-rank hint.
   - Current: Flat alphabetic list; muted `Clase` / `Transclase` tags per row almost invisible.
   - Target: Two `<section>` containers (`Habilidades de clase · coste 1 pt/rango`, `Habilidades transclase · coste 2 pts/rango`). `Solo entrenada` stays as a per-row badge orthogonal to grouping. Ranks cap still `Tope: 4` (class) / `Tope: 2` (cross-class) enforced by store.
   - Acceptance: RTL spec asserts (a) two sections with correct copy, (b) each skill id lands in the correct section given a Guerrero L1 build, (c) section header's cost copy renders at ≥ 12px legibility, (d) `Solo entrenada` badges appear independently of section.

5. **Dotes selectability states + visibility policy + slot counter + collapse-on-complete**: Feat picker exposes four row states with visibility policy locked.
   - Current: Flat list with no slot counter, no selectability states; only Esquiva shows prereq.
   - Target: Per-row state resolves to `selectable | blocked-prereq | blocked-already-taken | blocked-budget`. `blocked-*` rows remain visible (never hidden — user must be able to plan forward and see unmet requirements) with inline copy (e.g. `Requiere dote: Esquiva`, `Ya seleccionada`, `Slots agotados en este nivel`). Panel header shows `Dotes del nivel N: {chosen}/{slots}`. When `chosen === slots`, list body collapses to a summary card listing chosen feats + `Modificar selección` expand link. Feats are also bucketed by category (`Generales`, `Metamágicas`, `Creación de objetos`, `Combate`, `De clase`, `De raza`) via existing `categoryKey` or added to extractor output.
   - Acceptance: RTL spec asserts all four row states under fixtures; `blocked-prereq` never applies `display: none`; slot counter renders correct `{chosen}/{slots}` for L1 Humano+Guerrero (2/2) and L1 Mago (1/1); collapse engages when counter hits max and `Modificar selección` link expands it.

6. **L1 sub-steps render neutral until user affirms**: Clase / Habilidades / Dotes sub-step ✓ is earned, not default.
   - Current: Opening L1 auto-seeds class to Explorador and shows ✓ on Clase / Habilidades / Dotes before any user interaction.
   - Target: L1 build state starts with `classByLevel[1] = null`. No auto-default. Sub-step ✓ predicate: Clase ✓ only when `classByLevel[level] !== null`; Habilidades ✓ only when `skillPoints.remaining === 0`; Dotes ✓ only when `featSlots.filled === featSlots.total`. Rail tile for L1 shows blank label (no `Explo…` truncation) until class is selected.
   - Acceptance: RTL spec asserts initial L1 mount shows Clase / Habilidades / Dotes as neutral (no ✓); rail tile shows blank; selecting Guerrero flips Clase to ✓ only; allocating full skill-point budget flips Habilidades to ✓; picking required feats flips Dotes to ✓.

7. **Parameterized feat families fold behind secondary picker**: Generic `parameterizedFeatFamily` contract collapses any feat with a target slot into one canonical row.
   - Current: `Soltura con una habilidad (Trato con los animales)`, `Soltura con una escuela de magia (Abjuración)`, and analogous Weapon Focus / Weapon Specialization variants render as one row per target, inflating the list.
   - Target: Extractor / catalog exposes `featFamily: { canonicalId: 'feat:skill-focus', label: 'Soltura con una habilidad', targets: FeatTarget[] }` for every parameterized feat (confirmed families: `feat:skill-focus` skill variants, `feat:spell-focus` school variants, `feat:weapon-focus` + `feat:weapon-specialization` + `feat:greater-weapon-focus` weapon variants). UI renders one row per family in the main list; clicking opens a secondary modal/expander scoped to valid targets with the same selectability treatment from R5. Stored feat id in `BuildDocument.featsByLevel[level][slot]` still carries the specific target (e.g. `feat:skill-focus:animal-empathy`) so rules-engine logic is unchanged.
   - Acceptance: Fixture test asserts `featFamily` contract exists and is non-empty for skill-focus + spell-focus + weapon-focus. RTL spec asserts one main-list row per family, secondary picker opens with valid targets enumerated, selecting a target persists the target-specific feat id to store. Zod schema for `BuildDocument` accepts target-qualified ids.

8. **Extractor + UI strip DELETED / sentinel rows**: Fail-closed filter on both ends.
   - Current: Extractor emits rows whose 2DA `Label` or `Description` equals `DELETED`, `***DELETED***`, `UNUSED`, `****`, etc. These pollute the Dotes catalog.
   - Target: `assembleFeatCatalog` rejects rows whose label OR description matches sentinel regex (exact patterns confirmed by scouting `feat.2da` + `nwsync` TLK during plan). UI feat selector additionally skips any canonical id whose display name hits the sentinel as belt-and-braces. Skill + class + race catalogs audited for the same pattern in the same plan.
   - Acceptance: Vitest spec asserts feats catalog post-extraction contains zero rows whose label matches sentinel regex; RTL spec asserts zero sentinel rows render in the Dotes list; skill + class + race fixture specs assert same.

9. **Dotes scroll container on list column**: Overflow-y auto moves from shared wrapper to `.feat-picker__list`.
   - Current: Scrollbar attaches to the right detail / description panel (or shared wrapper) — the list column does not own its own scroll.
   - Target: `.feat-picker__list` owns `overflow-y: auto`; right detail panel scrolls only when its own content exceeds its height. Same pattern verified on class + race pickers (F8-adjacent audit).
   - Acceptance: RTL spec + Playwright happy-path assert: viewport 1280×900 shows list scrollbar on left column, not right; list scroll position persists when description changes; class + race pickers exhibit the same scroll-on-list behavior.

## Boundaries

**In scope:**
- Construcción rail level picker + per-level editor (Clase / Habilidades / Dotes sub-steps).
- Class picker grouping + prereq gating + `reachableAtLevelOne` predicate.
- `@rules-engine/progression/per-level-budget` pure selector.
- Feat picker (Dotes): selectability states, slot counter, collapse-on-complete, family folding, DELETED filter, scroll relocation.
- Skills board: class vs transclase section split + cost hint copy.
- Advance affordance inside `LevelEditorActionBar`.
- L1 sub-step neutrality (remove auto-seed Explorador).
- Extractor fix: DELETED / sentinel filter in `assembleFeatCatalog` + audit of skill / class / race assemblers.
- Regression coverage: fixture specs L1..L16, prestige gate per-class, RTL component specs, one Playwright E2E happy path.

**Out of scope:**
- Equipamiento, inventario, armas, armaduras, objetos — outside Construcción; may be its own future phase.
- Build comparador, presets, saved templates UI — not part of 12.4; backlog.
- Share-URL / JSON import-export changes — Phase 8 owns the `BuildDocument` schema. 12.4 MUST NOT modify the schema version or the URL payload encoding beyond adding target-qualified feat ids (which Phase 8 schema already accepts via string id).
- New catalog entries (classes / feats / skills) — 12.4 only applies filters + UI to existing extractor output.
- Shell surfaces (Menú drawer, Resumen, Utilidades) — unchanged by 12.4.
- Animations, easing tuning, icon system — 12.4 is correctness + clarity, not polish.
- Milestone v1.1 feature expansion (levels 17-20, custom content) — defer to future milestone.

## Constraints

- `@rules-engine/progression/per-level-budget` selector MUST be a pure function (no `window`, no `fetch`, no store import). Memoizable via zustand selectors + structural equality.
- L16 full-build budget computation p95 < 1ms on the project's vitest bench runner (node 24, Windows dev hardware baseline). If bench exceeds gate, plan MUST introduce memoization before merge.
- Math source follows Phase 1 decision chain: `manual override > Puerta snapshot > NWN1 EE canon`. Forum material remains evidence-only and MUST NOT promote to runtime truth.
- Zero modifications to `BuildDocument` schema version or `fflate`-compressed URL payload encoding. Target-qualified feat ids fit the existing string id shape; no new top-level schema fields.
- Scoped CSS — no global `.nwn-button--secondary` modifications. All styling inside `.feat-picker__*`, `.skill-board__*`, `.class-picker__*`, `.level-editor__*` selector namespaces.
- Spanish-first UI copy. New strings land in `@planner/lib/copy/es.ts`. English strings NEVER hardcoded in component files.
- Extractor sentinel filter MUST regex-match on `Label` OR `Description` — case-insensitive — to cover `DELETED`, `***DELETED***`, `UNUSED`, `****`. Add the regex to a shared constant used by feat/skill/class/race assemblers.
