# Phase 12.4: Construcción Correctness & Clarity — Research

**Researched:** 2026-04-19
**Domain:** React/TypeScript SPA refactor — rules-engine selector extraction, zustand store reshaping, extractor hygiene filter, scoped-CSS component work for the NWN1 EE / Puerta de Baldur planner's Construcción flow.
**Confidence:** HIGH

## Summary

Phase 12.4 is a correctness + clarity phase layered over an already-mature codebase. The 9 locked requirements (R1..R9) and the 10 implementation decisions (D-01..D-10 in CONTEXT.md) leave zero open architectural questions — what remains is deep grounding in the existing source so the planner can write file-scoped plans that do not collide. This research confirms every prescribed touchpoint exists on disk, surfaces three high-impact reuses the planner MUST exploit (existing `determineFeatSlots`, existing `evaluateSkillLevel`, existing `skillPointsPerLevel` on every compiled class), and flags two extractor gaps that CONTEXT.md's `<deferred>` section already acknowledges.

The critical finding: **the per-level feat-slot + skill-point math (R3) is ~70% already in the rules-engine.** `packages/rules-engine/src/feats/feat-eligibility.ts` exposes `determineFeatSlots(characterLevel, classId, classLevelInClass, classFeatLists)` — it handles Guerrero (L1/L2/L4/L6…), Swashbuckler (L1/L2/L5/L9/L13), Caballero arcano (L1/L3/L5…), the auto-granted feat pool, and the general-feat cadence (L1/L3/L6/L9/L12/L15). Meanwhile `packages/rules-engine/src/skills/skill-allocation.ts::evaluateSkillLevel` already implements the L1 ×4 rule, Int modifier, class (cap L+3, cost 1) / cross-class (cap (L+3)/2, cost 2) split. Every `CompiledClass` row already carries a `skillPointsPerLevel: number` field (verified: Barbarian 4, Bard 6, Cleric 2, Rogue 8, Wizard 2). `computePerLevelBudget` (R3) is therefore a *composition + re-shape*, not a from-scratch implementation — its job is to fold these three signals into one shared selector plus add the two missing pieces the old path did not cover: **(1) Humano L1 bonus feat, (2) Humano +1 skill point per level**.

**Primary recommendation:** Ship `computePerLevelBudget` as a thin orchestrator that composes `determineFeatSlots` + an `evaluateSkillLevel`-derived `skillPoints` surface, hardcode the Humano race-bonus constants inline with a comment pointing to the race-assembler backlog (extractor does not yet surface `bonusFeatAtLevelOne` or `extraSkillPointsPerLevel` on `CompiledRace`), and treat every canon number that the existing engine does NOT already expose as `[ASSUMED]` pending a Puerta snapshot audit.

### Pattern 1: Pure selector in rules-engine, consumer memoized in UI (D-07)

**What:** Rules-engine helpers take plain inputs (build snapshot, catalogs) and return plain outputs. No React, no store, no browser APIs. UI consumers subscribe narrowly via zustand + `useShallow`.

**When to use:** R3 `computePerLevelBudget`, R6 predicates, R1 `reachableAtLevelN`.

**Example (existing pattern — mirrored verbatim):**

// Source: packages/rules-engine/src/skills/skill-allocation.ts (L111-115 verified)
function getAvailablePoints(level: SkillLevelInput) {
  const base = Math.max(1, level.skillPointsBase + level.intelligenceModifier);
  return level.level === 1 ? base * 4 : base;
}

// How R3 composes it:
// packages/rules-engine/src/progression/per-level-budget.ts (NEW)
export function computePerLevelBudget(
  build: BuildState,
  level: number,
  classCatalog: ClassCatalog,
  raceCatalog: RaceCatalog,
): PerLevelBudget {
  const classId = build.classByLevel[level];
  const classRow = classCatalog.classes.find(c => c.id === classId);
  const classLevelInClass = /* count of this class in build.classByLevel up to `level` */;

  // REUSE: determineFeatSlots already handles Fighter/Swashbuckler/Caballero arcano + general cadence
  const slots = determineFeatSlots(level, classId, classLevelInClass, featCatalog.classFeatLists);
  const general = slots.generalFeatSlot ? 1 : 0;
  const classBonus = slots.classBonusFeatSlot ? 1 : 0;

  // NEW: Humano L1 bonus feat (race catalog has no bonusFeatAtLevelOne field → hardcode here w/ canon tag)
  const raceBonus = (build.raceId === 'race:human' && level === 1) ? 1 : 0;

  // REUSE: L1 ×4 + INT modifier formula from skill-allocation (inline, not by import)
  const intMod = Math.floor((build.abilityScores.int + /*levelIncreases*/ 0 - 10) / 2);
  const base = Math.max(1, (classRow?.skillPointsPerLevel ?? 2) + intMod);
  const humanSkillBonus = build.raceId === 'race:human' ? 1 : 0;
  const budget = (level === 1 ? base * 4 : base) + humanSkillBonus;

  return {
    featSlots: { general, classBonus, raceBonus, total: general + classBonus + raceBonus, chosen: /*count*/, remaining: /*diff*/ },
    skillPoints: { budget, spent: /*sum ranks*/, remaining: /*diff*/ },
  };
}

## Common Pitfalls

### Pitfall 1: R3 consumer re-renders on unrelated store changes
**What goes wrong:** `computePerLevelBudget` is called inside a zustand subscriber that reads the whole store object. Any dispatch in any feature causes re-computation + re-render. Bench p95 creeps above 1ms L16.
**Why it happens:** No planner component currently uses `useShallow`; naive `useLevelProgressionStore()` returns the whole state object (verified — `feat-board.tsx` L14 does exactly this).
**How to avoid:** 12.4-03 introduces the `useShallow` pattern. Every consumer of `computePerLevelBudget` subscribes ONLY to the inputs: `build.classByLevel[level]`, `build.abilityScores.int`, `build.raceId` (added for Humano). Example:
```typescript
const budgetInputs = useLevelProgressionStore(useShallow(s => ({
  classId: s.levels.find(l => l.level === level)?.classId,
  abilityIncreasesInt: s.levels.filter(l => l.level <= level && l.abilityIncrease === 'int').length,
})));
const raceId = useCharacterFoundationStore(s => s.raceId);
const budget = useMemo(() => computePerLevelBudget(..., level, ...), [level, budgetInputs, raceId]);
```
**Warning signs:** React DevTools flame graph shows `<FeatBoard>` re-rendering on a Habilidades click; bench exceeds 1ms p95.

### Pitfall 2: R8 regex too narrow → misses DELETED variants
**What goes wrong:** Regex `/^DELETED$/i` misses `***DELETED***`, `DELETED_`, `DELETED_DO_NOT_USE`, `PADDING`, etc. Post-extraction catalog still has pollution.
**Why it happens:** BioWare used multiple sentinel conventions. The NWN wiki confirms at least `DELETED`, `PADDING`, `****` (cell-level, already null-converted). Cut Feats wiki documents `DELETED_` prefix. `[CITED: https://nwn.wiki/spaces/NWN1/pages/38175424/Cut+Feats]`
**How to avoid:** Match the CONTEXT.md regex suggestion verbatim and extend if scouting surfaces more:
```typescript
// packages/data-extractor/src/lib/sentinel-regex.ts (NEW, plan 12.4-01)
export const SENTINEL_REGEX = /^(\*{3,}\s*DELETED\s*\*{3,}|DELETED(?:_.*)?|UNUSED|PADDING|\*{3,})$/i;
```
Test against all 4 catalog assembler outputs (feat, skill, class, race). Plan 12.4-01 MUST scout `compiled-feats.ts` first (grep for `"label": "DELETED`) to capture actual BioWare string forms present in the Puerta snapshot — the grep I ran found 2 hits at sourceRow 385 and 487, both exact literal `"DELETED"`. If more exotic forms exist upstream in nwsync, scouting surfaces them before regex lands.
**Warning signs:** `extractor-deleted-sentinel.spec.ts` asserts `catalog.feats.filter(f => /* regex */.test(f.label)).length === 0` → if this passes but UI still shows "DELETED", the UI belt-and-braces is misaligned with extractor regex. Keep both in one constant.

## Code Examples

### Example 1: `computePerLevelBudget` skeleton (plan 12.4-03)

```typescript
// packages/rules-engine/src/progression/per-level-budget.ts (NEW)
// Source: composes determineFeatSlots (feat-eligibility.ts L60) + skill-allocation.ts L111-115 formula

import { determineFeatSlots } from '../feats/feat-eligibility';
import type { ClassCatalog } from '@data-extractor/contracts/class-catalog';
import type { RaceCatalog } from '@data-extractor/contracts/race-catalog';
import type { FeatCatalog } from '@data-extractor/contracts/feat-catalog';

export interface PerLevelBudget {
  featSlots: {
    general: number;
    classBonus: number;
    raceBonus: number;
    total: number;
    chosen: number;
    remaining: number;
  };
  skillPoints: { budget: number; spent: number; remaining: number };
}

export interface BuildSnapshot {
  raceId: string | null;
  classByLevel: Record<number, string | null>;
  abilityScores: { int: number };
  intAbilityIncreasesBeforeLevel: (level: number) => number;  // helper
  chosenFeatIdsAtLevel: (level: number) => string[];
  spentSkillPointsAtLevel: (level: number) => number;
}

export function computePerLevelBudget(
  build: BuildSnapshot,
  level: number,
  classCatalog: ClassCatalog,
  featCatalog: FeatCatalog,
  _raceCatalog: RaceCatalog,  // currently unused — see Assumptions A1/A2
): PerLevelBudget {
  const classId = build.classByLevel[level] ?? null;
  const classRow = classCatalog.classes.find(c => c.id === classId) ?? null;

  // Count this class's levels through `level`
  const classLevelInClass = Object.entries(build.classByLevel)
    .filter(([lvl, cid]) => Number(lvl) <= level && cid === classId)
    .length;

  // REUSE: feat-eligibility covers Fighter/Swashbuckler/Caballero arcano cadences + general L1/L3/L6/L9/L12/L15
  const slots = classId
    ? determineFeatSlots(level, classId, classLevelInClass, featCatalog.classFeatLists)
    : { classBonusFeatSlot: false, generalFeatSlot: false, autoGrantedFeatIds: [] };

  const general = slots.generalFeatSlot ? 1 : 0;
  const classBonus = slots.classBonusFeatSlot ? 1 : 0;

  // [ASSUMED A1] Humano L1 bonus feat — extractor does NOT surface this field on CompiledRace
  const raceBonus = build.raceId === 'race:human' && level === 1 ? 1 : 0;

  const total = general + classBonus + raceBonus;
  const chosen = build.chosenFeatIdsAtLevel(level).length;

  // REUSE pattern: skill-allocation.ts::getAvailablePoints L111-115
  const intMod = Math.floor((build.abilityScores.int + build.intAbilityIncreasesBeforeLevel(level) - 10) / 2);
  const base = Math.max(1, (classRow?.skillPointsPerLevel ?? 2) + intMod);
  // [ASSUMED A2] Humano +1 skill point per level — extractor does NOT surface this
  const humanSkillBonus = build.raceId === 'race:human' ? 1 : 0;
  const budget = (level === 1 ? base * 4 : base) + humanSkillBonus;

  return {
    featSlots: { general, classBonus, raceBonus, total, chosen, remaining: total - chosen },
    skillPoints: {
      budget,
      spent: build.spentSkillPointsAtLevel(level),
      remaining: budget - build.spentSkillPointsAtLevel(level),
    },
  };
}
```

/**
 * BioWare + NWN EE sentinel conventions observed in 2DA LABEL column:
 *   - "DELETED" literal (Puerta snapshot confirmed: compiled-feats.ts sourceRow 385, 487)
 *   - "DELETED_*" family (NWN Cut Feats wiki)

export const SENTINEL_REGEX = /^(\s*\*{3,}\s*(DELETED|UNUSED|PADDING)\s*\*{3,}\s*|\s*(DELETED|UNUSED|PADDING)(_.*)?\s*|\s*\*{3,}\s*)$/i;

export function isSentinelLabel(label: string | null | undefined): boolean {
  if (!label) return false;
  return SENTINEL_REGEX.test(label.trim());
}
