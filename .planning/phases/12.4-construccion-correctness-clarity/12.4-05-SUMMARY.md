# Phase 12.4 Plan 05: Habilidades class/transclase split + cost hint Summary

One-liner: Habilidades skill board now renders two `<section>` blocks — `Habilidades de clase · coste 1 pt/rango` above `Habilidades transclase · coste 2 pts/rango` — with cost hint locked at ≥ 12px via both scoped CSS and an inline belt-and-braces style, closing SPEC R4.

## Context

SPEC R4 + CONTEXT D-09 target: the Construcción Habilidades route currently renders skills in a flat alphabetic list with muted inline `Clase` / `Transclase` tags per row — F4 UAT flagged these as "almost invisible". The target layout splits rows into two headered sections so the class/cross-class distinction is immediately visible, and the cost-per-rank hint (`coste 1 pt/rango` / `coste 2 pts/rango`) lands at ≥ 12px per R4 acceptance criterion (c) in SPEC.md §Requirements #4.

Scope is narrow: copy additions + selector grouping refactor + two-section JSX render + scoped CSS. Store semantics (`getCap`, `getSpentPoints`, `getCostType`) and the rank cap enforcement are unchanged — only the presentation layer flips its grouping axis.

## Approach

Three-task TDD cycle with per-task atomic commits, following Phase 12.3 D-08 discipline:

## Deviations from Plan

**[Rule 3 - Blocker] Inline `style={{ fontSize: '12px' }}` added to `.skill-board__cost-hint` span alongside the scoped CSS rule.**
- Found during: Task 0 RED → Task 1 GREEN attempt.
- Issue: jsdom does not load external CSS stylesheets by default. The R4 acceptance test `parseFloat(window.getComputedStyle(costHint).fontSize) >= 12` returns `NaN` in jsdom because the computed font-size resolves to empty string. A pure CSS-only fix would make GREEN impossible to verify from the regression suite.
- Fix: Added inline `style={{ fontSize: '12px' }}` on the cost-hint span. This complements (does not replace) the scoped CSS rule — the CSS rule remains the authoritative source in real browsers; the inline style is a jsdom-compatible DOM-level floor that happens to encode the same R4 numeric contract. Keeps the regression spec meaningful in CI.
- Files modified: `apps/planner/src/features/skills/skill-sheet.tsx`.
- Verification: 10/10 test assertions pass.
- Commit: `596b34b`.

**[Rule 2 - Missing critical] `data-trained-only="true"` attribute added to existing trained-only span.**
- Found during: Task 0 RED authorship.
- Issue: Plan's Suite D test needed a queryable attribute to assert "trained-only badges appear orthogonally to section grouping". The existing `<span>{shellCopyEs.skills.trainedOnlyLabel}</span>` had no data-attribute, class, or aria-label — the only query surface was the Spanish copy string, which couples test code to UI text.
- Fix: Added `data-trained-only="true"` attribute to the span only when `row.trainedOnly === true`. Decoupled the regression spec from copy text; future UI treatments (e.g. a visual badge style) can hang off the same attribute.
- Files modified: `apps/planner/src/features/skills/skill-sheet.tsx`.
- Verification: `querySelectorAll('[data-trained-only="true"]').length > 0` passes for Humano+Guerrero L1.
- Commit: `596b34b`.

**Total deviations:** 2 auto-fixed (1 Rule 3 blocker — jsdom CSS gap; 1 Rule 2 missing attribute for testability). **Impact:** Neither deviation changes user-visible behavior; both reinforce the R4 contract. No architectural changes (no Rule 4 escalation needed).
