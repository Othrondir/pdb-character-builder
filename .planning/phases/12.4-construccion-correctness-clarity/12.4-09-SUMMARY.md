# Phase 12.4-09: R2 LevelEditorActionBar sticky footer + atomic dispatch + E2E Summary

## One-liner

Ships `<LevelEditorActionBar>` — a sticky footer inside `.level-sheet` rendering a single `NwnButton` whose label (`Continuar al nivel {N+1}` / `Faltan {N} dotes…` / `Faltan {N} puntos…`, plural-aware) and disabled state derive from the new pure selector `selectLevelCompletionState` + `computeAdvanceLabel`; enabled click dispatches `setActiveLevelSubStep('class')` + `setActiveLevel(N+1)` + `setExpandedLevel(N+1)` atomically; component returns `null` at L16; OQ-1 resolved to RTL fallback because `@playwright/test` is not installed, E2E preserved with ≥4 concrete interactions + ≥2 DOM assertions; closes F2 of UAT-FINDINGS-2026-04-19 and the final R2 requirement of Phase 12.4.

## Problem

UAT 2026-04-19 finding F2: the Construcción level editor had no explicit "advance to next level" affordance. The only way to move from L1 to L2 was clicking the numbered rail tile — a cryptic interaction that required knowing the rail existed + understanding which tile was "next". No visual signal told users whether the current level was fully allocated; the button that would have been the natural "Continuar" surface simply did not exist. SPEC R2 (D-06) locks the fix: a sticky footer with a single dynamic-label button rooted in completion state.

## Approach

Three pure/ergonomic layers:

### 1. Pure selectors (selectors.ts additions)

```typescript
export interface LevelCompletionState {
  level: ProgressionLevel;
  featDeficit: number;
  skillDeficit: number;
  isComplete: boolean;
}

export function selectLevelCompletionState(
  progressionState, foundationState, featState, skillState, level,
): LevelCompletionState;

export function computeAdvanceLabel(
  state: LevelCompletionState,
): { label: string; disabled: boolean };
```

`selectLevelCompletionState` reuses the existing `buildSnapshotFromStores` composer + `CLASS_CATALOG_INPUT` / `FEAT_CATALOG_INPUT` / `RACE_CATALOG_INPUT` boundary adapters from Plan 12.4-04 — rules-engine stays framework-agnostic per Plan 12.4-03's purity decision. Deficits are `max(0, total - chosen)` / `max(0, budget - spent)`; `isComplete` requires `featSlots.total > 0` so a class-less level doesn't trigger a spurious enabled state.

`computeAdvanceLabel` is copy-layer and priority-resolved: feat deficit wins over skill deficit, plural-aware singular templates render when deficit === 1. Returns `{label, disabled}` ready-to-render — JSX never interpolates copy.

### 2. Component (level-editor-action-bar.tsx, NEW ~90 lines)

- Subscribes to all four planner stores (progression/foundation/feat/skill) + shell store via narrow selectors for the three dispatch actions.
- Short-circuits to `null` when `activeLevel >= 16` (terminal).
- Otherwise computes `{label, disabled}` via the two new selectors and renders:

```tsx
<footer className="level-editor__action-bar" data-testid="level-editor-action-bar">
  <NwnButton variant="primary"
             disabled={disabled}
             onClick={handleClick}
             data-testid={`advance-to-level-${nextLevel}`}>
    {label}
  </NwnButton>
</footer>
```

- `handleClick` short-circuits on `disabled` then dispatches atomically in order:
  1. `setActiveLevelSubStep('class')` (must precede setExpandedLevel to defeat its `?? 'class'` fallback).
  2. `setActiveLevel(next)` (progression-store).
  3. `setExpandedLevel(next)` (shell-store).

- Mount point: last child of `.level-sheet` in `level-sheet.tsx` (pinned to the scrolling aside bottom via `position: sticky`).

### 3. Copy + CSS + mount

- `shellCopyEs.progression.advanceButton` gains 5 keys: `continueTemplate`, `deficitFeatsSingular`, `deficitFeatsPluralTemplate`, `deficitSkillsSingular`, `deficitSkillsPluralTemplate`.
- `app.css` gains a single scoped block:

```css
.level-editor__action-bar {
  background: var(--color-panel);
  border-top: 1px solid var(--color-panel-edge);
  bottom: 0;
  display: flex;
  justify-content: flex-end;
  padding: var(--space-md);
  position: sticky;
  z-index: 1;
}
```

Zero new `font-weight: 700`, zero new hex colors, zero new spacing tokens — all via existing `var(--color-*)` + `var(--space-*)`.

**2. [Rule 3 — Blocking issue] jsdom cannot cascade `position: sticky` from injected <style>**

- **Fix:** Replaced `getComputedStyle` assertion with source-text regex against `app.css`:
  ```typescript
  expect(appCss).toMatch(/\.level-editor__action-bar\s*\{[^}]*position:\s*sticky/);
  expect(appCss).toMatch(/\.level-editor__action-bar\s*\{[^}]*border-top:\s*1px\s+solid\s+var\(--color-panel-edge\)/);
  ```
  Mirrors 12.4-07 Suite D's "app.css text regex as source-of-truth" pattern. Locks the CSS rule existence; real sticky behavior is covered by human UAT walkthroughs.
- **Files modified:** `tests/phase-12.4/level-editor-action-bar.spec.tsx` (bundled into GREEN commit `dff972c`).
- **Why Rule 3:** jsdom limitation blocked the straightforward assertion; source-text regex achieves the same regression-guard intent without requiring a browser.
