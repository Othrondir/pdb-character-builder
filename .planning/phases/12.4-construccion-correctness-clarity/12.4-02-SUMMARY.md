## Problem

Before this plan, the Dotes picker delegated scroll to the two-column wrapper `.selection-screen__content` (app.css L1158 — `overflow-y: auto` on the grid). UAT sweep 2026-04-19 filed **F8**: "Dotes scroll attaches to description column instead of the list column." Concretely:

- Scrolling the feat list also scrolled the description panel (they share the wrapper's scroll).
- When the description panel was long, the outer grid gained the scrollbar instead of the list column — visually noisy and breaks the expected "left column scrolls, right column stays pinned" idiom already used elsewhere (`.option-list` at L1167-1173 owns its own scroll).

SPEC R9 locks this as: `.feat-picker__list` owns `overflow-y: auto`; description panel scrolls only when its own content exceeds its height; audit `.class-picker__list` + `.race-picker__list` for the same pattern.

## Approach

1. **TDD RED first.** Wrote `tests/phase-12.4/feat-picker-scroll.spec.tsx` with three complementary assertion styles:
   - CSS-text grep (theme-contract pattern from `tests/phase-02/theme-contract.spec.ts`) — locks the rule shape in app.css.
   - DOM presence (`document.querySelector('.feat-picker__list')`) — locks component wiring.
   - jsdom `getComputedStyle` — after injecting app.css into `<style>`, proves the cascade yields `overflow-y: auto` on the list node and NOT on wrapper ancestors.

   This hybrid is necessary because jsdom does not apply CSS from imported `.css` files during module graph resolution — we must either read app.css text OR inject it into `<head>`. Both are used so the spec survives even if either verification style regresses in isolation.

2. **GREEN — minimal CSS surface.** Added `.feat-picker__list { overflow-y: auto }` and, critically, a scoped `.feat-board .selection-screen__content { overflow-y: visible }` override. The wrapper override is essential because without it the grid's `overflow-y: auto` still wins for the ancestor scroll (even when the list has its own `auto`, the grid owns scroll once content overflows the grid). Scoping via the `.feat-board` descendant selector keeps class/race/skills screens unaffected.

3. **GREEN — wire the class onto the list.** `<OptionList>` already accepts a `className` prop; added `className="feat-picker__list"` on both invocations in `feat-sheet.tsx`. The rendered DOM now reads `<div class="option-list feat-picker__list">`, so `.option-list` keeps providing its flex column + padding + gap while `.feat-picker__list` adds the scroll-on-list contract.

4. **Audit via grep-only.** For class-picker and race-picker, the plan explicitly forbids emitting CSS here (cross-wave coupling). Documented findings in this SUMMARY.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Visual regression] Minimal `.feat-picker__list` declaration surface**

- **Found during:** Task 1 CSS authoring (before running GREEN spec).
- **Issue:** The plan's CSS template (12.4-02-PLAN.md L177-184) includes `list-style: none; margin: 0; padding: 0; overflow-y: auto`. Because the feat list `<div>` also wears `option-list` (rendered by `<OptionList>`), and `.option-list` at L1167-1173 declares `padding: var(--space-sm)`, the `.feat-picker__list` rule appearing LATER in the stylesheet would win the equal-specificity cascade on `padding`. Result: removing the existing list-column padding — a silent visual regression.
- **Fix:** Reduced `.feat-picker__list` to declare only the R9-contract property (`overflow-y: auto`). Preserved `.option-list`'s padding/flex/gap composition. Documented the rationale in the inline comment block so future readers don't "simplify" by adding the dropped properties back.
- **Files modified:** `apps/planner/src/styles/app.css` (comment block expanded; declaration shrunk).
- **Commit:** `e27306a`.

**2. [Rule 3 — Blocking issue] Scoped wrapper-scroll removal**

- **Found during:** Task 1 CSS planning (when identifying "wrappers that produce the regression").
- **Issue:** The plan (L188) asks "Remove `overflow-y: auto` (and `overflow-y: scroll`) from any ancestor wrapper rules that currently catch feat-board scroll. Likely candidates: `.feat-board`, `.feat-board__body`, `.selection-screen__content`, `.feat-sheet`." Grepping app.css showed NEITHER `.feat-board` NOR `.feat-sheet` owns `overflow-y` — the regression-producing rule is actually `.selection-screen__content` (L1158, global). Removing it globally would break class/race/skills screens that still rely on it for inner-grid scroll.
- **Fix:** Kept the global `.selection-screen__content { overflow-y: auto }` intact and added a scoped descendant override: `.feat-board .selection-screen__content { overflow-y: visible }`. This touches ONLY the feat-board context. Other SelectionScreen users (class picker, race picker, skills board) continue to use the wrapper scroll until their own Wave 2 / Wave 3 plans relocate it to their respective `__list` classes.
- **Files modified:** `apps/planner/src/styles/app.css`.
- **Commit:** `e27306a`.

### Non-deviation: `.class-picker__list` deliberately absent

Per plan L190-196 and checker feedback, `.class-picker__list` CSS is NOT emitted from this plan. Plan 12.4-06 owns the full definition. Test A2 (`app.css does NOT contain .class-picker__list {`) passes — anti-coupling guarded.
