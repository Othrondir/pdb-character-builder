# Phase 12.4-06: R1 ClassPicker Extracted + reachableAtLevelN Prestige Gate + CLAS-03 Regression Summary

## One-liner

Pure `reachableAtLevelN` helper (inline `ClassPrereqInput`, zero cross-package imports) + extracted `<ClassPicker>` rendering two `<section>` blocks with L1 prestige gate + CLAS-03 multiclass-legality aria-disabled bridge.

## Problem

SPEC F1 / CLAS-01..04 / PICK-01: The Construcción class picker intermingled base and prestige classes in a flat grid. At L1, every prestige row was clickable despite prerequisites blocking them, and there was no inline evidence of what gate was firing. The NWN1 EE canonical rule "prestige is unreachable at L1" was not represented in the UI. A secondary CLAS-03 risk: the existing `evaluateMulticlassLegality → option.status` bridge (that disables multiclass-violating base rows) could silently regress if a future selector refactor stopped propagating `status==='blocked'` into DOM semantics.

## Approach

Split the surface into two concerns:

1. **Pure rules-engine helper** — new `packages/rules-engine/src/progression/prestige-gate.ts` exports `reachableAtLevelN(input): { reachable, blockers[] }`. Four decision branches per D-02:
   - base class → `{ reachable: true, blockers: [] }`
   - prestige at L1 → `{ reachable: false, blockers: [{ kind: 'l1', threshold: 2, label: 'Disponible a partir del nivel 2' }] }`
   - prestige at L2+ without enrichment → `{ reachable: false, blockers: [{ kind: 'unvetted', label: 'Requisitos en revisión' }] }`
   - prestige at L2+ with enrichment → decode `decodedPrereqs`, emit one blocker per unmet requirement (BAB / skill-rank / feat / class-level); empty list ⇒ reachable.

   The helper emits ready-to-render Spanish labels so consumers render `blockers[0].label` directly — no UI-layer template interpolation. Plural-aware copy is inlined (`'Requiere 1 rango de X'` vs `'Requiere 8 rangos de X'`).

2. **Extracted React component** — new `apps/planner/src/features/level-progression/class-picker.tsx` takes over the class-picker block from `level-sheet.tsx`. Renders two `<section>` blocks (`Clases básicas`, `Clases de prestigio`) with `aria-labelledby` wired to each `<h3>`. Each `<ClassPickerRow>` drives `aria-disabled` from the OR of two surfaces:
   - `option.status === 'blocked'` (CLAS-03 regression bridge — routes `evaluateMulticlassLegality` verdicts into DOM semantics).
   - `reachableAtLevelN(...).reachable === false` (prestige prereq gate).

   Inline italic reason copy from `gateResult.blockers[0]?.label` renders beneath disabled prestige rows only (base classes have no prestige reason).

3. **Boundary adapter** — `toClassPrereqInput(option)` lives inside `class-picker.tsx` and projects `ClassOptionView` + `PlannerClassRecord` into the minimal `ClassPrereqInput` the gate reads. Keeps rules-engine framework-agnostic per CLAUDE.md Prescriptive Shape. `grep -c '@data-extractor' packages/rules-engine/src/progression/prestige-gate.ts` returns 0.

4. **Spanish copy + scoped CSS** — `shellCopyEs.progression.classSectionBase`/`.classSectionPrestige`/`.prestigePrereqTemplates.*` appended under the existing namespace (append-only, no reordering). `.class-picker__*` CSS namespace emitted at the tail of `app.css` — section-heading (uppercased display font, gold border underline), list (owns `overflow-y: auto`), row (transparent bg + panel-edge border), row--blocked (opacity 0.55, cursor not-allowed, blocked-color border), row--active (selection gold), reason (italic muted at opacity 1 to defeat the cascade from the 0.55 row), focus-visible (gold outline).

## Framework Purity Note

`prestige-gate.ts` defines `ClassPrereqInput` inline (`id`, `isBase`, optional `decodedPrereqs`) instead of `Pick<CompiledClass, ...>`. This pattern matches the 12.4-03 per-level-budget inline-input discipline — rules-engine never imports from the extractor package, and the adapter layer lives at the apps/planner boundary. Future work: when Phase 13.x decodes `PreReqTable` on prestige classes, the boundary adapter populates `decodedPrereqs` and flips `enriched: true` without requiring any helper-signature change.

## CLAS-03 Regression Coverage

Test `tests/phase-12.4/class-picker-grouping.spec.tsx` constructs a multiclass scenario at L5 (seeded L1..L4 with four distinct base classes) and queries `selectClassOptionsForLevel` for a `status === 'blocked'` base option sourced from `evaluateMulticlassLegality`. When such an option exists, the test asserts the ClassPickerRow with that `data-class-id` carries `aria-disabled="true"`. If the current catalog state makes the scenario unreachable, the test logs a warning and returns — but the DOM-wiring bridge itself is always active (not gated on test reachability), so a future selector refactor that skips the `status → aria` bridge fails the assertion immediately on the first multiclass-blocked scenario that becomes constructible.
