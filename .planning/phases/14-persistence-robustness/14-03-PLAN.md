<objective>
Close ROADMAP SC#3 (Phase 14): `hydrateBuildDocument` currently does NOT read `doc.build.name`. The schema (`build-document-schema.ts:38`) supports `name: z.string().max(80).optional()`, and `projectBuildDocument(name?)` writes it on save (line 90: `...(name ? { name } : {})`). But on load, hydrate ignores the field and the foundation store has no slot for it — so a load → re-save flow silently loses the name.

Fix: add `buildName: string | null` to the foundation store (with `setBuildName` setter and reset wiring); call `setBuildName` from `hydrateBuildDocument`; let `projectBuildDocument` fall back to `foundation.buildName` when no `name` argument is supplied.

Purpose: Phase 8 hardening — surfaced by 2026-04-24 v1.0 re-audit (`v1.0-MILESTONE-AUDIT.md` Phase 08 line 70). Closes "hydrate drops build.name silently". Defensive — no schema change, no requirement reopens.

Output:
- Foundation store gains `buildName` slice + setter + reset behavior.
- `hydrateBuildDocument` calls `setBuildName`.
- `projectBuildDocument` reads from store when arg absent.
- New Vitest spec covering the round-trip.
- Existing `tests/phase-08/hydrate-build-document.spec.ts` round-trip case updated to assert `name` parity instead of stripping it.
</objective>
