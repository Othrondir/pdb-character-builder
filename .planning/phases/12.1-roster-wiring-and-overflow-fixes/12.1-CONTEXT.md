# Phase 12.1: Roster Wiring & Overflow Fixes — Context

**Gathered:** 2026-04-18
**Status:** Ready for planning
**Source:** `/gsd-discuss-phase 12.1` (mode: `--auto` defaults after user elected C)

<domain>
## Phase Boundary

Replace the hand-authored Phase-3 and Phase-4 fixtures that ship 3 races + 7 classes with the PDB-extractor-emitted compiled catalogs (46 races + 41 classes) so the planner's L1 class picker and race picker render the full authoritative Puerta de Baldur roster. Also restore scrollability on the one confirmed grid container that clips content (`.selection-screen__content`).

**In scope:**
1. Class roster wiring: planner L1 class picker sources from `apps/planner/src/data/compiled-classes.ts` (41 entries), not `phase04ClassFixture` (7 entries).
2. Race roster wiring: race picker sources from `apps/planner/src/data/compiled-races.ts` (46 entries with parent/subrace hierarchy), not `foundation-fixture.ts` (3 entries).
3. CSS overflow fix: `.selection-screen__content` (`app.css:1149`) gets `overflow-y: auto` so large option lists scroll instead of clipping.
4. Regression unit tests on the selectors that project compiled catalogs through so the roster count doesn't silently regress.

**Not in scope:**
- Class-kind business logic changes (base vs prestige visibility rules stay on the rules engine's existing `collectVisibleClassOptions` — Phase 12.1 preserves that filter, does not modify it).
- Race-picker UX pattern change (the existing parent → subrace dropdown pair stays; no flatten-to-list refactor).
- Any other `overflow: hidden` container (all inspected-in-triage siblings are legit grid wrappers with children correctly using `overflow-y: auto`).
- Data corrections to the compiled catalogs themselves (if a race or class is missing from the PDB extractor emission, that is a Phase 05.1 extractor bug, not Phase 12.1).
- Playwright / E2E regression — unit tests at the selector level are the chosen gate.

</domain>

<decisions>
## Implementation Decisions

### D-01 — Fixture replacement strategy
- **Decision**: Delete the hand-authored race list in `apps/planner/src/features/character-foundation/foundation-fixture.ts` and the hand-authored class list in `apps/planner/src/features/level-progression/class-fixture.ts`. The planner consumes the compiled catalogs directly; remaining fixture content that is NOT roster data (e.g. alignment list, class-entry rules, test helpers) is kept, factored into thinner modules if the files become awkward.
- **Rationale**: The compiled catalogs are stamped `// Generated by data-extractor on 2026-04-17T…. Do not edit manually.` and represent the PDB TLK roster per the user's 2026-04-18 directive (*"las clases y razas sólo deben ser las de PDB el TLK que analizaste y demás"*). A thin adapter wrapper (option b) or env-overrides (option c) would add indirection with no current use case; if a test harness later needs a restricted subset it can compose from the compiled catalog at that point.
- **Expected touchpoints**: `apps/planner/src/features/character-foundation/foundation-fixture.ts`, `apps/planner/src/features/level-progression/class-fixture.ts`, `apps/planner/src/features/character-foundation/selectors.ts`, `apps/planner/src/features/level-progression/selectors.ts`, any tests that import these fixtures (update to import from compiled catalogs or to compose a minimal per-test fixture inline).

### D-02 — Class-kind filter for L1 picker
- **Decision**: Preserve the existing filter path. The L1 picker's visible classes are already gated by `collectVisibleClassOptions` in `packages/rules-engine/src/progression/class-entry-rules.ts`; that function reads class `kind` / `prerequisites` metadata and correctly hides prestige classes with unmet prereqs. When `selectClassOptionsForLevel` switches its input from `phase04ClassFixture.classes` to `compiledClassCatalog.classes`, the same filter applies — prestige classes stay hidden at L1, base classes appear.
- **Rationale**: The bug is upstream (wrong roster) not downstream (wrong filter). `collectVisibleClassOptions` has shipped, is tested, and already handles the base-vs-prestige distinction via the rules engine. Re-adding a `kind: 'base'` grep at the selector layer would duplicate logic and risk drift.
- **Expected touchpoints**: `apps/planner/src/features/level-progression/selectors.ts` (single line — `phase04ClassFixture.classes` → `compiledClassCatalog.classes`). No rules-engine change.

### D-03 — Race hierarchy rendering
- **Decision**: Keep the existing parent → subrace dropdown pair (Humano + subraces-empty, Elfo → [Elfo lunar / Elfo salvaje / …], Enano → [Enano escudero / …]). When sourcing from `compiledRaceCatalog`, group the 46 entries by parent ID and render the parent list in the primary dropdown; the secondary dropdown renders the subraces of the currently selected parent. Flat rendering is rejected.
- **Rationale**: NWN2DB-familiar UX (see PROJECT.md — planner mirrors NWN2DB flow). The current two-dropdown shape is what users expect; flattening 46 rows into one list hurts scannability and contradicts FLOW-02. Phase 05.2 UX overhaul already committed to NWN1-compact presentation via dropdowns; extending it to the full roster is a mechanical swap, not a redesign.
- **Expected touchpoints**: `apps/planner/src/features/character-foundation/selectors.ts` (grouping derivation), race picker component (render the grouped tree — whatever the current component name is, it already consumes the fixture's structure).

### D-04 — CSS overflow scope
- **Decision**: Patch only `.selection-screen__content` (`apps/planner/src/styles/app.css:1149`) by changing `overflow: hidden` to `overflow-y: auto; overflow-x: hidden;` so the selection panel scrolls when the roster exceeds the viewport. Leave all other inspected containers untouched — they are legit grid wrappers whose children already carry the correct `overflow-y: auto`.
- **Rationale**: Triage identified 5 other `overflow: hidden` selectors (`planner-layout__stepper`, `planner-layout__center`, `.character-sheet`, `.character-sheet__stat-grid`, `.character-sheet__derived dt`) — all are either intentional layout containers (grid/flex parents that delegate scrolling to children) or ellipsis-truncation scopes. Touching them risks double-scroll, layout thrash, or breaking the ellipsis. The confirmed bug is a single selector.
- **Expected touchpoints**: `apps/planner/src/styles/app.css:1149` only. No JSX or component change.

### D-05 — Regression test posture
- **Decision**: Unit test at the selector projection level. Add `tests/phase-12.1/class-roster-wiring.spec.ts` and `tests/phase-12.1/race-roster-wiring.spec.ts` asserting that `selectClassOptionsForLevel({ level: 1 })` returns ≥ N_BASE classes (with N_BASE = count of `kind: 'base'` classes in `compiledClassCatalog`) and `selectFoundationOptions()` (or equivalent) returns ≥ 7 parent races with every subrace attached. Playwright E2E rejected — heavier, flakier, harder to debug; the wiring bug is at the selector boundary and that is where the assertion belongs. CSS overflow fix is visual — no automated test; verified manually during Phase 11-style browser UAT at closure.
- **Rationale**: The observed failure is a data-projection bug (fixture shadowed compiled catalog). The projection is a pure function — a unit test locks it against regression with minimum test-infrastructure weight. Count-based assertions stay stable even when the extractor adds/removes entries (assertion uses `compiledXxxCatalog.length`, not a hardcoded number).
- **Expected touchpoints**: `tests/phase-12.1/class-roster-wiring.spec.ts`, `tests/phase-12.1/race-roster-wiring.spec.ts`.

### D-06 — Plan split
- **Decision**: 3 plans matching the ROADMAP entry:
  - **12.1-01**: Wire PDB class roster into L1 picker (swap fixture → compiled catalog; add selector regression spec).
  - **12.1-02**: Wire PDB race roster + subrace hierarchy (delete fixture race list; source compiled catalog grouped by parent; add selector regression spec).
  - **12.1-03**: Restore `overflow-y: auto` on `.selection-screen__content`.
- **Rationale**: Atomic per-file-area commits; each plan is independently verifiable and reversible. Wave: 12.1-01 + 12.1-02 + 12.1-03 are file-disjoint (selectors.ts files differ by feature module; app.css is separate), so all three can run Wave 1 in parallel. If any plan's scope grows unexpectedly and breaks a sibling, the atomic split keeps blast radius contained.

</decisions>

<specifics>
## Specific Ideas

- **Do not modify the compiled catalogs.** They are PDB-extractor emissions and carry a do-not-edit-manually header. If a race or class looks wrong in the UI, the fix is upstream in `packages/data-extractor`, not in the planner.
- **Fixture files become thin or disappear.** `class-fixture.ts` and `foundation-fixture.ts` currently mix roster data with rules-engine plumbing + test helpers. Planner should plan for a minimal surgical cut (delete the roster constants, keep anything else the module exports) rather than a full module rewrite. Any surviving exports get a comment noting the roster content was moved to the compiled catalog.
- **Race grouping helper.** The planner will probably introduce a small pure function like `groupRacesByParent(races: RaceRecord[]): Map<parentId, { parent: RaceRecord; subraces: RaceRecord[] }>` and pipe the compiled catalog through it. That helper belongs in `packages/rules-engine/src/foundation/` so it is testable without touching React.
- **Test-tier discipline.** Regression specs go under `tests/phase-12.1/`, not inside the apps/planner unit-test tree. This mirrors the Phase 12 pattern (`tests/phase-12/class-prereq-label.spec.ts`) and keeps Nyquist coverage traceable to the phase that introduced it.
- **Commit style.** Mirror Phase 12's atomic-per-task pattern: one commit per logical change (`fix(12.1-01): swap class fixture for compiled catalog`, `test(12.1-01): add class-roster regression spec`, etc), plus a `docs(12.1-XX): write plan summary` commit per plan.
- **Manual UAT after execution.** After Phase 12.1 ships, re-run Phase 11 UAT Scenarios 1 and 4 against a live dev server to confirm the L1 class picker shows the full base roster, the race picker shows all parent races with their subraces, and the SelectionScreen scrolls when the roster overflows. This closes the observations that triggered 12.1.
</specifics>
