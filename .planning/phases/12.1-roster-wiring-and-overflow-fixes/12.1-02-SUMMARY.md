## Decisions Made

- **dedupeByCanonicalId first-wins at the projection boundary** — the compiled catalog currently emits duplicate `race:drow` (rows 196 + 676). Rendering the full roster without a dedupe guard produces React duplicate-key warnings in every spec that renders the race picker (caught first in `tests/phase-02/navigation-copy.spec.ts`). Dedupe at the consumer keeps the UI correct regardless of extractor timing.
- **Test subrace inlined (not a fixture extension) for the phase-03 parent-mismatch contract** — adding a permanent test subrace to the fixture would leak test data into production state; inlining it in the single spec that needs it preserves the rules-engine coverage without coupling the fixture to test-only rows.
- **Drop subrace assertions in phase-03/summary-status + phase-08/project-build-document** instead of teaching the store to accept unknown IDs — the store's `subraceMatchesRace` gate is a real correctness invariant (prevents orphaned state after a race swap); relaxing it would be a regression. When the extractor emits subraces, these assertions can be re-enabled with real IDs.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Dedupe against extractor-emitted duplicate race IDs**
- **Found during:** Task 2 (GREEN) full-suite vitest run
- **Issue:** `compiledRaceCatalog.races` contains `race:drow` at both row 196 and row 676 in the 2026-04-17 extractor emission. Projecting the raw array produced a React duplicate-key warning: `Encountered two children with the same key, 'race:drow'`. The hand-authored 3-race fixture never surfaced this because only Humano/Elfo/Enano were rendered.
- **Fix:** Added generic `dedupeByCanonicalId<T extends { id: CanonicalId }>` helper applied to both `races` and `subraces` projections. First-wins: row 196's `race:drow` entry wins; row 676 is silently dropped.
- **Files modified:** `apps/planner/src/features/character-foundation/foundation-fixture.ts`
- **Verification:** `grep "same key, \`race" /tmp/vitest-run.log` — 0 hits post-fix; full vitest 406/406 green.
- **Committed in:** `4450733`

**2. [Rule 1 - Bug] Regression spec floor had to assert UNIQUE IDs, not raw array length**
- **Found during:** Task 2 (GREEN) — first full-suite run after dedupe landed
- **Issue:** My original spec used `expect(options.races.length).toBeGreaterThanOrEqual(compiledRaceCatalog.races.length)` (= 46). After the Rule 2 dedupe, `options.races.length` is 45 (one `race:drow` deduped away). Spec failed with `expected 45 to be >= 46`.
- **Fix:** Changed the floor to `new Set(compiledRaceCatalog.races.map(r => r.id)).size` (= 45). This is the true D-05 contract — "every UNIQUE parent race surfaces" — and is robust to both extractor duplicates and future growth.
- **Files modified:** `tests/phase-12.1/race-roster-wiring.spec.ts`
- **Verification:** Spec 10/10 green.
- **Committed in:** `4450733`

**3. [Rule 1 - Bug] Phase-03 + Phase-08 specs hard-coded subrace:moon-elf, which the compiled catalog does not emit**
- **Found during:** Task 2 (GREEN) full-suite run — 2 phase-03/phase-08 tests broke when the fixture's hand-authored subraces were deleted.
- **Issue:** `tests/phase-03/summary-status.spec.tsx` + `tests/phase-08/project-build-document.spec.ts` called `foundation.setSubrace('subrace:moon-elf')`. The store's `subraceMatchesRace` gate (driven by the fixture's `subraces` array) silently dropped the call because `compiledRaceCatalog.subraces === []`. Label assertions for "Elfo lunar" then failed.
- **Fix:** Removed the subrace assertions. The label-composition contract (race + alignment → characterLabel) still stands without a subrace; the subrace dropdown-pair is covered by the new 12.1-02 spec. Added plan-referenced comments so the next editor knows which spec owns the subrace contract.
- **Files modified:** `tests/phase-03/summary-status.spec.tsx`, `tests/phase-08/project-build-document.spec.ts`
- **Verification:** both specs green (3/3 and 5/5 respectively).
- **Committed in:** `4450733`
