## Spacing Scale

Declared values (multiples of 4). Inherited unchanged from `tokens.css`:

| Token | Value | Usage in Phase 7 |
|-------|-------|------------------|
| `--space-xs` | 4px | Gap between rejection-reason lines in detail panel; prereq-list item vertical padding |
| `--space-sm` | 8px | Gap between spell rows, gap between level tabs (0..9), inline meta-inline gap inside rows |
| `--space-md` | 16px | `MagicBoard` outer padding, section block padding, search bar block padding |
| `--space-lg` | 24px | Gap between magic sections (Dominios vs Grimorio vs Conocidos), vertical gap between selected-list and available-list |
| `--space-xl` | 32px | Reserved; not expected to be used by magic surface |
| `--space-2xl` / `--space-3xl` | 48px / 64px | Not used this phase |

Exceptions:
- Spell level tabs (0..9): 40px min-height, 40px min-width — matches `.feat-board__search input` control size and gives an accessible touch target for ten adjacent tabs (LANG-02 readability + 44px-adjacent accessibility tolerance).
- Spell row stepper controls (`+` / `-` for wizard spellbook remove, sorcerer/bard swap): 44px min-height / min-width — matches the existing `.skill-sheet__stepper` control size so the magic row rhythm aligns with skills.
- Domain card (cleric level 1 only): 2-column grid of selectable domain tiles, each tile height auto, min 64px so two tiles always fit within one scroll viewport.

Source: `tokens.css`, `app.css` `.skill-sheet__stepper` (line 611), `app.css` `.feat-board__search input` (line 1473).

## Interaction Model

### Domain selection (cleric level 1)
- Click an unselected tile and it becomes "preview" (selection-border, not committed).
- `Aceptar` commits both preview picks; store commit triggers full recompute (D-05).
- Hard block (alignment/deity incompatible domain): `+`/click suppressed, tile stays muted with inline "Requiere alineamiento bueno".
- Click selected tile a second time to deselect.

### Wizard spellbook (grimorio)
- "Añadir al grimorio" reveals an `OptionList` of spells at the active level, scrollable, 40px min row height.
- Row `+` button disabled when slot cap reached (hard block on adding) OR when spell is already in grimoire (deduplication hard block).
- Existing entries list: each row has an inline `NwnButton` variant="muted" labelled "Eliminar del grimorio" (D-16 explicit forget).
- Confirmation dialog for Eliminar uses `ConfirmDialog` primitive with body: "¿Eliminar «{spellName}» del grimorio? Se perderá el acceso a este conjuro."

### Sorcerer / Bard known spells
- `+` adds a new known spell until cap reached → hard block with inline "No puedes aprender más conjuros de nivel N en este nivel de clase."
- At swap levels (D-15), the "Cambiar conjuro conocido" `NwnButton` variant="primary" appears above the known list. Click opens `ConfirmDialog` with two sequential `OptionList` steps: pick spell to forget, then pick replacement. Confirmation body: "¿Olvidar «{X}» y aprender «{Y}»? Este cambio solo es posible en el nivel {level}."

### Cascade repair (D-06, D-17)
- When an upstream change (class swap at level N-k, ability change, prereq feat removed) invalidates a magic selection at level N:
  - Selection is preserved (never auto-deleted).
  - Row gains `is-repair-needed` styling (amber border, selection fill).
  - Inline row text gains "Reparar: {reason}" prefix in `var(--color-blocked)` color.
  - Stepper `magia` sub-step for that level shows amber border with aria-label "Magia en reparación".
  - Shell summary rolls up to severity `repair_needed` or `illegal` depending on whether build would export.

### Recomputation
- Every zustand store commit triggers selectors; no explicit "recomputar" UI (D-05).
- Latency target < 50ms at level-16 builds — achieved by memoizing selectors at store slice level. No visible spinner required for this target; if ever exceeded, render an accessible `aria-live="polite"` banner "Revalidando…" that auto-dismisses on completion. For MVP assume the target holds and omit the banner.

### Rejection presentation (VALI-02)
- Inline short reason under the row (one line, never wraps beyond 2): e.g. "Requiere Hechicero nivel 5" — styled identically to `.feat-board__blocked-reason .is-failed` (red left-border, 14px, 1.4 line-height).
- Detail panel full breakdown: mirrors `.feat-board__prereq-list` with `.is-met` / `.is-failed` items, each showing required vs. current. For spell chains: show prerequisite spells as separate list entries ("Requiere: Misil mágico ✓", "Requiere: Dominio de Fuego ✗").

## Shell Summary Severity

Extend `planner-shell` validation projection (D-09, D-10) by folding magic into the severity chain:

Order (highest priority first):
1. `foundation` blocked/illegal
2. `progression` blocked/illegal
3. `skills` blocked/illegal
4. `feats` blocked/illegal
5. **`magic` blocked/illegal** *(new)*
6. `repair_needed` (any tier: new `magia` repair joins skills + feats repair set)
7. `legal` (export-ready)

Visual contract (unchanged from 05.2):
- Summary pill color = severity color (gold = legal, blocked-amber, destructive-red)
- Share / export gate (Phase 8 handoff) MUST check `severity === 'legal'` — soft blocks prevent export even though they are preserved in state (D-10).
- Summary pill copy uses `shellCopyEs.summaryValues.validation` — Phase 7 must add Spanish strings for new states (see Copywriting below).
