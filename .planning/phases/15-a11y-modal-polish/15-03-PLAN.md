The selector `selectFeatBoardView(featState, ...)` consumes the FULL `FeatStoreState` object. That object's relevant data fields (per `apps/planner/src/features/feats/store.ts:17-37`) are: `activeLevel`, `datasetId`, `lastEditedLevel`, `levels`. Action functions live alongside (`clearClassFeat`, `clearGeneralFeat`, `resetFeatSelections`, `resetLevel`, `setActiveLevel`, `setClassFeat`, `setGeneralFeat`).

The selector signature accepts the whole `FeatStoreState` shape (data + action fields). Pick **strategy A (slice-as-input via `useShallow`)** — the destructured slice IS the selector input. This eliminates the per-render `useFeatStore.getState()` snapshot read AND eliminates `void` discards, because every destructured field is consumed (passed to `selectFeatBoardView`).

The reactive + selector-input shape is therefore:

```tsx
const featState = useFeatStore(
  useShallow((s) => ({
    levels: s.levels,
    activeLevel: s.activeLevel,
    datasetId: s.datasetId,
    lastEditedLevel: s.lastEditedLevel,
    clearClassFeat: s.clearClassFeat,
    clearGeneralFeat: s.clearGeneralFeat,
    resetFeatSelections: s.resetFeatSelections,
    resetLevel: s.resetLevel,
    setActiveLevel: s.setActiveLevel,
    setClassFeat: s.setClassFeat,
    setGeneralFeat: s.setGeneralFeat,
  })),
);
```

Why this is correct:

- The slice covers EVERY field `FeatStoreState` exposes — selector receives the full shape it expects with no typing dance.
- `useShallow` referentially compares each field; action references are stable across zustand renders (zustand keeps action identity constant), so the shallow check passes for actions and only data-field identity changes will re-render.
- Single source of subscription truth: subscription slice IS selector input, so subscription/selector drift is structurally impossible.
- Closes the per-render `getState()` thrash that was the Phase 06 WR-04 root cause (selector input was a fresh object every render under the older `getState()` strategy).
