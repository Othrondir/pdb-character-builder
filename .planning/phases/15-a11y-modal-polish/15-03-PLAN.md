The selector `selectFeatBoardView(featState, ...)` consumes the FULL `FeatStoreState` object. That object's relevant data fields (per `apps/planner/src/features/feats/store.ts:17-37`) are: `activeLevel`, `datasetId`, `lastEditedLevel`, `levels`. Action functions live alongside (`clearClassFeat`, `clearGeneralFeat`, `resetFeatSelections`, `resetLevel`, `setActiveLevel`, `setClassFeat`, `setGeneralFeat`).

The selector signature accepts the whole state shape; passing a synthesised partial would be a typing dance. The PATTERNS.md "Caveat" section lists the two viable strategies. Pick **strategy B (`useFeatStore.getState()` for selector input)** because:
- It preserves the selector's typed shape (no `as FeatStoreState` casts that would re-introduce the very kind of unsafe assertion D-07 closes).
- `getState()` is a one-shot read on each render — equivalent to today's full subscription for SELECTION purposes.
- The `useShallow` slice is what controls SUBSCRIPTION (which fields trigger re-render). Selector input is unrelated to subscription — that is what causes WR-01 in the first place.

For each consumer, the reactive shape is therefore:
```tsx
const { levels, activeLevel, datasetId, lastEditedLevel } = useFeatStore(
  useShallow((s) => ({
    levels: s.levels,
    activeLevel: s.activeLevel,
    datasetId: s.datasetId,
    lastEditedLevel: s.lastEditedLevel,
  })),
);
```

…but `selectFeatBoardView` is invoked with the raw store snapshot (`useFeatStore.getState()`) so the selector keeps its full-shape input unchanged.

| `useFeatStore.getState()` snapshot read | One-shot read for selector input. The selector receives a possibly-newer state than the subscription slice; this is fine because selectors are pure and the next render reconciles. |

| T-15-03-02 | Information Disclosure | `useFeatStore.getState()` read | accept | Reads the same data already accessible via the previous full subscription; no new info exposed. |
| T-15-03-03 | Denial of Service | Selector input vs subscription mismatch | mitigate | If subscription slice and selector input drift, the consumer would render stale data. Mitigated by including ALL fields the selector reads in the useShallow slice.
