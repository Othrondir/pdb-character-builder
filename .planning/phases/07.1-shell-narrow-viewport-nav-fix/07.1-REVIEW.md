# Phase 07.1: Code Review Report

**Reviewed:** 2026-04-17
**Depth:** standard
**Files Reviewed:** 9
**Status:** issues_found

## Summary

Phase 07.1 delivers a mobile nav toggle + backdrop + escape/close affordances for ≤1023px viewports, backed by two new Zustand actions (`toggleMobileNav`, `closeMobileNav`) and three test specs. The store code is correct and tightly scoped. Spanish copy keys are in place. The CSS correctly confines the drawer machinery to the intended breakpoint and ships sensible desktop defaults (`display: none`).

The component's core open/close/escape behavior is sound and well-tested, but the accessibility contract claimed by the markup (`aria-modal`, focus-to-close-button-on-open, focus-return-to-toggle-on-close) is not fully delivered by the implementation:

- The focus-management effect runs on initial mount and steals focus to the toggle button even when the drawer was never opened.
- `aria-modal="true"` is asserted on the drawer, but no focus trap is installed — Tab from within the drawer leaks focus to the page behind the backdrop, which contradicts the `aria-modal` semantics.
- None of the three test specs exercise focus behavior at all, so the a11y contract is unverified by CI.
- Body scroll lock is missing while the drawer is open, so mobile users can scroll the background content beneath the backdrop.

No security issues, no data bugs, no store regressions. Findings are concentrated in a11y/UX correctness and test coverage.

## Warnings

### WR-01: Focus-management effect steals focus on initial mount

**File:** `apps/planner/src/components/shell/mobile-nav-toggle.tsx:28-47`
**Issue:** The second `useEffect` runs on initial mount with `mobileNavOpen === false`. That path skips the `if (mobileNavOpen)` branch and falls through to `toggleButtonRef.current?.focus()`. This means: when the planner shell first renders on a ≤1023px viewport, focus is yanked to the "Menú" toggle button before the user has interacted with anything. On desktop the button is `display: none`, so `.focus()` is a no-op and the bug is masked — but on mobile any existing focused element (URL bar focus, a previously focused control surviving a route change, etc.) will lose focus.

The effect conflates "drawer just closed" with "drawer is in its default-closed state," and only the former should restore focus.

**Fix:** Track open state transitions with a ref, or gate the else branch on "previously open":
```tsx
const wasOpenRef = useRef(false);

useEffect(() => {
  if (mobileNavOpen) {
    wasOpenRef.current = true;
    const raf =
      typeof requestAnimationFrame === 'function'
        ? requestAnimationFrame(() => {
            closeButtonRef.current?.focus();
          })
        : null;
    return () => {
      if (raf !== null && typeof cancelAnimationFrame === 'function') {
        cancelAnimationFrame(raf);
      }
    };
  }
  // Only restore focus if we are transitioning open -> closed.
  if (wasOpenRef.current) {
    wasOpenRef.current = false;
    toggleButtonRef.current?.focus();
  }
  return undefined;
}, [mobileNavOpen]);
```

### WR-02: `aria-modal="true"` asserted without a focus trap

**File:** `apps/planner/src/components/shell/planner-shell-frame.tsx:17-21`
**Issue:** When `mobileNavOpen` is true, the drawer is tagged `role="dialog"` + `aria-modal={true}`. ARIA 1.2 defines `aria-modal="true"` as a commitment that focus is confined to the dialog. The current implementation does not install a focus trap, so pressing Tab from the close button (or any focusable element inside `<CreationStepper />`) will move focus onto elements behind the backdrop. Screen-reader users will also be told "dialog, modal" while still being able to reach the main content, which is misleading.

Either drop `aria-modal` (keep `role="dialog"` for structural semantics only) or install a focus trap that loops Tab / Shift+Tab within the drawer while open.

**Fix (cheapest path — weaken the claim):**
```tsx
// In planner-shell-frame.tsx, remove aria-modal:
<div
  aria-label={shellCopyEs.stepper.heading}
  className={`planner-layout__stepper${mobileNavOpen ? ' is-open' : ''}`}
  id="planner-stepper-drawer"
  role={mobileNavOpen ? 'dialog' : undefined}
>
```

**Fix (stronger path — add a focus trap):** Install `focus-trap-react` or a small custom trap keyed on `mobileNavOpen` that intercepts Tab/Shift+Tab at the drawer root.

### WR-03: No test coverage for the accessibility contract

**File:** `tests/phase-07.1/mobile-nav-toggle.spec.tsx`, `tests/phase-07.1/mobile-nav-close-affordances.spec.tsx`
**Issue:** Three things the component promises in code and comments are not asserted in any test:

1. Opening the drawer moves focus to the close button (line 28-37 in `mobile-nav-toggle.tsx`).
2. Closing the drawer returns focus to the toggle button (line 45).
3. The drawer is `role="dialog"` with `aria-modal="true"` only while open (verifiable in `planner-shell-frame.tsx`).

Without these assertions, any of WR-01, WR-02, or a future regression to the focus effect will pass CI silently. Given that the phase plan explicitly calls out accessibility as a deliverable, this is a meaningful coverage gap.

**Fix:** Add assertions to `mobile-nav-close-affordances.spec.tsx`:
```tsx
it('moves focus to the close button when the drawer opens', async () => {
  render(createElement(MobileNavToggle));
  openDrawer();
  // Flush requestAnimationFrame before asserting focus.
  await new Promise((r) => requestAnimationFrame(() => r(null)));
  const close = screen.getByRole('button', { name: 'Cerrar menú' });
  expect(document.activeElement).toBe(close);
});

it('returns focus to the toggle when the drawer closes via Escape', async () => {
  render(createElement(MobileNavToggle));
  openDrawer();
  await new Promise((r) => requestAnimationFrame(() => r(null)));
  fireEvent.keyDown(window, { key: 'Escape' });
  const toggle = screen.getByRole('button', { name: 'Menú' });
  expect(document.activeElement).toBe(toggle);
});

it('does NOT steal focus on initial mount (WR-01)', () => {
  const outside = document.createElement('button');
  outside.textContent = 'outside';
  document.body.appendChild(outside);
  outside.focus();
  expect(document.activeElement).toBe(outside);
  render(createElement(MobileNavToggle));
  expect(document.activeElement).toBe(outside);
  document.body.removeChild(outside);
});
```
And in an integration spec over `PlannerShellFrame`, assert that `[role="dialog"][aria-modal="true"]` is present only when `mobileNavOpen` is true.

### WR-04: Body scroll is not locked while drawer is open

**File:** `apps/planner/src/components/shell/mobile-nav-toggle.tsx` (or `planner-shell-frame.tsx`)
**Issue:** On mobile viewports the drawer sits above a backdrop, but the document body is still scrollable. Touch-scrolling on the backdrop moves the underlying planner content. This both breaks the perceptual "modal" affordance and works against `aria-modal`. For iOS Safari in particular, momentum scroll on a `position: fixed` overlay is known to bleed through to `body` unless `overflow: hidden` is applied at the document root.

**Fix:** Toggle `overflow: hidden` on `document.body` while open:
```tsx
useEffect(() => {
  if (!mobileNavOpen) return;
  const previous = document.body.style.overflow;
  document.body.style.overflow = 'hidden';
  return () => {
    document.body.style.overflow = previous;
  };
}, [mobileNavOpen]);
```

## Info

### IN-01: Copy verifier regex accepts any non-empty string

**File:** `scripts/verify-phase-07.1-copy.cjs:17-23`
**Issue:** The regex `openNav:\s*'[^']+'` confirms only that the key exists and has a non-empty single-quoted value. A regression that replaced `'Menú'` with `'TODO'`, `'Menu'` (ASCII), or `'placeholder'` would pass. Given that Spanish-first copy is a hard project constraint (see `CLAUDE.md`), a stricter check would catch accidental ASCII fallbacks.
**Fix:** Assert the expected literals, or at least that the value contains a non-ASCII character:
```js
if (!/stepper:\s*\{[\s\S]*?openNav:\s*'Menú'/m.test(es)) {
  console.error("FAIL: shellCopyEs.stepper.openNav is not 'Menú'");
  failed = true;
}
if (!/stepper:\s*\{[\s\S]*?closeNav:\s*'Cerrar menú'/m.test(es)) {
  console.error("FAIL: shellCopyEs.stepper.closeNav is not 'Cerrar menú'");
  failed = true;
}
```

### IN-02: Redundant `display: none` on `.planner-layout__backdrop` inside the media query

**File:** `apps/planner/src/styles/app.css:1417,1465`
**Issue:** `.planner-layout__backdrop` is set to `display: none` in the desktop defaults at line 1417, and then set to `display: none` again inside the `@media (max-width: 1023px)` block at line 1465, before `.is-open { display: block }` on line 1470 overrides it. The second `display: none` is redundant — the desktop default already covers the closed state.
**Fix:** Drop line 1465 (the inner `display: none`). Keep the `inset: 0; position: fixed; z-index: 9; background: ...;` declarations and let `.is-open { display: block }` do the toggling.

### IN-03: `--color-surface-raised` token has no matching definition

**File:** `apps/planner/src/styles/app.css:1436,1451`
**Issue:** Both the toggle and close buttons reference `var(--color-surface-raised, rgba(0, 0, 0, 0.4))` / `rgba(0, 0, 0, 0.6)` as background. I did not find `--color-surface-raised` defined elsewhere in `app.css`, which means the fallback is always what renders. If the token is intended to be part of the design system, it should be declared in `:root`; if not, using a direct rgba removes the dead indirection.
**Fix:** Either declare `--color-surface-raised` alongside other design tokens, or replace the two references with their rgba fallbacks and drop the `var()` wrapper.

### IN-04: `aria-expanded` set on toggle button even when toggle is `display: none` on desktop

**File:** `apps/planner/src/components/shell/mobile-nav-toggle.tsx:53`
**Issue:** The toggle carries `aria-expanded={mobileNavOpen}` at all viewport sizes, but the button is visually hidden on desktop (≥1024px). This is harmless — AT ignores `display: none` subtrees — but it means the drawer state is semantically coupled to a widget the user cannot see on desktop. Not a bug; flagging only because phase comments emphasize the ≤1023px scope.
**Fix:** No change required. If desired, render `<MobileNavToggle />` conditionally via a `useMediaQuery` hook, but CSS-only hiding is the simpler approach and is what this phase chose.

### IN-05: `toggleMobileNav` could be wrapped in a single `set` selector

**File:** `apps/planner/src/state/planner-shell.ts:43`
**Issue:** `toggleMobileNav: () => set((state) => ({ mobileNavOpen: !state.mobileNavOpen }))` is fine, but the store also exposes `closeMobileNav` for parity. If the store ever grows a `openMobileNav` action, the three should live side-by-side. Minor — current shape is correct and matches how the UI uses them (toggle for the open button, close for all close affordances).
**Fix:** No change required for this phase. Consider adding `openMobileNav` if a future flow needs to open the drawer programmatically (e.g., deep-link to a hidden nav item).
