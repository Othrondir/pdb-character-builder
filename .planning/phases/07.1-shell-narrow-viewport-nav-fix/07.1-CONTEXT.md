# Phase 07.1: Shell narrow viewport nav fix - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning
**Source:** Direct diagnosis during Phase 07 UAT (no discuss-phase)

<domain>
## Phase Boundary

Fix the narrow-viewport (тЙд1023px) usability regression introduced by the Phase 05.2 UX overhaul: wire the already-existing `toggleMobileNav` store action to a visible hamburger control, add a close affordance, and guarantee the creation stepper is always reachable regardless of viewport width.

**In scope:**
- Hamburger / drawer-toggle button visible only at тЙд1023px.
- Drawer open/close state bound to `usePlannerShellStore.mobileNavOpen` + `toggleMobileNav`.
- Close affordances: explicit close button inside the drawer, click-outside (backdrop), and Escape key.
- Backdrop overlay that does not destroy attribute/build state on dismiss.
- Spanish labels for all new affordances via `shellCopyEs`.
- Regression tests (jsdom / Vitest) covering open, close via each affordance, and that stepper steps are tabbable when open.

**Out of scope:**
- Any redesign of the stepper itself or its steps.
- Touching Phase 07 magic features, routing, or data selectors.
- Changing the desktop (>1023px) layout.
- Adding new sections or screens to the shell.

</domain>

<decisions>
## Implementation Decisions

### Root cause (locked)
- Media query at `apps/planner/src/styles/app.css:1417-1425` sets `.planner-layout__stepper { display: none; }` at `max-width: 1023px`, visible only when `.is-open` class is present.
- Store exposes `mobileNavOpen` + `toggleMobileNav` at `apps/planner/src/state/planner-shell.ts:12,19,28,42` but **no component consumes `toggleMobileNav`** тАФ grep returns zero call sites outside the store definition itself.
- Net effect: at тЙд1023px the stepper is permanently hidden with no way for the user to reveal it. This blocks all progression past whichever origin step is active when the viewport narrows.

### Component placement
- The toggle button lives inside the main shell header area, NOT inside `CreationStepper` (which is itself the element being hidden).
- Mount point: `apps/planner/src/components/shell/planner-shell-frame.tsx` (the layout already imports the shell store).
- Rendered only at тЙд1023px (CSS-driven, not JS-driven, so SSR / no-JS paths stay predictable).

### Close affordances (all three required)
1. Explicit close button inside the drawer header (e.g. an `├Ч` button labeled `shellCopyEs.stepper.closeNav`).
2. Click on the backdrop overlay dismisses the drawer.
3. `Escape` key on `window` while `mobileNavOpen` is true closes it.

### State semantics
- Toggle action stays `toggleMobileNav` (no rename). Add a dedicated `closeMobileNav` action for explicit dismiss flows so event handlers are not forced to read current state before flipping.
- `mobileNavOpen` resets to `false` on route changes are NOT in scope тАФ current shell has no URL-level routing changes that would require it.
- No persistence тАФ nav state is session-only, must stay `false` on fresh load to preserve desktop defaults.

### Accessibility
- Toggle button: `aria-controls` points at the stepper drawer id, `aria-expanded` reflects `mobileNavOpen`, Spanish visible label (e.g. `Men├║`).
- Drawer: `role="dialog"` or `aria-modal="true"` when open so screen readers trap focus correctly; revert to normal nav semantics when closed.
- Focus management: opening the drawer moves focus to the close button; closing returns focus to the toggle.

### Styling
- No new font imports, no new color tokens. Reuse existing gold/frame tokens from `app.css` for visual consistency with Phase 05.2.
- Backdrop is semi-transparent (e.g. `rgba(0,0,0,0.55)`) and z-indexed below the drawer but above the rest of the shell.
- Drawer slides in from the left to match the existing desktop stepper column position.

### Tests
- Unit/component tests under `tests/phase-07.1/` covering:
  - Toggle button renders at narrow viewport (simulate via jsdom `window.matchMedia` mock or inline class presence).
  - Clicking toggle flips `mobileNavOpen` to `true` and applies `.is-open`.
  - Clicking close button / backdrop / pressing Escape sets `mobileNavOpen` back to `false`.
  - Desktop viewport path does NOT render the toggle.
- Copy regression: extend `scripts/verify-phase-07-copy.cjs` (or add a sibling script) to assert the new `shellCopyEs.stepper.openNav` / `.closeNav` keys exist.

### Claude's Discretion
- Exact visual treatment of the hamburger icon (use `lucide-react` `Menu` / `X` icons, already a dep).
- Exact CSS selectors for the backdrop and drawer slide transition.
- Whether to add a small focus-trap helper or reuse `ConfirmDialog`-style focus management.

</decisions>

<specifics>
## Specific Ideas

- Hamburger icon: `lucide-react` `Menu` (open) and `X` (close). Both already used elsewhere in the shell.
- Spanish copy keys (suggested): `shellCopyEs.stepper.openNav = "Men├║"`, `shellCopyEs.stepper.closeNav = "Cerrar men├║"`.
- The existing `.planner-layout__stepper.is-open` class already triggers visibility тАФ the fix is additive: keep that CSS and introduce the toggle/backdrop UI that drives it.
- Viewport breakpoint: stay at `max-width: 1023px` to match the current media query. Do not introduce a new breakpoint.

</specifics>
