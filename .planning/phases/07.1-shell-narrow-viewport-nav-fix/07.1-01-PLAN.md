    1. **Create `apps/planner/src/components/shell/mobile-nav-toggle.tsx`**:
       ```tsx
       import { useEffect, useRef } from 'react';
       import { Menu, X } from 'lucide-react';
       import { shellCopyEs } from '@planner/lib/copy/es';
       import { usePlannerShellStore } from '@planner/state/planner-shell';

       export function MobileNavToggle() {
         const mobileNavOpen = usePlannerShellStore((state) => state.mobileNavOpen);
         const toggleMobileNav = usePlannerShellStore((state) => state.toggleMobileNav);
         const closeMobileNav = usePlannerShellStore((state) => state.closeMobileNav);

         const toggleButtonRef = useRef<HTMLButtonElement>(null);
         const closeButtonRef = useRef<HTMLButtonElement>(null);

         // Escape key handler — only attached while drawer is open (T-07.1-02: no listener leak).
         useEffect(() => {
           if (!mobileNavOpen) return;
           function onKeyDown(event: KeyboardEvent) {
             if (event.key === 'Escape') {
               event.preventDefault();
               closeMobileNav();
             }
           }
           window.addEventListener('keydown', onKeyDown);
           return () => window.removeEventListener('keydown', onKeyDown);
         }, [mobileNavOpen, closeMobileNav]);

         // Focus management — open moves focus to close button, close returns to toggle.
         useEffect(() => {
           if (mobileNavOpen) {
             requestAnimationFrame(() => {
               closeButtonRef.current?.focus();
             });
           } else {
             // Only return focus to the toggle if it is currently in the DOM
             // (CSS hides it at desktop, but the element still exists).
             toggleButtonRef.current?.focus();
           }
         }, [mobileNavOpen]);

         return (
           <>
             <button
               aria-controls="planner-stepper-drawer"
               aria-expanded={mobileNavOpen}
               aria-label={shellCopyEs.stepper.openNav}
               className="planner-shell__mobile-toggle"
               onClick={toggleMobileNav}
               ref={toggleButtonRef}
               type="button"
             >
               <Menu aria-hidden="true" size={20} />
               <span>{shellCopyEs.stepper.openNav}</span>
             </button>
             <div
               aria-hidden="true"
               className={`planner-layout__backdrop${mobileNavOpen ? ' is-open' : ''}`}
               onClick={closeMobileNav}
               role="presentation"
             />
             {mobileNavOpen && (
               <button
                 aria-label={shellCopyEs.stepper.closeNav}
                 className="planner-shell__mobile-close"
                 onClick={closeMobileNav}
                 ref={closeButtonRef}
                 type="button"
               >
                 <X aria-hidden="true" size={20} />
                 <span>{shellCopyEs.stepper.closeNav}</span>
               </button>
             )}
           </>
         );
       }
       ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| window → keydown listener | User keyboard input reaches a global listener installed by `MobileNavToggle`; must be removed on unmount / close to avoid leaking handlers into other shell state. |
| backdrop click → stepper state | Click event outside the active content must dismiss the drawer WITHOUT mutating origin / progression / skill / magic state. |
| viewport resize → store.mobileNavOpen | User resizes from narrow to desktop while `mobileNavOpen === true`; CSS hides the drawer but the store flag stays true. Must not leak into desktop rendering or a11y tree. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.1-01 | Information Disclosure (focus leak) | `MobileNavToggle` focus effect | mitigate | Focus only moves between the toggle button and the close button (both within the shell). The effect runs `useEffect` with `[mobileNavOpen]` deps — no race with unrelated components. `requestAnimationFrame` guards against focusing an unmounted element. Tested in `mobile-nav-toggle.spec.tsx`. |
| T-07.1-02 | Denial of Service (event-listener leak) | `MobileNavToggle` Escape handler | mitigate | The `window.addEventListener('keydown', ...)` is guarded by `if (!mobileNavOpen) return;` — the listener is attached ONLY while the drawer is open and removed in the effect cleanup when `mobileNavOpen` flips to false or the component unmounts. Regression test `Escape key is a no-op while drawer is already closed` asserts no handler runs when closed. |
| T-07.1-03 | Tampering (state leak between viewports) | `usePlannerShellStore.closeMobileNav` | mitigate | `closeMobileNav` uses `set({ mobileNavOpen: false })` (object form) — it explicitly writes only `mobileNavOpen`, preserving all other slices. Regression test `closing does NOT mutate unrelated store slices` asserts `activeOriginStep`, `expandedLevel`, `characterSheetTab` are untouched across close. |
| T-07.1-04 | Information Disclosure (tab-focus trap bypass) | Drawer backdrop + `role="dialog"` | accept | No full focus trap is installed (CONTEXT "Claude's Discretion" allows reuse of basic focus-return patterns). Tab key can escape the drawer into the center content. Disposition: ACCEPT — content behind the drawer is the user's own build state, not a security-sensitive surface. If this becomes painful in UAT, a focus-trap helper is a P1 follow-up, not a P0 blocker. |
| T-07.1-05 | Elevation of Privilege (CSS-only visibility bypass) | `.planner-shell__mobile-toggle` display rules | accept | A user with devtools can force the toggle visible at desktop by editing CSS. Disposition: ACCEPT — zero security implication (this is a static local-first planner with no backend; there is no "privileged" state to elevate to). Desktop rendering still works correctly because `mobileNavOpen === true` at desktop is a no-op: the drawer is already visible as a column. |
| T-07.1-06 | Spoofing (fake Escape via programmatic KeyboardEvent) | Escape handler | accept | An attacker with JS access can dispatch Escape programmatically and force-close the drawer. Disposition: ACCEPT — there is no cross-origin JS in a static GitHub Pages deployment, and any attacker with JS access already has full store access via `usePlannerShellStore.getState()`. |
