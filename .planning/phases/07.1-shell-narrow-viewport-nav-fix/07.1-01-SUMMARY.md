## Decisions Made

- **Object-form `set` for `closeMobileNav`.** `set({ mobileNavOpen: false })` avoids reading state and physically cannot mutate other slices, which is the T-07.1-03 mitigation. Regression test `closing does NOT mutate unrelated store slices` locks this invariant.
- **Escape listener gated by `mobileNavOpen`.** The `useEffect` short-circuits with `if (!mobileNavOpen) return;` before registering the `window` listener, and the cleanup removes it on close OR unmount. Regression test `Escape key is a no-op while drawer is already closed` proves the listener is absent when the drawer is closed.
- **Dialog semantics only when open.** `role="dialog"` / `aria-modal` applied conditionally on `mobileNavOpen`. At desktop the stepper column is just a `<div>` — no dialog semantics on a persistently-visible layout region.
- **CSS-only viewport gating.** Visibility of the new chrome is controlled exclusively by the `@media (max-width: 1023px)` block; no JS `matchMedia` hook. Keeps SSR / no-JS paths predictable and matches the existing `.planner-layout__stepper` breakpoint precisely.
- **Focus return via refs + rAF.** No focus-trap library. `requestAnimationFrame` schedules the focus call for after the render pass, and the rAF handle is cancelled in cleanup to avoid focusing an unmounted element.

None substantive — plan executed as written. Two minor, non-semantic adjustments worth noting:

1. **rAF cleanup.** The plan's example code ran `requestAnimationFrame(...)` inside the open branch without cancellation. The shipped version stores the rAF handle and cancels it on cleanup (standard React effect hygiene). This only affects the rapid-unmount-during-transition edge case — behavior under any of the tested scenarios is identical.
