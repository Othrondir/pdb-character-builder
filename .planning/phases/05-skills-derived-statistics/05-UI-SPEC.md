## Layout Contract

- Preserve the existing routed shell, left navigation, and persistent summary panel. Phase 5 changes the route bodies for `Habilidades` and `Estadísticas`, not the shell frame.
- `Habilidades` owns all editable skill allocation. It uses the established dark board treatment from Phases 3-4 with one board containing:
  1. a compact skill summary strip near the top,
  2. a left level rail for levels `1-16`,
  3. a right active skill sheet for the selected level.
- The skill summary strip is secondary to the editor and should show only the technical fields needed to orient editing: current class, available points, spent points, remaining points, and current severity.
- The level rail remains the primary navigator. It should feel like a skills ledger tied to progression, not a spreadsheet index.
- The active skill sheet owns per-level rank editing for the selected level only. It must not show a dense all-level matrix.
- The active skill sheet is divided into three visible sections:
  1. level context and status,
  2. editable skill rows grouped by purpose,
  3. inline legality and repair explanations.
- `Estadísticas` stays read-only and technical. It uses the same board language but presents synchronized derived output in grouped read-only panels instead of controls.
- `Estadísticas` should contain three technical groups only:
  1. skill totals,
  2. caps and class or transclase costs,
  3. penalties or blocked reasons affecting current results.
- Desktop layout target for `Habilidades`:
  - top: compact skill summary strip,
  - left column: dense rail with all levels visible,
  - right column: wider skill sheet with grouped rows and inline explanations.
- Desktop layout target for `Estadísticas`:
  - top: page heading and short technical description,
  - below: two-column derived panels that collapse cleanly on smaller widths.
- Mobile layout target:
  - summary strip first,
  - rail collapses to a stacked list or compact horizontal strip,
  - active skill sheet appears directly below the active rail entry,
  - `Estadísticas` panels stack vertically with no horizontal scrolling.

## Interaction Contract

- Clicking a level in the `Habilidades` rail switches the active skill sheet without discarding preserved downstream allocations.
- Skill editing is tied to the class and build snapshot of the selected level. The user should always understand which level context they are editing.
- Rank controls must be row-local and explicit. Use plus or minus buttons, stepper-like controls, or equivalent framed actions that match the existing handcrafted shell language.
- Every editable skill row must expose, without requiring hover:
  - current invested ranks for that level,
  - whether the skill is clase or transclase at that level,
  - current cap for that level,
  - point cost for the next valid increment,
  - inline blocked or invalid reason when editing is not allowed.
- The active level uses the existing restrained blue-black selection fill with gold framing.
- Downstream preservation is mandatory. When an earlier class, INT, or related input changes, later skill allocations remain visible and become `Bloqueada` or `Inválida` until repaired.
- Repair state must be shown in all three places using the same severity vocabulary:
  - the level rail,
  - the active skill sheet,
  - the persistent summary panel.
- `Estadísticas` must never become a second editing surface. It explains the evaluated result only.
- Heavy-armor or equivalent server exceptions must appear as hard-stop inline feedback near the affected skill or stat block, not as dismissible toast UI.
- Primary route action for this phase stays local and subdued. This phase does not introduce a large page-level save or continue button.

## Visual Direction

- Preserve the approved NWN1-inspired dark-and-gold creation-board language. `Habilidades` should feel like the next technical stage after `Construcción`, not a fresh app area.
- The board should read like a training ledger:
  - compact framed level entries,
  - grouped skill lines with clear numeric rhythm,
  - restrained gold separators,
  - no generic SaaS table styling.
- Skill rows should feel denser than Phase 4 class rows, but still comfortably readable. Density is allowed; clutter is not.
- Group headings inside the skill sheet and stats view should use the display font and feel slightly ceremonial, while the row content stays practical and compact.
- Use accent sparingly for active level context, editable controls, caps, and key dividers only.
- `Bloqueada` states use muted bronze framing; `Inválida` states use dark red framing. Do not use bright warning yellow, green success, or modern toast badges.
- Reuse the existing shell reveal and section fade motion. Do not introduce swipe carousels, tab animations, or spreadsheet-style highlight flashes.

## State and Feedback Contract

- `pending`: the level exists but still has unspent points, incomplete choices, or missing context for full skill evaluation.
- `legal`: show quiet completion only. Do not introduce bright success green or celebratory language.
- `blocked`: use when evaluation cannot proceed safely because earlier decisions or missing data prevent a legal outcome.
- `illegal`: use when the current skill allocation is known to violate a cap, cost rule, or server-specific restriction.
- Inline feedback should identify the exact failing rule whenever the dataset can support it.
- When evaluation is blocked by missing evidence for a Puerta-specific rule, present that as blocked, not legal-by-default.
- Severity must project consistently across `Habilidades`, `Estadísticas`, and the summary panel.
- `Estadísticas` should present penalties and blocked reasons as read-only technical facts, not recommendations.

---

## Mobile and Responsiveness Contract

- At widths below the current board breakpoint, the level rail and active skill sheet stack into one column without losing the selected-level context.
- Skill rows may reflow into two-line layouts on small screens, but rank controls and current values must stay adjacent.
- Avoid any all-skill horizontal table that requires side scrolling.
- `Estadísticas` technical panels stack vertically and keep labels above values when horizontal space tightens.
- All interactive controls remain at or above a `44px` touch target.

---

## Motion Contract

- Reuse subtle fade or rise-in transitions already established in the shell when switching active levels or revealing updated technical panels.
- Motion should clarify:
  - active level changes,
  - derived stat refresh after an edit,
  - downstream repair state after upstream changes.
- Do not animate every skill row independently after each point change.
