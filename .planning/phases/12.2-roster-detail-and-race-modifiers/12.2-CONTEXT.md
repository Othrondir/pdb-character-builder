# Phase 12.2: Roster Detail & Race Ability Modifiers — Context

<domain>
## Phase Boundary

Close the four user-visible correctness gaps Phase 12.1 UAT surfaced after roster wiring shipped:

1. **Picker detail panes are descriptionless** — race + class picker DetailPanel currently shows only the label + "Selecciona una opcion para ver su descripcion." placeholder. The TLK description (present in compiled catalogs as meaningful Spanish text) never reaches the UI because the projection adapters drop `description` during FoundationRaceOption / PlannerClassRecord construction.
2. **Race ability modifiers never apply to attributes** (CHAR-02 structural gap) — `CompiledRace.abilityAdjustments` is populated (Enano `con +2 / cha −2`, Elfo `dex +2 / con −2`, etc.) but the projection adapter drops it, the foundation store has no `racialModifiers` field, no rules-engine helper computes `finalAttributes = base + racial`, and the Atributos board + character sheet display raw `baseAttributes` only. NWN1 races MUST adjust stats.
3. **Prestige-filter mis-gates at L1** — Clérigo wrongly blocked for a Legal Bueno L1 Humano build; `Alma Predilecta`, `Caballero de Luz`, `Paladin Oscuro`, `Paladin Vengador`, `Artífice` wrongly enabled despite unreachable BAB / class-level prereqs. Root cause: `CompiledClass.isBase` is the only filter signal; `prerequisiteColumns.AlignRestrict` hex + `MaxLevel` / PreReqTable columns are unparsed.
4. **Compiled class catalog ships duplicate canonical IDs** — `class:harper` and `class:shadowadept` each appear twice (Arcano / Divino or Int / Wis variants collapse to same slug). Produces React key warnings + silent first-wins collision.

**In scope:**
- Project `description` through both roster adapters and surface it in both picker DetailPanel bodies.
- Introduce a race-ability-modifier pipeline: projection → store → rules-engine helper → Atributos board + character sheet.
- Decode `prerequisiteColumns` / overlay metadata in `projectCompiledClass` so prestige + deity-gated classes resolve correctly at L1.
- Dedupe duplicate canonical IDs at the class-fixture adapter layer (mirroring the race-fixture first-wins pattern from 12.1-02).

**Not in scope:**
- Extractor-level schema expansion (emit new prereq fields from `class-assembler.ts`). That is a bigger Phase 05.1 rework and deferred.
- Source data differentiation (renaming `Harper (Arcano)` vs `Harper (Divino)` in NWN 2DA). Not under our control.
- Attributes UI redesign beyond adding a racial column. Keep NWN2DB-familiar layout.
- Resolving subrace-emission gap in the race catalog (`compiled-races.subraces: []`). Already deferred from 12.1-02; stays parked.
- Any v2 feature work, share-URL schema changes, or character-sheet redesign.

</domain>

<decisions>
## Implementation Decisions

### D-01 — TLK description plumbing (Bug 1)
- **Decision**: Add `description: string` to `FoundationRaceOption` and `PlannerClassRecord` interfaces. Map `description: compiled.description` in both projection adapters. Render the description text in the DetailPanel body of race picker (`origin-board.tsx:77-85`) and class picker (`build-progression-board.tsx:36-42`). No typography or layout changes beyond filling the body with the description string.
- **Rationale**: TLK data is already in the compiled catalogs; the only missing wiring is 4 lines across 2 interface definitions, 2 adapter calls, and 2 render sites. Smallest fix with the largest UX payoff.
- **Expected touchpoints**: 4 files (interfaces in both fixture files, 2 render sites).

### D-02 — Race-ability-modifier data model (Bug 2)
- **Decision**: Keep point-buy state separate from racial adjustments. Foundation store adds `racialModifiers: Record<AttributeKey, number> | null` alongside existing `baseAttributes`. On `setRace(raceId)` the store sets `racialModifiers` from the projected race option's `abilityAdjustments`. A pure selector in `packages/rules-engine/src/foundation/` computes `finalAttributes = base + racial` (zero-fill racial if null).
- **Rationale**: Separating layers preserves point-buy semantics (rebuild or clear race without losing user-entered point-buy) and enables the three-column UI in D-03. Combining at the store level would collapse this information irreversibly.
- **Expected touchpoints**: `apps/planner/src/features/character-foundation/store.ts` (new field + setter), `apps/planner/src/features/character-foundation/foundation-fixture.ts` (`FoundationRaceOption` gets `racialModifiers`; adapter carries it), new `packages/rules-engine/src/foundation/apply-race-modifiers.ts` + `packages/rules-engine/src/foundation/index.ts` export.

### D-03 — Atributos board + character sheet display (Bug 2 UI)
- **Decision**: Atributos board displays three columns: **Base** (point-buy value, editable) · **Racial** (signed modifier, read-only, tinted or `(+2)` / `(−2)` format) · **Final** (computed total, read-only, prominently styled). When no race is selected, the Racial column shows `—` and Final equals Base. When a modifier is 0, display `(0)` not blank, so the user always sees the pipeline is wired. The HOJA DE PERSONAJE stats column displays `Final` values (what matters in play), with the (+mod) next to it derived from the final score.
- **Rationale**: NWN2DB-familiar transparency; explicit Racial column beats hidden math. Users who choose Enano should immediately see CON +2 / CHA −2 applied, not wonder why the sheet shows different numbers than Atributos.
- **Expected touchpoints**: `apps/planner/src/features/character-foundation/attributes-board.tsx` (grid template + three cells per row), `apps/planner/src/features/character-sheet/` (selector consuming finalAttributes instead of baseAttributes).

### D-04 — Prestige-filter fix strategy (Bug 3)
- **Decision**: Adapter-level fix. Enhance `projectCompiledClass` in `class-fixture.ts` to decode the information that already ships in `CompiledClass`: `prerequisiteColumns.AlignRestrict` (hex alignment mask), `prerequisiteColumns.MaxLevel` (class-level caps), `prerequisiteColumns.PreReqTable` (prereq-table reference). Populate `implementedRequirements` + `deferredRequirementLabels` from these decoded fields so `collectVisibleClassOptions` and `evaluateClassEntry` gate correctly. Keep the existing `CLASS_SERVER_RULE_OVERLAY` as the second-layer escape hatch for exceptions the decoded metadata cannot express.
- **Rationale**: Extractor-level fix (re-emitting prereq metadata) would drag in a Phase 05.1 rework. Adapter-level fix is localized, testable with fixture-driven spec, reversible, and correct for the common cases (alignment gate for Clérigo, BAB prereq for combat prestige classes).
- **Expected touchpoints**: `apps/planner/src/features/level-progression/class-fixture.ts` (decoder helpers + projection call site), `apps/planner/src/features/level-progression/selectors.ts` only if overlay defaults need adjustment, new regression spec covering Clérigo legal at L1 + the five wrongly-enabled prestige classes blocked at L1.

### D-05 — Class dup-key dedupe (Bug 4)
- **Decision**: Fixture-level dedupe mirroring the race-fixture pattern from 12.1-02. Apply a first-wins `dedupeByCanonicalId` guard in `projectCompiledClass` before the `collectVisibleClassOptions` call. Emit a warning log with the dropped source-row index so the extractor backlog stays visible.
- **Rationale**: The true fix is upstream data (two Harper variants share one label) which is outside our control. Fixture-level dedupe mirrors the race pattern already shipped and committed in Phase 12.1; consistency beats per-entity novelty. Extractor-level slug disambiguation (appending source-row index) is the long-term right move but bigger scope and parked for a future extractor pass.
- **Expected touchpoints**: `apps/planner/src/features/level-progression/class-fixture.ts` (one helper + one call in projection). Regression spec asserts no duplicate IDs emerge after projection.

### D-06 — Plan split
- **Decision**: 4 plans matching the ROADMAP entry:
  - **12.2-01**: Surface TLK descriptions in both pickers (Bug 1).
  - **12.2-02**: Race ability-modifier pipeline end to end (Bug 2) — the largest plan; projection → store → selector → Atributos board → character sheet.
  - **12.2-03**: Prestige-filter adapter fix + Clérigo L1 regression (Bug 3).
  - **12.2-04**: Class-catalog duplicate-ID dedupe (Bug 4).
- **Rationale**: Atomic-per-bug keeps blast radius small. Bug 2 is genuinely large (5 layers) and deserves isolation. Bugs 1 / 3 / 4 are medium-to-small; split also lets them run in parallel waves if dependency graph permits.

### D-07 — Wave / dependency graph
- **Decision**: All four plans declare `wave: 1` tentatively; the planner must verify `files_modified` disjointness and only downgrade overlapping plans to later waves. Expected overlap:
  - **12.2-01 + 12.2-02** both touch `foundation-fixture.ts` / `class-fixture.ts` (interface additions). Likely Wave 1 + Wave 2 or coordinated single-commit interface change.
  - **12.2-03 + 12.2-04** both touch `class-fixture.ts`. Likely Wave 1 + Wave 2.
- The planner resolves the final waves after analyzing overlap; this CONTEXT does not prejudge.

### D-08 — Regression test posture
- **Decision**: TDD per bug with unit / selector tests under `tests/phase-12.2/`.
  - Bug 1: spec asserting `FoundationRaceOption.description` + `PlannerClassRecord.description` are non-empty Spanish strings for at least one canary entry; snapshot or grep `description.length > 0`.
  - Bug 2: pure selector test on `applyRaceModifiers` for Enano (+2 con / −2 cha), Elfo (+2 dex / −2 con), Humano (all zeros). Plus an integration-style store test asserting `setRace(enano)` populates `racialModifiers` and the selector returns expected final values.
  - Bug 3: fixture-driven spec calling `collectVisibleClassOptions` for a L1 Legal Bueno Humano build and asserting Clérigo is legal, the five prestige classes are blocked, and at least one legitimate prestige entry remains legal (e.g., if any L1-legal prestige exists).
  - Bug 4: grep / Set-based spec asserting no duplicate `class:*` IDs after projection + no React-key-warning patterns.
- **Rationale**: Each bug has a clean falsifiable assertion surface; E2E tests deferred as in Phase 12.1 (UX discoverable manually after the unit layer is locked).

<specifics>
## Specific Ideas

- **Do not modify compiled catalogs.** Data-level fixes (Bug 3 extractor metadata, Bug 4 slug disambiguation) stay deferred. All 12.2 plans are adapter-layer or UI-layer.
- **Atributos three-column styling reuses existing tokens.** The Racial column should use a secondary / amber tint and the Final column the primary stat color. No new design tokens; the CSS variables already cover it.
- **Selector purity is a hard rule.** `applyRaceModifiers` is a pure function. No store reads, no React imports, no side effects — so the rules-engine tier stays framework-agnostic and the unit tests run without a DOM.
- **Prestige-filter decoder is best-effort.** Decoding `AlignRestrict` hex → alignment-mask is well-defined (NWN 2DA convention). Decoding `PreReqTable` → concrete rules is the complex case; where metadata is ambiguous, default to emitting a `deferredRequirementLabel` string instead of enabling a class unsafely. Fail closed, not open.
- **Dedupe preserves first-wins.** Matches the race-fixture pattern exactly. Log the dropped source-row index so the extractor backlog stays traceable.
- **Commit style.** Mirror Phase 12 / 12.1's atomic-per-task pattern: one commit per logical change (`test(12.2-01): …`, `fix(12.2-01): …`, etc.) + a `docs(12.2-XX): write plan summary` per plan.
- **Manual UAT after execution.** After 12.2 ships, re-run the Phase 11 + Phase 12.1 UAT walk-through: Humano + Legal Bueno + Clérigo (now legal) + Enano to confirm CON +2 in Atributos. Close the session when the sheet reflects final stats with racial bonuses.
</specifics>

<deferred>
## Deferred Ideas

- **Extractor-level metadata emission** — re-emit `prerequisites`, `implementedRequirements` fields from `class-assembler.ts` so the adapter no longer has to decode `prerequisiteColumns`. Large Phase 05.1 rework; defer to a future extractor pass.
- **Extractor-level slug disambiguation** — when two 2DA rows share a label, append source-row index so canonical IDs stay unique at source (e.g., `class:harper-28`, `class:harper-54`). Future extractor pass.
- **Subrace emission** — `CompiledRace.subraces` is `[]` today; the foundation-fixture `groupRacesByParent` helper handles empty gracefully, but the UI only renders parent-level entries. Separate Phase 05.1 followup.
- **Character-sheet redesign** — surfacing racial modifiers in the character sheet requires selector plumbing only; a full redesign (skills section, derived-stats visuals, etc.) is out of scope.
- **Racial-modifier overrides for custom content** — Puerta de Baldur may layer server-specific racial bonuses that diverge from stock NWN1. If a future UAT surfaces mismatches, record and handle via a `RACE_SERVER_RULE_OVERLAY` mirroring `CLASS_SERVER_RULE_OVERLAY`.
- **Migration / share-URL version bump** — if the store schema changes in D-02 invalidate previously serialized builds, bump `buildEncodingVersion` and add a migration. Only required if UAT surfaces loadability regressions; planner should confirm during execution.

</deferred>
