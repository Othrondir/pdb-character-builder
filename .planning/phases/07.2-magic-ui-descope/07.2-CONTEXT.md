Strip the entire magic vertical out of the planner. The product scope is now bounded by the reference Excel template the user maintained previously (`excel simulador de fichas/Plantilla Base.xlsx`), which contains no spell selection, no cleric domain picker, and no per-class spell progression. - Delete `scripts/verify-phase-07-copy.cjs`.
- Remove `shellCopyEs.magic`, `shellCopyEs.stepper.levelSubSteps.spells`, `shellCopyEs.sheetTabs.spells`.
- Remove the `Conjuros` tab from `CharacterSheet`; keep Estadísticas / Habilidades / Dotes.
- Remove the `spells` sub-step and `spells` union member from `level-sub-steps.tsx`.
- Remove the D-02 non-caster filter (`classHasCastingAtLevel` helper is no longer needed).
- Remove the `spells` route branch from `center-content.tsx`.
- Audit `PlannerValidationStatus` — remove `repair_needed` if no non-magic consumer remains; keep it if progression or skills rely on it.
- Audit `ConfirmDialog` — `confirmDisabled` prop and the `children` slot were added in Phase 07-02 and Phase 07-04. Keep whichever still has a non-magic consumer; remove the rest.
- Archive `.planning/phases/07-magic-full-legality-engine/deferred-items.md` into the SUMMARY trail so the descope decision is traceable.
- Regenerate or hand-edit the remaining compiled catalogs (`compiled-classes.ts`, `compiled-feats.ts`, `compiled-races.ts`, `compiled-skills.ts`, `compiled-deities.ts`) to drop any spell or domain references.

**Out of scope:**
- Touching Phase 07.1 (narrow-viewport shell fix) — that plan is independent and survives this purge. Execute 07.2 BEFORE 07.1 so the drawer lands over the slimmed shell.
- Any redesign of the Dotes (feats) or Habilidades (skills) surfaces — they stay as shipped.
- Extractor source code in `packages/data-extractor/src/assemblers/spell-assembler.ts` and `domain-assembler.ts` — KEEP the code (extractor can still emit the JSON if anyone runs it), but disconnect it from the default `pnpm extract` pipeline so the planner no longer bundles spell/domain artefacts.
- Re-introducing any form of spell selection in a future phase — if it ever returns, it's a new feature phase, not a revert.

<decisions>
## Implementation Decisions

### Reference surface (locked)
The user's prior Excel tool (`excel simulador de fichas/Plantilla Base.xlsx`) defines the product scope. Its sheets:
1. **Resumen Ficha** — summary (maps to planner Summary + character sheet).
2. **Caracteristicas&Dotes** — attributes + feats (maps to Atributos + Dotes panels).
3. **Puntos de habilidad** — skills (maps to Habilidades).
4. **Base de datos** — dataset (maps to the compiled catalogs + extractor).
5. **Dotes** — feats catalog (maps to compiled-feats.ts).

No sheet describes spell selection, spell slots, domain choice, or known/prepared spellbooks. That is the authoritative scoping signal.

### Deletion-first strategy
This phase is net-destructive. Prefer `rm -rf` on whole directories and regen/edit of the compiled catalogs over piecemeal surgical removal. Every delete must be followed by a typecheck + test-run to catch dangling imports immediately.

### Task-level ordering
1. Delete the planner magic feature directory (unblocks shell imports).
2. Delete the rules-engine magic module (unblocks aggregator-reading selectors).
3. Remove shell integration points (sub-step, tab, route, copy).
4. Delete compiled spell + domain catalogs and re-emit or hand-edit the other compiled catalogs.
5. Delete tests/phase-07, scripts/verify-phase-07-copy.cjs.
6. Audit PlannerValidationStatus.repair_needed and ConfirmDialog.confirmDisabled / children for remaining consumers; remove what's orphaned.
7. Typecheck + full vitest suite green; dev server boot smoke; ensure the UI has no magic residue.

### Data catalogs
- Hand-editing the compiled catalogs is acceptable if rerunning the extractor is heavy. A single `compiled-*.ts` that still mentions `spells` or `domains` is a bug.
- If `compiled-classes.ts` exposes a `spellCaster` or similar flag, DROP the flag from the catalog type (no consumer after this phase).

### PlannerValidationStatus audit
`repair_needed` was added in Phase 07-03 for magic cascade repair. Grep all callers before removing. If progression or skills branch on it for non-magic reasons, keep; otherwise drop the union member and any switch-branch referencing it.

### ConfirmDialog audit
- `confirmDisabled` prop — added in Phase 07-04 for SwapSpellDialog. If the only consumer is magic, remove the prop.
- `children` slot — added in Phase 07-02 to let SwapSpellDialog embed an OptionList. If nothing else uses it, remove.
- If either still has a non-magic consumer (e.g., a future dotes swap dialog, or an already-shipped skill swap), keep that specific prop but strip dead branches.

### Test strategy
- `tests/phase-07/` is deleted wholesale.
- `tests/phase-05.2/character-sheet.spec.tsx` must be updated: the "Conjuros" tab is gone, the assertion currently checks `/\d+ conjuros/` — rewrite to check only Estadísticas / Habilidades / Dotes tabs are present.
- No new spec files are introduced for this phase. The existing non-magic specs must continue to pass.

### Copy verifier
- `scripts/verify-phase-07-copy.cjs` is deleted.
- No replacement needed — the Phase 05.2 shell still has its own copy verifier or inline checks for the remaining keys.

### Extractor pipeline
- `packages/data-extractor/src/assemblers/spell-assembler.ts` and `domain-assembler.ts` STAY in the repo (future-proof — if the product ever reintroduces spells, the extractor path is ready).
- `packages/data-extractor/src/cli.ts` must stop emitting `compiled-spells.ts` and `compiled-domains.ts` on the default `pnpm extract` run. A flag or explicit subcommand may be added later; for now, comment out or conditionally skip the magic emit calls.
- `packages/data-extractor/extraction-report.txt` is regenerated without spell/domain sections after the CLI change.

### Claude's Discretion
- Whether to delete `compiled-spells.ts` / `compiled-domains.ts` immediately or leave them as empty-export stubs during transition — deletion is cleaner, do that.
- Whether to remove `@planner/features/magic` path alias from `vite.config.ts` / `tsconfig.json` — yes, if present.
- The exact remediation for any dangling test snapshots.

</decisions>
