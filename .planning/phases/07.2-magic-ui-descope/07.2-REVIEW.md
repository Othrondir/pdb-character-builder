# Phase 07.2: Code Review Report

**Reviewed:** 2026-04-17
**Depth:** standard
**Files Reviewed:** 19
**Status:** issues_found

## Summary

Phase 07.2 descoped the entire Phase 07 magic runtime. The surgery is clean overall: no
orphaned imports from `@planner/features/magic/*`, `@rules-engine/magic/*`,
`compiled-spells`, or `compiled-domains` remain in active source (`apps/`, `packages/`,
`tests/` all verified). The compiled feat catalog has zero `minSpellLevel` entries
(grep count = 0), the Zod schema is aligned with the emitted data, and the extractor
gates `EMIT_MAGIC_CATALOGS` cleanly without leaving dangling writes.

The issues found are leftover surface from the descope — unused copy keys, an unused
component prop, and a dead shared UI primitive (`ConfirmDialog`) with no callers. None
block the build or runtime, but a follow-up cleanup sweep would keep the codebase honest
about what's actually live. Residual `.claude/worktrees/` copies of the deleted magic
modules are intentional (agent worktree scratch space) and were ignored.

## Warnings

### WR-01: `LevelSubSteps` retains dead `level` prop and degraded aria-label after magic-filter removal

**File:** `apps/planner/src/components/shell/level-sub-steps.tsx:6-21`
**Issue:** The `level: ProgressionLevel` prop is declared on `LevelSubStepsProps` and
destructured in the function signature (line 14), but the component body never reads
it — the pre-descope `classHasCastingAtLevel(level, progressionState)` filter was its
only consumer. TypeScript will not flag this because the prop is destructured, but:

1. The prop is live-dead code: callers (`creation-stepper.tsx:92`) still compute and
   pass `expandedLevel`, so the prop is part of the public API surface with no effect.
2. The `aria-label` on line 21 is ``` `Sub-pasos del nivel` ``` — a template literal
   with no interpolation. Before the descope the intent was very likely
   `` `Sub-pasos del nivel ${level}` ``. As-is, every level's sub-step group has the
   same accessible name, which is a screen-reader regression relative to the prior
   behaviour.

**Fix:** Either drop the prop (if the aria-label was always intended to be generic) or
use it:

```tsx
interface LevelSubStepsProps {
  level: ProgressionLevel;
}

export function LevelSubSteps({ level }: LevelSubStepsProps) {
  // ...
  return (
    <div
      className="level-sub-steps"
      role="group"
      aria-label={`Sub-pasos del nivel ${level}`}
    >
      {/* ... */}
    </div>
  );
}
```

If the prop is dropped, also remove the call-site argument in
`creation-stepper.tsx:92` and the `ProgressionLevel` import on line 2 of this file.

### WR-02: `ConfirmDialog` primitive is now fully dead code

**File:** `apps/planner/src/components/ui/confirm-dialog.tsx:1-41`
**Issue:** Grep for `from.*confirm-dialog`, `import.*ConfirmDialog`, and
`<ConfirmDialog` across `apps/planner/` and `tests/` returns zero hits. The component's
only consumers under the Phase 07 design were the magic-flow confirm dialogs (the diff
shows `children` and `confirmDisabled` props were removed because `SwapSpellDialog`
was deleted). The component itself was retained as a shared primitive but now has no
callers. The CSS rules (`.confirm-dialog*` in `styles/app.css:1226+`) are also
unreachable.

This is not a bug — the file typechecks and is tree-shaken out of the bundle by Vite —
but it silently claims to be a shared UI primitive while no one uses it, which is
misleading for future work (e.g., the orphaned copy keys in WR flagged below assume a
confirm dialog consumer exists).

**Fix:** Choose one:

1. **Delete** `apps/planner/src/components/ui/confirm-dialog.tsx` and the associated
   `.confirm-dialog*` CSS rules in `styles/app.css`. Re-introduce when a caller
   actually needs it.
2. **Keep** the primitive deliberately, and add a comment explaining it is held for
   future reset-flow dialogs (the `resetOriginConfirm` / `resetLevelConfirm` /
   `resetAllConfirm` copy keys are still present and still unused — see IN-01).

### IN-03: `FEAT_CATEGORY_LABELS` still maps magic-flavoured categories

**File:** `apps/planner/src/features/feats/selectors.ts:33-45`
**Issue:** The category-label lookup contains `'3': 'Arcana'` and `'15': 'Divina'`.
Compiled feats with `category: "3"` and `category: "15"` still exist in
`compiled-feats.ts` (confirmed by `grep -oE '"category":\s*"[^"]+"' | sort -u`). Since
the Magia sub-step and Conjuros tab are gone, the user-visible surface where these
labels would appear is only the general/class feat picker, where a feat can now be
tagged as "Arcana" or "Divina" in the Spanish category chip without the rest of the

This is not a bug — the labels are still semantically correct for the feats' D&D
taxonomy (metamagic / divine feats) — but it's worth confirming the product pivot
accepts these labels being visible in the post-descope UI. If the intent is a totally
magic-free UI string surface, these two entries should map to `'General'` or be
omitted.

### IN-04: Duplicate `ProgressionLevel` type definition (pre-existing)

**File:** `apps/planner/src/lib/sections.ts:40` and
`apps/planner/src/features/level-progression/progression-fixture.ts:26`
**Issue:** `ProgressionLevel` is exported from two modules with structurally
equivalent definitions (both are the `1 | 2 | ... | 16` literal union). Consumers
currently pick one or the other arbitrarily — e.g. `level-sub-steps.tsx` imports from
`@planner/lib/sections`, while `feats/selectors.ts` and `skills/store.ts` import from
`level-progression/progression-fixture`. Both work at compile time, but any future
change to the cap (say, level 20 for epic) would require touching both.

This pre-dates Phase 07.2 but is adjacent to the `sections.ts` edits in this phase.
Worth flagging as a cleanup target for a follow-up refactor.

**Fix:** Pick one source of truth (the `progression-fixture.ts` definition is closer
to the domain) and re-export from `sections.ts` or delete the duplicate.

### IN-05: Extractor CLI progress counter misaligned when magic catalogs are gated off

**File:** `packages/data-extractor/src/cli.ts:269, 277, 291, 305, 319, 341, 372, 391`
**Issue:** The CLI logs progress as `[N/7]` (e.g. `[5/7] Assembling spells...`). When
`EMIT_MAGIC_CATALOGS` is unset (the default), steps 5 and 6 are skipped silently, so
the user sees `[4/7]` → `[7/7]` with no indication of the gap. This is cosmetic, but
it would mislead someone debugging why step 5/6 appear to vanish.

**Fix (optional):**

```ts
console.log('  [5/7] Assembling spells... SKIPPED (EMIT_MAGIC_CATALOGS not set)');
console.log('  [6/7] Assembling domains... SKIPPED (EMIT_MAGIC_CATALOGS not set)');
```

Or renumber to `[5/5]` when the flag is off.

### IN-06: Unused local state in extractor CLI when magic gate is off

```ts
let classLabelsByRow: Map<number, string> = new Map();

if (EMIT_MAGIC_CATALOGS) {
```

### IN-07: `getClassLabel` in feats selectors scans feats array for a class ID

**File:** `apps/planner/src/features/feats/selectors.ts:259-265` — and inherited by
`packages/rules-engine/src/feats/feat-prerequisite.ts:199-200`
**Issue:** The `getClassLabel` helper in selectors is correct: it uses
`compiledClassCatalog.classes.find((c) => c.id === classId)`. However, in
`feat-prerequisite.ts:199` the `minLevelClass` check does:

```ts
const classDef = featCatalog.feats.find((f) => f.id === classId);
const classLabel = classDef?.label ?? classId;
```

This looks up a **class** ID in the **feats** array, which will never match (feat IDs
start with `feat:`, class IDs start with `class:`). The fallback therefore always
fires, showing the raw `class:fighter` string to users when a `minLevelClass`
prerequisite is unmet. This is a pre-existing bug — not introduced by Phase 07.2 —
but it surfaced during the review because the descope reviewer is looking at magic-
adjacent helpers. The fix would route the class catalog through
`evaluateFeatPrerequisites` or accept the class label as a parameter.

Flagging as Info because it is out of Phase 07.2 scope; worth a separate ticket if
the `minLevelClass` surface is user-visible.

**Fix (out of scope for 07.2):** Pass `classCatalog` to `evaluateFeatPrerequisites`
and look up the class label there:

