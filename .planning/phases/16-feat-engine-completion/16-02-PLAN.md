<architectural_decision>
**B-01 Resolution — Pattern S7 compliance for `per-level-budget.ts`'s `determineFeatSlots` call:**

`per-level-budget.ts` lives INSIDE `packages/rules-engine/`. Pattern S7 forbids importing the planner-level `compiledClassCatalog` from the rules-engine package. To preserve framework-agnostic rules-engine while supporting the D-01 precedence ladder at planner-side call sites:

- **Decision (option a):** Make `compiledClass?: CompiledClass | null` an OPTIONAL parameter on `determineFeatSlots`, defaulting to `null` in `per-level-budget.ts`'s call. When omitted/null, the function falls through to `LEGACY_CLASS_BONUS_FEAT_SCHEDULES` for the schedule signal.
- **Why this is safe:** `per-level-budget.ts` is engine-internal budget math. The Phase 12.4-03 fixture sweep already validates against the legacy cadence. Any cadence divergence between extractor and legacy is reconciled at the fixture layer (Step 6 below) — not at the budget-math layer.
- **Why NOT option (b) (callback adapter):** Adds an indirection layer with zero current consumer; option (a) is simpler, preserves the framework-agnostic boundary, and keeps budget math working with legacy schedules until fixture parity is confirmed.
- **Planner-side call sites** (`selectors.ts`, `prestige-gate-build.ts`) DO pass the resolved `compiledClass` arg — that is where D-01's "extractor wins" precedence is actually exercised. Engine-internal callers stay legacy-only.

Concrete consequence:
- `per-level-budget.ts:184` call site OMITS the third argument (or passes `null`) — see Task 16-02-04 Step 2.
- `selectors.ts` + `prestige-gate-build.ts` call sites PASS `compiledClassCatalog.classes.find(...)` — see Task 16-02-04 Step 3 + Step 4.
</architectural_decision>
