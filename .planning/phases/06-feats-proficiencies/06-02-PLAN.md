**2. `feat-sheet.tsx`** -- Left panel with grouped feat list and search.

```typescript
import { useState, useMemo } from 'react';
import { OptionList, type OptionItem } from '@planner/components/ui/option-list';
import { shellCopyEs } from '@planner/lib/copy/es';
import { useFeatStore } from './store';
import { FeatSearch } from './feat-search';
import type { FeatBoardView, FeatOptionView } from './selectors';
```

Props: `{ boardView: FeatBoardView }`.

Component logic:
- State: `const [selectedFeatId, setSelectedFeatId] = useState<string | null>(null)` for highlighting in detail panel.
- State: `const [searchQuery, setSearchQuery] = useState('')` for search filtering.
- Get store actions: `const setClassFeat = useFeatStore(s => s.setClassFeat)`, `const setGeneralFeat = useFeatStore(s => s.setGeneralFeat)`.
- Get activeLevel from boardView: `boardView.activeSheet.level`.
- Determine which feats to show based on `boardView.sequentialStep`:
  - If 'class-bonus': show `boardView.activeSheet.eligibleClassFeats`.
  - If 'general': show `boardView.activeSheet.eligibleGeneralFeats`.
  - If null (no slot available or all filled): show empty or both as read-only.
- Map `FeatOptionView[]` to `OptionItem[]`: `{ id: feat.featId, label: feat.label, secondary: feat.prereqSummary, selected: selectedFeatId === feat.featId }`.
- If search is active (searchQuery.length >= 2), delegate to `FeatSearch` which handles its own results (see below).
- onSelect handler: set `selectedFeatId` for detail panel highlighting, and call `setClassFeat(level, featId)` or `setGeneralFeat(level, featId)` depending on `sequentialStep`.

Render structure:
```tsx
<aside className="planner-panel planner-panel--inner feat-sheet">
  <div className="feat-board__search">
    <input
      type="search"
      role="searchbox"
      aria-label="Buscar dotes"
      placeholder={shellCopyEs.feats.searchPlaceholder}
      value={searchQuery}
      onChange={(e) => setSearchQuery(e.target.value)}
    />
    {searchQuery && (
      <button className="feat-board__search-clear" onClick={() => setSearchQuery('')} type="button" aria-label="Limpiar busqueda">X</button>
    )}

  {searchQuery.length >= 2 ? (
    <FeatSearch query={searchQuery} boardView={boardView} onSelectFeat={setSelectedFeatId} />
  ) : (
    <>
      {boardView.sequentialStep === 'class-bonus' || (boardView.activeSheet.hasClassBonusSlot && boardView.activeSheet.eligibleClassFeats.length > 0) ? (
        <section className="feat-sheet__group">
          <h3 className="feat-board__section-heading">{shellCopyEs.feats.sectionClassFeats}</h3>
          <OptionList items={classItems} onSelect={handleSelectClassFeat} renderItem={renderFeatItem} />
        </section>
      ) : null}
      {boardView.sequentialStep === 'general' || (!boardView.sequentialStep && boardView.activeSheet.hasGeneralSlot) ? (
        <section className="feat-sheet__group">
          <h3 className="feat-board__section-heading">{shellCopyEs.feats.sectionGeneralFeats}</h3>
          <OptionList items={generalItems} onSelect={handleSelectGeneralFeat} renderItem={renderFeatItem} />

  )}
```

Import `DetailPanel` from UI primitives. Import `evaluateFeatPrerequisites` from rules engine and `compiledFeatCatalog` + `compiledClassCatalog` from catalog re-exports.

Props: `{ boardView: FeatBoardView }`.

Logic:
- Find the selected feat (from boardView or a selectedFeatId prop/state -- coordinate with FeatSheet via a shared selected state or by FeatBoard lifting state).
- ALTERNATIVE simpler approach: Add a `focusedFeatId` to the `useFeatStore` or use React context / state lifted to FeatBoard. The simplest approach: FeatBoard holds `const [focusedFeatId, setFocusedFeatId] = useState<string | null>(null)` and passes it down.
- If no feat focused, show default text: the phase step title and an instructional body.

- Compute its prerequisite status by calling `evaluateFeatPrerequisites(feat, buildState, featCatalog)` (get buildState from selector or pass it from the board). The simplest approach: the selectors already expose enough data. Pass the `buildState` computation through the boardView or through a separate selector call inside the detail panel.

**4. `feat-search.tsx`** -- Search component showing both eligible and blocked feats.

Import `evaluateAllFeatsForSearch` from rules-engine (or evaluate inline). This component implements D-06 and D-07.

Props: `{ query: string; boardView: FeatBoardView; onSelectFeat: (featId: string) => void }`.

Logic:
- Normalize query for accent-insensitive search: `const normalizedQuery = query.normalize('NFD').replace(/[\u0300-\u036f]/g, '').toLowerCase()`.
- Read stores directly (foundation, progression, skill, feat) and compute `buildState` for current level.
- For each feat in `compiledFeatCatalog.feats`:
  - Normalize feat label the same way.
  - If normalized label includes normalizedQuery, include in results.
  - Evaluate prerequisites: `evaluateFeatPrerequisites(feat, buildState, featCatalog)`.
  - If all met: show as normal OptionItem.
  - If not met: show as `blocked: true` OptionItem with custom `renderItem` showing failure reasons below the name.
- Debounce is handled by the parent (FeatSheet) using a 200ms debounce on the search query state update. Use `useEffect` with a 200ms timeout for the debounced value.
- If no results match, show `<p>{shellCopyEs.feats.noSearchResults}</p>`.

Render blocked feat items with inline failure reasons (D-07):
```tsx
function renderSearchItem(item: OptionItem & { failedChecks?: PrerequisiteCheck[] }) {
```
