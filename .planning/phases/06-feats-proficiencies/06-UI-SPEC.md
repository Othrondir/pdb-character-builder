## Interaction Contract

### Feat selection flow (D-01 through D-05)

1. **Entry:** User expands a level in the stepper rail and clicks the "Dotes" sub-step.
2. **Center content:** `FeatBoard` renders inside `SelectionScreen` with title "Selecciona las dotes del nivel".
3. **Sequential selection (D-03):** If the level grants both a class bonus feat and a general feat, the board presents class feat selection first. After the class feat is confirmed, it transitions to general feat selection. Title bar updates to reflect current selection step ("Dote de clase" then "Dote general").
4. **Default view (D-01):** OptionList shows only eligible feats. Feats that fail prerequisites are filtered out entirely from the default list.
5. **Grouped sections (D-02):** Eligible feats are divided into "Dotes de clase" (bonus feat list, `list >= 1` in `classFeatLists`) and "Dotes generales" (general feat list, `list === 0` or `allClassesCanUse`). Each section has its own heading.
6. **Inline prerequisites (D-04):** Each feat row in the list shows the feat name followed by fulfilled prerequisite summary in muted text. Format: "Gran Poder [Fue 13, Poder]" -- all met, so shown as a brief label.
7. **Auto-grants (D-05):** Auto-granted feats (from `features[]` in `level-gains.ts`) appear in the level gains summary section, not as selectable items.

### Search and blocked feat feedback (D-06, D-07)

1. **Search field:** Positioned above the OptionList, inside the left panel. Placeholder text: "Buscar dotes...".
2. **Search behavior:** When the user types 2+ characters, the list filters to matching feats by name (case-insensitive, accent-insensitive). Both eligible and ineligible feats appear in search results.
3. **Blocked feats in search (D-06, D-07):** Ineligible feats appear with `is-blocked` modifier (opacity 0.6, `--color-blocked` border). Below the feat name, each failed prerequisite is listed on its own line in `--color-destructive` tinted text: "Requiere: Fuerza 13 (tienes 10)" / "Requiere: Dote Poder (no tomada)".
4. **Search debounce:** 200ms after the last keystroke.
5. **Clear behavior:** An X button or clearing the field returns to the default eligible-only grouped view.

### Prerequisite display in DetailPanel

When a feat is selected (clicked) in the OptionList:

1. **Title:** Feat Spanish name in `detail-panel__title` style.
2. **Description:** Feat Spanish description from `compiledFeatCatalog.feats[].description` in `detail-panel__body`.
3. **Prerequisite checklist:** Below the description, a checklist of all prerequisites. Each line shows:
   - Fulfilled: feat/attribute name in `--color-text-muted` with a check mark character
   - Failed: feat/attribute name in destructive-tinted text with an X character and the current value in parentheses

### Proficiencies (D-08, D-09)

Proficiencies are feats in the catalog. They appear in the feat selection list grouped by their category. No separate proficiency UI or tab. Puerta's custom proficiency splits are already in the extracted data.

### Revalidation (D-10, D-11)

1. **Upstream change invalidation:** When a class, ability score, or skill change removes a prerequisite that a previously selected feat depended on, the feat stays in the selection but gains the `is-illegal` modifier (border: `--color-destructive`, background: `rgba(122, 31, 31, 0.22)`).
2. **Inline reason:** The invalidated feat shows the broken prerequisite inline, same format as D-07.
3. **Severity projection (D-11):** Invalid feat state projects to:
   - Level rail button: `is-illegal` modifier
   - Character sheet Feats tab: red marking on the invalidated feat row
   - Summary strip: invalid feat count
4. **No auto-removal:** The user must manually change the feat or fix the upstream dependency.

### Destructive action: removing a feat that is a prerequisite for later feats

When the user deselects a feat at level N that serves as a prerequisite for a feat selected at level M (where M > N):

1. Show `ConfirmDialog` with title "Quitar dote" and body: "Quitar {feat name} invalidara {dependent feat name} en el nivel {M}. Continuar?".
2. On confirm: remove the feat. The dependent feat at level M transitions to `is-illegal` with the broken prerequisite shown inline.
3. On cancel: no change.

## State Model

### Feat store (`features/feats/store.ts`)

Follows the per-level skill store pattern from `features/skills/store.ts`:

```
featSelections: Record<level, { classFeatId: string | null, generalFeatId: string | null }>
```

Actions:
- `setClassFeat(level, featId)` -- set class bonus feat for level
- `setGeneralFeat(level, featId)` -- set general feat for level
- `clearClassFeat(level)` -- remove class feat selection
- `clearGeneralFeat(level)` -- remove general feat selection
- `resetLevel(level)` -- clear all feat selections for a level

### Feat selectors (`features/feats/selectors.ts`)

Follows the skill selectors pattern from `features/skills/selectors.ts`:

- `selectFeatBoardView(featState, progressionState, foundationState, skillState)` -- returns empty state body or feat board data
- `selectEligibleFeats(level, slotType, featState, progressionState, foundationState, skillState)` -- returns eligible feats for a slot
- `selectFeatPrerequisiteStatus(featId, level, progressionState, foundationState, skillState, featState)` -- returns pass/fail per prerequisite
- `selectFeatSheetView(featState, progressionState)` -- returns all feats for character sheet tab with validity
- `selectFeatSeverity(featState, progressionState, foundationState, skillState)` -- returns severity for shell summary projection

## Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| OptionList keyboard navigation | `role="listbox"` with `role="option"` items. Arrow keys move focus. Enter selects. Already implemented in `option-list.tsx`. |
| Search field | `<input type="search" role="searchbox" aria-label="Buscar dotes">` with `aria-describedby` linking to result count. |
| Blocked feat communication | `aria-disabled="true"` on blocked feats. Failure reasons in `aria-description` attribute. |
| Prerequisite checklist | `role="list"` with `role="listitem"` children. Status conveyed via `aria-label` on each item ("cumplido" / "no cumplido"). |
| SelectionScreen title | `<h2>` in title bar serves as section heading for screen readers. |
| ConfirmDialog | Uses native `<dialog>` with `.showModal()` for focus trapping. Existing `confirm-dialog.tsx` pattern. |
| Sheet tab panel | `role="tabpanel"` with `aria-labelledby` referencing the tab button. Already wired in `character-sheet.tsx`. |
