# Phase 06: Feats & Proficiencies - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can choose Puerta feats and proficiencies with exact prerequisite feedback at each level. This phase delivers the feat selection UI, prerequisite evaluation engine, proficiency modeling, and revalidation when upstream changes (class, abilities, skills) invalidate previously chosen feats. It does not add spells, domains, or other mechanics beyond feats and proficiencies.

</domain>

<decisions>
## Implementation Decisions

### Feat selection flow — NWN1 faithful
- **D-01:** Show only eligible feats by default. Feats that fail prerequisites are filtered out of the selection list entirely.
- **D-02:** Group eligible feats into two sections: "Dotes de clase" (bonus feat from class level) and "Dotes generales". Replicates NWN1's slot distinction. Maps to `classFeatLists` with `list` field (0=general availability, 1+=class bonus).
- **D-03:** When a level grants both a class bonus feat and a general feat, present selection sequentially: class feat first, then general feat. Two clear steps per the NWN1 flow.
- **D-04:** Each feat in the eligible list shows name + fulfilled prerequisites inline (e.g., "Gran Poder [Fue 13 ✓, Poder ✓]").
- **D-05:** Auto-granted class feats (e.g., Barbarian "Furia" at level 1) appear in the level gains summary (as `features[]` in level-gains.ts already does), NOT as a selectable slot. The player sees them but doesn't choose them.

### Prerequisite feedback — Search with block reasons
- **D-06:** A search field lets the player look up any feat by name. If the searched feat is not eligible, it appears in the results but blocked/greyed with the exact reason inline below the name (e.g., "Requiere: Fuerza 13 (tienes 10), Dote: Poder (no tomada)"). Each failed prerequisite listed individually.
- **D-07:** Blocked feats in search results show failure reasons inline under the feat name in red/amber text. No extra click needed to see why.

### Proficiencies and Puerta overrides — Within feat catalog
- **D-08:** Proficiencies (weapons, armor, shields) live as feats in the feat catalog. Puerta's custom splits and overrides should already be captured by the extractor from the server's 2DA data. No separate proficiency data file — verify the extracted catalog covers Puerta's custom proficiency feats.
- **D-09:** Proficiencies appear in the feat selection list and character sheet Feats tab as normal feats, grouped by their category. No separate "Proficiencies" section or tab. This matches NWN1's treatment.

### Revalidation — Preserve and mark invalid
- **D-10:** When an upstream change (class, ability score, skills) invalidates a previously selected feat, the feat stays selected but is marked invalid/red with the reason. The player sees the problem and decides whether to fix upstream or change the feat. Same pattern as Phase 4 progression revalidation.
- **D-11:** Invalid feat severity projects to: level rail button (error state), character sheet Feats tab (red marking), and summary strip (invalid feat count). Same projection pattern as skills and progression.

### Claude's Discretion
- Exact mapping of raw 2DA category numbers (0, 2, 3, 7, 8, 10, 12, 15, 17, 22) to human-readable category labels
- Search field implementation details (debounce, minimum characters, exact placement)
- How feat prerequisites are evaluated internally (pure function signature, caching strategy)
- Exact visual treatment of the "class feat" vs "general feat" section headers
- Whether the prerequisite checklist on eligible feats uses icons, text, or both

</decisions>

## Existing Code Insights

### Reusable Assets
- **compiled-feats.ts**: Full Puerta feat catalog with 1,487 feats, prerequisites, and class feat lists — already extracted and validated against Zod schema
- **SelectionScreen/OptionList/DetailPanel**: NWN1 UI primitives from Phase 05.2 — feat board should compose these
- **skill store pattern**: Per-level allocation with zustand store + derived selectors — direct template for feat store
- **level-gains.ts**: Already emits `features[]` for auto-granted feats per class level
- **shellCopyEs**: Spanish copy module — feat labels should be added here

### Established Patterns
- **Per-level store + selectors**: Skills use a store with per-level records and derived selectors for legality. Feats should follow the same shape.
- **Revalidation via selectors**: Progression and skills recompute legality from raw state — feats should do the same rather than caching validity.
- **Severity projection**: Invalid state projects through shared selectors to rail, sheet tab, and summary strip.

### Integration Points
- **LevelSubStep 'feats'**: Already defined in sections.ts — center-content.tsx needs a case for it
- **SheetTab 'feats'**: Already defined — character-sheet.tsx needs a FeatSheet component
- **Level gains**: `choicePrompts[]` may signal feat choice slots — needs investigation
- **Progression store**: Feat selections per level need to be stored alongside class choices

</code_context>

<specifics>
## Specific Ideas

- Feat category numbers from 2DA (0, 2, 3, 7, 8, 10, 12, 15, 17, 22) need mapping to Spanish labels. The extractor already has "general" for some but others are raw integers.
- The `classFeatLists` structure with `list` field (0-3) and `grantedOnLevel` gives enough info to distinguish class bonus feats from general feats and auto-grants from choices.
- Search-based exploration of blocked feats fulfills FEAT-02 without cluttering the default view.

</specifics>
